# Fuse V3.1 Plan: sharpen the open-ended floor, offline and local

The V3 wave (R0 through R9, archived in [V3-Plan.md](V3-Plan.md)) built the semantic moat, hybrid retrieval, abstention, a wider analyzer set, warm latency, peer and agent runs, and a task-resolution harness. It also surfaced, honestly, the one mode where Fuse is weak: open-ended discovery from a vague description. This plan attacks that, plus the freshness limit that gates graph-aware discovery, using only deterministic, offline, local mechanisms. No model rewrites the query: an agent explores the code freely, so the engine must bridge the vocabulary gap and stay fresh on its own, not depend on a paraphrase.

This file is the forward plan. Each item carries its engine work plus the tests, docs, and benchmark changes that land with it. Tick a box when its definition of done is met; record the measured result in `tests/benchmarks/results` and resync `AGENTS.md` and the benchmarks page in the same change. Plain ASCII prose only.

## Strategy decision: refuse to guess, hand back a map

Fuse is an MCP server in a loop with a model, so it does not have to win the vague one-shot. The primary answer to a low-signal request is not to return a low-precision list, it is to refuse-and-route: tell the caller the request lacks a usable anchor, hand back the partial structure Fuse did see (candidate concepts, entry points, namespaces, nearest symbols), and ask for a symbol, route, service, config, git base, or a narrower description. The calling model, which is exploring the code anyway, then issues a better-anchored query. This converts the weakest mode from an engine-recall problem into a protocol contract, and it takes the hardest input (genuinely no-signal titles) off the table by design.

Three rules keep the contract helpful rather than a wall:

- Refuse-and-route, never just refuse. The response must include what matched (top candidate areas and symbols), so the refusal is a navigation step, not a closed door. An agent that is exploring does not yet know the right symbol; the map is how it finds the next probe.
- Graded, not binary. Three states: confident (return the tight set), partial (return a small best-effort set, flagged low-confidence, with refinement options), insufficient (refuse-and-route). A hard refusal on a borderline-but-answerable query is the new way to be wrong, so the classifier that draws these lines is what must be measured.
- Strict is opt-in with a graceful default. Default to best-effort-plus-flag so a client that cannot refine still gets something; offer a strict mode that hard-requires an anchor for cooperative clients (an agent loop). Do not break callers that send one query and cannot re-prompt.

What this does not remove: once a good, code-anchored query arrives, the engine must be excellent. The contract removes the vague-input burden, not the quality burden. So the deterministic quality work below stays; what is demoted is recall-chasing aimed only at no-signal input.

## Where V3.1 starts (recorded V3 baseline)

All from `tests/benchmarks/results`, counted with `o200k_base`.

- Open-ended localization (Suite C, title only, no base, 108 PRs): overall changed-file recall 30.3 percent lexical (R1) and 34.6 percent with the local dense channel (R2). By bucket: identifier-rich 41 percent (still short of the retired 49 percent lexical floor), natural-language domain 27 percent lexical and 34 percent with dense, formatting 8 percent, route/API low, test-only about 34 percent.
- Precision on the open-ended path is 7 to 8 percent: the engine returns a fixed candidate count regardless of confidence, so most returned files are noise.
- Low-signal detection (R3) is F1 1.0: it correctly abstains on merge, dependency, and CI titles. This must not regress.
- Freshness: incremental re-index (R7) refreshes a changed file's syntax rows (symbols, chunks, full-text, routes) but not the cross-file semantic edges (DI, route, MediatR, EF), which only a full re-index recomputes.
- Agent loop (Suite D): the Fuse MCP tools edge out native file tools on recall at parity tokens; the win concentrates on tasks whose description names wiring.

The root cause of the open-ended weakness: a prose description ("discount rounding is off at checkout") shares almost no tokens with the code (`PriceCalculator`, `ApplyDiscount`, `decimal`, `MidpointRounding`). The default FTS tokenizer (`unicode61`) treats `ApplyRoundingMode` as one opaque token, so "rounding" does not match it. Everything below closes that gap deterministically, then makes the result precise, fresh, and good to explore iteratively.

## S1. Subword identifier indexing (the foundation)

The single biggest offline lever. The FTS index must contain the subwords of every identifier so a prose word matches a compound name. `ApplyRoundingMode` must be findable by `apply`, `rounding`, and `mode`.

- Work: at index time, split each identifier (in the chunk name, declared symbols, and signature, and optionally body identifiers) on CamelCase humps, `snake_case`, and letter/digit boundaries, and write the expansion into a dedicated FTS column (for example `subtokens`) alongside the original. Keep the original token too so exact matches still win. Apply the same splitting to the query before it hits FTS (the lexical generator already splits CamelCase for ranking; extend that to the FTS match expression and add the new column to the `bm25()` weighting, weighted below the exact name but above the body). No tokenizer plugin is required: the expansion is computed in C# and stored as text, so it works on any SQLite with FTS5.
- Tests: a query for a single subword (`rounding`) returns a file whose only match is a compound identifier (`ApplyRoundingMode`); an exact full-identifier query still ranks the exact declaration first; the subtokens column is populated at index time and survives store disposal; snake_case and digit boundaries split.
- Docs: `internals/scoping-internals.mdx` (the subword field and its weight), `internals/caching-internals.mdx` if the schema version bumps.
- Benchmark: Suite C by bucket, expecting identifier-rich to recover past the 49 percent floor and natural-language to gain, no-model. Report precision alongside (subword expansion widens recall, so watch the trade and let S3 tighten it).

## S2. Stemming and a comment/doc bridge field

Two cheap, purely-offline vocabulary bridges that compose with S1.

- Work: (1) Stemming. Switch the full-text tokenizer for the prose-bearing columns to the built-in `porter` stemmer wrapping `unicode61`, and stem the query terms to match, so "rounding", "rounds", and "rounded" collapse. Keep an unstemmed path for exact identifier matches so stemming does not corrupt code tokens. (2) Comment bridge. Index XML doc comments and inline comments as their own weighted FTS field (the `FUSE_FIELDED_COMMENTS` lever exists but is off and unmeasured here): human-written prose in comments uses the developer's vocabulary, which is exactly the vocabulary a vague query is written in. Weight comments above body but below names.
- Tests: a stemmed query ("calculate") matches a file whose code says "Calculation"; a term that appears only in a doc comment localizes that file; exact identifier matching is unaffected by stemming (a regression guard).
- Docs: `internals/scoping-internals.mdx`, `reference/configuration-keys.mdx` (comment field now on by default or documented), the stemming note in scoping.
- Benchmark: Suite C natural-language-domain recall (the bucket comments help most), no-model.

## S3. The MCP signal-sufficiency contract (the centerpiece)

Because Fuse is an MCP server in a loop with a model, the answer to a vague request is to refuse-and-route, not to guess. This is the highest-leverage item: it removes the worst input from scope and fixes the precision disaster (7 to 8 percent) at the same time.

- Work: turn the localize and search path into a graded contract with three outcomes computed deterministically from the candidate score distribution and the existing `QuerySignalClassifier` (R3), no model:
  - Confident: a candidate (or a tight cluster) stands clear of the rest. Return that tight set; this is where precision is won.
  - Partial: there is some signal but no clear winner. Return a small best-effort set, marked low-confidence, together with refinement options (the top candidate areas and symbols) and the inputs that would sharpen it.
  - Insufficient: no usable anchor (the R3 no-signal classes, or a flat, near-uniform score distribution). Refuse-and-route: return no file list, and instead the map Fuse did see (candidate concepts, entry-point files, namespaces, nearest symbols) plus an explicit ask for a symbol, route, service, request, config section, git base, or a narrower description.
  Add a strictness policy: default is best-effort-plus-flag (a non-cooperative client still gets the partial set), and a strict mode (an `--strict`/option, off by default) hard-requires an anchor and returns only the refuse-and-route payload on insufficient input. The refuse-and-route payload is a typed response shape, not a bare error string, so a calling agent can parse the options and issue the next query. Any change to the MCP response DTO honors the AGENTS.md change-safety invariants (bump the protocol version and update the contract test).
- Tests: a query with a clear winner returns the confident tight set (precision up); a flat score distribution returns the partial set with refinement options; a no-signal title returns the insufficient refuse-and-route payload carrying candidate areas (not an empty error); strict mode refuses an unanchored query and answers an anchored one; the default mode never returns nothing for a client that cannot refine; the graded thresholds are reproducible for fixed scores; low-signal classification F1 holds at or above 0.80 and the false-rejection rate on answerable queries stays at or below 0.10.
- Docs: `concepts/scoping.mdx` (the three states and why refuse beats guess), `reference/mcp-tools.mdx` (the graded response shape and the strict option), `scenarios/ask-one-question.mdx` (the refine-and-retry loop).
- Benchmark: Suite C reframed: correct-refusal rate on the no-signal bucket, false-rejection rate on the answerable buckets, and precision at the reported recall when it does answer. This replaces "recall from a bare vague title" as the headline for this path.

## S4. Codebase-derived synonym and abbreviation thesaurus (demoted)

Demoted by the S3 contract: we no longer need to rescue no-signal input, so the thesaurus is justified only if it measurably helps code-anchored queries (an anchored query using an abbreviation the repo expands). Land it only after S1, S2, and S5 show a residual vocabulary gap on anchored queries; otherwise defer. Bridge vocabulary the codebase itself defines, learned offline from the repo, no model.

- Work: at index time, build a per-repo thesaurus from two signals: (1) abbreviation rules over the repo's own identifiers (btn to button, calc to calculate, msg to message, amt to amount, idx to index, repo to repository), derived by matching short identifiers against their longer co-occurring forms; (2) a distributional thesaurus over identifier and comment co-occurrence (the `DistributionalThesaurus` in `Fuse.Fusion.Scoping` exists from the V2 lexical era and is unplugged), so terms that appear in the same chunks become weak synonyms. Persist the thesaurus alongside the index. At query time, expand the query with the top-weighted synonyms (a bounded, low-weight expansion, like pseudo-relevance feedback but from the learned map rather than the top hits). Deterministic and offline.
- Tests: a known abbreviation expands to its full form and localizes the file; the thesaurus is built deterministically and persisted across disposal; expansion is bounded so it cannot flood the candidate pool.
- Docs: `internals/scoping-internals.mdx` (the learned thesaurus), `reference/configuration-keys.mdx` (any lever).
- Benchmark: Suite C, expecting the natural-language and identifier buckets to gain where the repo's naming is consistent; report the ceiling honestly (thesaurus quality is bounded by the repo's own naming discipline).

## S5. Structural priors: co-change and centrality

Use the structure Fuse already records to rank ambiguous candidates, no model.

- Work: blend two priors into the candidate score through the existing noisy-or scorer. (1) Git co-change: the `git_cochange` table (path, count, pmi, jaccard) is collected but not used on the open-ended path; boost a candidate when it co-changes with a stronger candidate, which recovers the sibling files of a multi-file change. (2) Dependency centrality: a degree or PageRank prior over the semantic graph (the `GraphCentrality` helper exists, unplugged), so a central, widely-depended-on file outranks a leaf for an otherwise-tied query. Both priors are small multipliers, capped so they tune rather than dominate. The `FUSE_GIT_CHURN_WEIGHT` lever shows the shape; co-change and centrality are the recall-oriented additions.
- Tests: a co-changing sibling of a strong hit is boosted into the returned set; a central file outranks a leaf for an ambiguous query; the priors are bounded (a high prior cannot by itself promote an irrelevant file).
- Docs: `internals/scoping-internals.mdx` (the priors and their caps), `reference/configuration-keys.mdx`.
- Benchmark: Suite C multi-file recall (per-PR file completeness, not just any-hit), and Suite B blast-radius precision.

## S6. Dependency-scoped incremental re-index (freshness)

The freshness fix. Today incremental refreshes only a changed file's own rows; the cross-file semantic edges go stale until a full re-index. That gates S7 (graph-aware discovery) and degrades the iterative-exploration story exactly when an agent is editing.

- Work: when a file changes, re-analyze that file plus the bounded set of files that reference its declared types (its project and direct dependents), and recompute only the affected semantic edges (DI, route, MediatR, EF, hosted, pipeline, decorator, factory, endpoint), leaving the rest of the graph intact. The semantic edges are cross-file, so the changed file's compilation neighborhood must be re-evaluated, but it is bounded, not the whole solution. Optionally, debounce a background full re-index after a burst of edits settles, serving the slightly-stale graph in the meantime with a freshness marker the caller can read. Honor the AGENTS.md change-safety invariants.
- Tests: editing a file to add a DI registration makes the new resolve edge appear after the incremental call (not just the symbol rows); an unrelated edge elsewhere is unchanged; the re-analyzed set is bounded to the dependent neighborhood (an invariant test, not the whole graph); a deleted file's edges are removed.
- Docs: `project/performance.mdx` (replace the current "syntax rows only" caveat with the dependency-scoped semantics and a freshness number), `internals/pipeline.mdx`.
- Benchmark: the `performance` suite gains a freshness measurement (single-file incremental that updates edges, time under 1 second) and a correctness check that the post-incremental graph matches a full re-index on the changed neighborhood.

## S7. Graph-aware discovery

Now that the graph is fresh (S6), let a weak text or dense seed inherit its neighborhood, so a vague query that barely hits one file pulls in its callers, implementers, and configuration.

- Work: after candidate generation and scoring, expand the top one or two seeds through the typed semantic graph to a shallow depth, admit the neighbors at a decayed score, and re-rank the enlarged set (reuse the `GraphExpansionEngine` already used by review and context, applied to the localize path). Bound the expansion so a single weak seed cannot pull in a large subtree. This is where the moat starts helping discovery, not only resolution.
- Tests: a query that lexically hits only an interface returns its implementation and a caller through expansion; expansion is bounded by depth and score; with no semantic graph (syntax mode) the path degrades to lexical without error.
- Docs: `concepts/scoping.mdx`, `internals/scoping-internals.mdx` (expansion on the discovery path).
- Benchmark: Suite C recall, especially on .NET-wiring-shaped queries; confirm precision does not collapse (the cutoff from S3 and the bounded expansion keep it in check).

## S8. Iterative exploration primitives

Accept that agents explore rather than ask one clean question. Give them cheap, fresh, structural navigation so a weak first guess becomes a strong few-call funnel.

- Work: expose first-class, deterministic operations an agent calls iteratively: the neighborhood of a set of files (graph neighbors plus co-changers plus same-folder cohesion), the callers and implementers of a symbol, and the structurally central files of an area, each returned ranked and bounded with provenance and no bodies. These compose the priors (S5), the fresh graph (S6), and expansion (S7) into tools shaped for exploration. Keep them warm and fast (tens of milliseconds) so an agent can chain several cheaply.
- Tests: neighborhood expansion returns callers, implementers, and co-changers of a seed, ranked and bounded; central-files returns the high-degree files of a folder; each operation is body-free and provenance-tagged.
- Docs: `reference/mcp-tools.mdx` (the exploration tools), `scenarios` (an iterative exploration walkthrough).
- Benchmark: Suite D (agent), measuring whether the funnel reaches higher file recall at fewer tool calls and fewer tokens than native exploration, on the wiring-shaped and the prose-shaped tasks.

## S9. Offline learning-to-rank and a richer-query eval

Tune the now-richer feature set without a query-time model, and measure the realistic case. With the S3 contract in place, the richer-query (code-anchored) regime is the primary measure and the bare-title regime is the refusal test, so this item's eval half is also what validates the contract's framing. The learning-to-rank half is for anchored queries; it is not aimed at rescuing vague input.

- Work: (1) Learning-to-rank. Train a small linear or gradient-boosted ranker on the PR ground truth, with features per candidate (exact-name match, subword match, stemmed match, dense similarity, path match, co-change prior, centrality prior, signal bucket), and ship the learned coefficients. Scoring at query time stays deterministic arithmetic over the features, no model loaded; the learning is offline and the result is a static weight table. This replaces or feeds the hand-tuned noisy-or weights. (2) Richer-query eval. The benchmark feeds bare PR titles, the worst case. Add a deterministic richer-query mode that synthesizes a developer-style query per PR from the diff (the changed symbol names plus a one-line description extracted from the diff, no model), so we measure the realistic exploration input as well as the title-only floor, and report both.
- Tests: the learned weights reproduce or beat the hand-tuned baseline on a held-out slice; scoring from coefficients is deterministic; the richer-query generator is deterministic and produces a query distinct from the title.
- Docs: `internals/scoping-internals.mdx` (the learned weights and how they are trained offline), `project/benchmarks.mdx` (the two query regimes).
- Benchmark: Suite C overall and by bucket under both the title-only and the richer-query regimes, no-model.

## S10. Multi-language foundation (make the seam load-bearing)

Today the moat is C# and Roslyn only. This item does not add a second language; it makes the architecture ready for one and proves the seam with a syntax-tier spike, so a later language drops in without reworking the engine or the V3.1 retrieval features. The plugin projects already exist (`Fuse.Plugins.Abstractions`, `Fuse.Plugins.Languages.CSharp`, `Fuse.Plugins.Languages.CSharp.Roslyn`, `Fuse.Plugins.Formats.Web`); the work is to make the indexer and retrieval consume that seam generically rather than hardwiring Roslyn, and to add the spike.

- Work:
  - Define a language-provider capability surface in `Fuse.Plugins.Abstractions`: discovery (the file extensions a provider claims), a syntax-tier extractor (symbols and chunks from a file, no compiler), and an optional semantic-tier extractor (a typed-graph builder backed by a compiler or language server). Declare a capability level per provider (syntax-only, or semantic) so the indexer records the achieved level the same way it records semantic, partial, and syntax today.
  - Refactor `SemanticIndexer` to select providers by file extension and call the generic seam, with the existing Roslyn path becoming the C# provider behind that seam (no behavior change for C#). The MSBuild and Roslyn loaders stay inside the C# semantic provider, not in the shared indexer.
  - Add a `language` tag to the index: a column on files, symbols, and graph nodes, and include the language in node ids where collisions are possible, so a multi-language graph and retrieval can filter or blend by language. Bump the schema version per the change-safety invariants.
  - Namespace the wiring analyzers by language: the C# DI, route, MediatR, EF, and endpoint analyzers register under the C# provider, and the seam lets a future provider register its own wiring detectors (for example a Python web-framework route analyzer) without touching the shared runner.
  - Keep every V3.1 retrieval feature language-agnostic by construction: the subword indexing (S1), stemming and comment field (S2), co-change and centrality priors (S5), graph-aware discovery (S7), exploration primitives (S8), and the signal-sufficiency contract (S3) all operate over the generic symbol, chunk, and edge tables, so they carry to a new language for free. This item verifies that and fixes any place a feature reached into the C# path directly.
  - Land a second-language spike at the syntax tier only: a minimal provider for one additional language (for example TypeScript or Python) that extracts symbols and chunks (tree-sitter or a bounded regex lexer, offline, no language server), enough to prove the pipeline indexes, full-text-searches, and localizes a non-C# file end to end. The semantic graph for that language is explicitly out of scope here; the point is the seam, not the second moat.
- Tests: a fake or spike provider registered alongside C# is selected by extension and indexes a non-C# fixture to symbols and chunks; the achieved capability level is recorded; a localize over a mixed-language fixture returns files from both languages; the `language` tag is set on files, symbols, and nodes; the C# golden outputs and Suite A are unchanged after the refactor (a no-regression guard); one V3.1 retrieval feature (for example subword search) is shown to work over the spike language without C#-specific code.
- Docs: a new internals page on the language-provider architecture and how to add a language (the capability surface, the syntax versus semantic tiers, where wiring analyzers register), and a note in `internals/pipeline.mdx` that the indexer is provider-driven. Update `AGENTS.md` design invariants to state the seam.
- Benchmark: a no-regression run of Suite A and the C# suites confirming identical numbers after the refactor, plus a small spike-language fixture scored for symbol and chunk extraction and a localize recall sanity check, recorded as the multi-language readiness baseline (not a full semantic benchmark for the new language).

## Honest ceilings

- The contract handles the vague-input ceiling, but it has its own ceilings. Refuse-and-route only helps a client that can act on the refusal; a non-cooperative client gets the graceful best-effort set instead, which is back to the old quality bar, so we cannot fully escape the recall problem for those callers. And the contract introduces a new way to be wrong: a false rejection of a borderline-but-answerable query. The classifier that draws confident, partial, and insufficient apart is therefore the thing to measure and tune (R3 scores the clear no-signal case at F1 1.0; the borderline line is the new work), and the refusal must always carry a navigation map or it just bounces the agent.
- A vague one-shot query still has an information ceiling. Even with the contract, when a client cannot refine, deterministic methods bridge the vocabulary gap and exploit structure but cannot recover a file the query gives no signal for. The realistic target is: refuse-and-route well on no-signal input, beat the lexical floor on identifier-rich anchored queries, lift natural-language meaningfully, raise precision substantially, and turn the free-exploration case into a strong few-call funnel.
- Recall and precision trade on the open-ended path. Subword indexing, stemming, thesaurus expansion, and graph expansion all widen recall and pressure precision; the calibrated cutoff (S3) and the learned ranker (S9) are the counterweights. Report the frontier at more than one operating point rather than claiming both ends.
- Thesaurus and co-change quality are bounded by the repo. A repo with inconsistent naming or a shallow git history gives weaker priors; these are aids, not guarantees, and must be capped so they tune rather than dominate.
- Freshness is bounded by the cross-file nature of semantic edges. A single-file edit can in principle affect distant edges; the dependency-scoped re-index covers the bounded common case (the file's project and direct dependents) and a debounced full re-index covers the rest, with a freshness marker so a stale answer is never presented as fresh.
- All of the above must hold with no model present. The local dense channel (R2) composes on top and adds where prose and code share no tokens at all, but it is never the floor: the deterministic path is the guarantee, the dense path is the bonus.
