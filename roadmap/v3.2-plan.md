# Fuse V3.2 Plan: finish the warm-server experience, make the seam load-bearing, lift the corpus ceiling

V3 built the .NET semantic moat (wiring resolution, change-impact review, hybrid retrieval, abstention, warm millisecond latency); its plan and record are in [v3-plan.md](v3-plan.md) and [v3-overhaul-history.md](v3-overhaul-history.md). V3.1 ([v3.1-plan.md](v3.1-plan.md)) made dense retrieval the default and offline, served the first answer fast, added the git co-change signal, widened language breadth at the syntax tier, and rewrote the positioning honestly. It also deferred, on purpose, the halves whose remaining work was large or risky: the resident Roslyn workspace, dependency-scoped freshness, the semantic-tier provider seam, the tree-sitter native engine, and the demo video.

V3.2 is the finishing wave. It closes those deferrals where the value is high (the warm-server speed-and-freshness experience), lifts the one ceiling that actually bounds the open-ended numbers (corpus index mode), and leaves the genuinely-speculative refactors gated until a concrete need exists. The honesty conventions are unchanged: every number sourced to `tests/benchmarks/results`, weaknesses published, no head-to-head claim the harness does not back.

## Where V3.2 starts (recorded V3.1 result)

All from `tests/benchmarks/results`, counted with `o200k_base`, on the rebuilt corpus (Scrutor, Ardalis.Specification, NodaTime, eShopOnWeb, plus the SampleShop fixture).

- Wiring resolution (Suite A): 22 of 22 edges, recall and precision 1.0 on the fixture. The moat, unchanged.
- Change impact (Suite B, `review.json`): 100 percent changed-file recall at 79.8 percent precision in a median 958 tokens over 53 PRs.
- Open-ended localization (Suite C, `localize.json`, dense default plus co-change prior): about 15 percent changed-file recall; the contract metrics hold (low-signal detection F1 1.0, false-rejection 0 percent, precision-when-confident bounded by the corpus). The ceiling is index mode: the localize main checkout loads partial 2, syntax 2.
- Cold start (`performance.json`): syntax tier served in about 20 seconds versus about 70 seconds for the full semantic pass; the background upgrade is opt-in (`FUSE_BG_UPGRADE=1`).
- Multi-language: C# (semantic), Python and JavaScript/TypeScript (syntax tier, offline regex providers); a `language` tag on files.

## W1. Resident Roslyn workspace, default cold start, and S6 freshness

The single highest-value follow-up. V3.1 serves the syntax tier in about 20 seconds and upgrades in the background, but only opt-in, because a detached background task in the serve host proved easy to orphan and to race teardown, and because each call still re-evaluates MSBuild. A resident workspace fixes both, and is the prerequisite S6 needs.

- Work: hold one `MSBuildWorkspace` (or the loaded project set) resident in the `mcp serve` host, with explicit lifetime management, so the MSBuild evaluation is shared across calls rather than re-paid; make the syntax-first cold start the default once the background upgrade is owned by the resident host (retire the `FUSE_BG_UPGRADE` opt-in gate); then S6: when a file changes, re-run the wiring analyzers over only the changed file's compilation neighborhood (its project and direct dependents) and recompute only the affected semantic edges, leaving the rest of the graph intact, with a freshness marker the caller can read.
- Tests: the resident workspace is reused across two index calls without a second MSBuild load (an invariant or timing assertion); the default cold path serves syntax-first then upgrades, with no orphaned process and no teardown race (the failure mode V3.1 hit); editing a file to add a DI registration makes the new resolve edge appear after the incremental call (not just the symbol rows); an unrelated edge elsewhere is unchanged; the re-analyzed set is bounded to the dependent neighborhood; a deleted file's edges are removed.
- Docs: `project/performance.mdx` (the few-seconds cold start, the resident workspace, and the S6 freshness number replacing the syntax-rows-only caveat), `internals/pipeline.mdx`.
- Benchmark: the `performance` suite gains a resident-workspace cold-start number (target a few seconds) and an S6 freshness number (single-file incremental that updates edges, under 1 second), plus a correctness check that the post-incremental graph matches a full re-index on the changed neighborhood.

## W4. Semantic-mode corpus coverage

The honest ceiling on every graph-dependent number (localize recall, the co-change and centrality priors, graph-aware discovery) is that the corpus loads mostly syntax on the localize main checkout. This is the lever that actually moves those numbers, and it unblocks W5.

- Work: diagnose and fix the main-checkout index-mode shortfall (Scrutor's NU1507 nested-NuGet-config conflict, eShopOnWeb's main-checkout syntax load), by repinning, adjusting the in-tree NuGet config, or replacing repos, so the localize main checkout reaches semantic or partial on the majority. Keep the misleading-maintenance-title drop rule when regenerating `prs.json`. Record the achieved index-mode distribution.
- Tests: the corpus manifest loads; each pinned repo reaches at or above partial on the main checkout (or is marked syntax-only with a reason); `prs.json` parses.
- Docs: `project/benchmarks.mdx` (the corpus table and the lifted index-mode distribution), `AGENTS.md` corpus description.
- Benchmark: re-run Suites A through C; the graph-dependent items (co-change, centrality, graph-aware discovery) should now move where they were flat, and the open-ended recall ceiling should rise.

## W2. Semantic-tier provider seam (the deferred S10b half)

V3.1 landed the language tag; the typed-graph builder is still hardwired to C#/Roslyn. This makes the seam load-bearing without adding a second semantic language.

- Work: define the semantic-tier provider capability so a language can supply a typed-graph builder; move the MSBuild and Roslyn loaders behind a C# semantic provider with no behavior change; namespace the wiring analyzers (DI, route, MediatR, EF, hosted, pipeline, decorator, factory, endpoint) by language so a future provider registers its own detectors; add a `language` column directly on the symbol and node rows (rather than inherited by file_id join) and a localize language-filter option on the request. Honor the change-safety invariants (schema bump, protocol version if a DTO changes).
- Tests: the C# golden outputs and Suite A are unchanged after the refactor (the no-regression guard); a localize request can filter by language; the language column is set on symbols and nodes; a fake semantic provider registered alongside C# is selected and contributes edges without touching the C# path.
- Docs: the internals language-provider page (the semantic tier and the registration), `AGENTS.md` design invariants.
- Benchmark: a no-regression run of Suite A and the C# suites confirming identical numbers after the refactor.

## W3. Tree-sitter native engine (the deferred A5 half)

V3.1 delivered JS/TS and Python breadth with offline regex providers. Tree-sitter raises extraction accuracy and adds languages cheaply, but it carries native grammar libraries that risk the self-contained publish.

- Work: add a tree-sitter-backed `ILanguageSyntaxProvider` for the top languages behind the existing seam, replacing or augmenting the regex providers; bundle the native grammars per runtime identifier (win-x64, linux-x64) and verify the self-contained publish smoke test passes on both before merging. The semantic typed graph for these languages stays out of scope.
- Tests: a tree-sitter provider extracts symbols and chunks from a fixture per added language; a mixed-language workspace indexes and localizes from all of them; the C# numbers are unchanged; the publish smoke test is green on both RIDs (the gating risk).
- Docs: the internals adding-a-language page (the tree-sitter tier), a `reference/mcp-tools.mdx` note.
- Benchmark: a small multi-language fixture scored for symbol and chunk extraction and a localize sanity check per language, recorded as the breadth baseline.

## W5. Richer-query eval and offline learning-to-rank (the deferred S9)

Gated on W4. Once the corpus loads semantic or partial broadly, the anchored-query regime is worth isolating and the fusion weights are worth tuning.

- Work: (1) a deterministic richer-query eval mode that synthesizes a developer-style anchored query per PR from the diff (changed symbol names plus a one-line description, no model), so the realistic exploration input is measured alongside the title-only floor; (2) an offline learning-to-rank pass over the PR ground truth (features per candidate: exact-name, subword, stemmed, dense, path, co-change, centrality, signal bucket) shipping static learned coefficients, with query-time scoring still deterministic arithmetic.
- Tests: the richer-query generator is deterministic and distinct from the title; the learned weights reproduce or beat the hand-tuned baseline on a held-out slice; scoring from coefficients is deterministic.
- Docs: `internals/scoping-internals.mdx` (the learned weights, trained offline), `project/benchmarks.mdx` (the two query regimes).
- Benchmark: Suite C overall and by bucket under both regimes on the W4 corpus.

## W6. Codebase-derived thesaurus (the deferred S4)

Only if W4 and W5 show a residual vocabulary gap on anchored queries that subwords, stems, and dense did not close. Otherwise defer indefinitely: a thesaurus widens the candidate pool and pressures precision, and the V3.1 evidence is that the gap is index mode, not vocabulary.

- Work: build a per-repo thesaurus offline from abbreviation rules over the repo's own identifiers and a distributional thesaurus over identifier and comment co-occurrence; expand the query with bounded, low-weight synonyms at query time. Deterministic and offline.
- Tests: a known abbreviation expands and localizes the file; the thesaurus is deterministic and persisted; expansion is bounded so it cannot flood the pool.
- Docs: `internals/scoping-internals.mdx`, `reference/configuration-keys.mdx`.
- Benchmark: Suite C, with the precision cost reported honestly alongside any recall gain.

## Sequencing

1. W1 first: the resident workspace plus default cold start plus S6 is the highest felt-value item and unblocks the warm-server story.
2. W4 next: lifting the corpus index mode is the only lever that moves the graph-dependent recall numbers, and it unblocks W5.
3. W2 and W3 are architecture and breadth; do them when a second semantic language (W2) or a breadth push (W3) is actually prioritized, not speculatively. W3 must not merge until the self-contained publish is verified on both RIDs.
4. W5 after W4. W6 only if W5 still shows a vocabulary gap.
5. G1 whenever a person can record and publish. G2 when the compute budget allows.

## Honest ceilings

- The cold-start and freshness wins depend on a correctly-managed resident workspace; a leaked workspace or a stale graph presented as fresh is worse than the current honest opt-in, so W1 ships only with the lifetime and freshness-marker tests green.
- Open-ended recall is bounded by index mode, not by cleverness. W4 is the real lever; W5 and W6 tune within whatever ceiling W4 establishes, and the recall-versus-precision trade is reported at more than one operating point.
- Tree-sitter (W3) is a publish-gate risk before it is a feature; it does not merge until the self-contained publish is green on win-x64 and linux-x64.
- The semantic-tier seam (W2) is architecture-readiness; its value is realized only when a second semantic language exists, so it is justified by a concrete plan for one, not by generality alone.

## Reminders and conventions (read before starting a W3.2 item)

- One item at a time, each as engine plus tests plus website docs (under `site/content/docs`) plus a benchmark in a single change. Run the three gates every item: `dotnet build Fuse.slnx -c Release`, `dotnet test Fuse.slnx -c Release --no-build`, `dotnet format Fuse.slnx --verify-no-changes`. New public API needs XML docs. New tests must actually run (the count rises). Plain ASCII prose only (a stop hook rejects em dashes, smart quotes, and emoji outside code fences).
- Change-safety (AGENTS.md): any `Fuse.Cli.Rpc` DTO or `[JsonRpcMethod]` change bumps `FuseHostService.ProtocolVersion` and `ext/vscode/src/host/protocol.ts` `PROTOCOL_VERSION` together, and updates the extension client and the contract test, in the same change. Any index schema change (W2 symbol/node language columns) bumps the schema version. Bound external-process arguments. No silent behavior changes. Never fabricate or weaken a benchmark number; quote only numbers recorded in `tests/benchmarks/results`.
- After each item: write results to `tests/benchmarks/results/*.json`, resync `AGENTS.md` measured results and the website docs in the same change with semantic-engine framing, tick the box in this file, append a timestamped progress-log entry (Status, Result, Verification, Blockers, Lessons, Time), commit with the item id, push. Keep a single open PR; never merge, self-approve, or enable auto-merge.
- If an item is genuinely blocked or not warranted by the evidence, log the concrete reason, leave the box unticked, and surface it, as V3.1 did for S4, S9, and the deferred halves.
- Environment facts: the build SDK is .NET 10 (`global.json` pins `10.0.100`); the CI vulnerability audit suppression lists in `Directory.Build.props` and `ci.yml` must stay in sync when touching dependencies.
