## Product philosophy (4.2)

Fuse is primarily an **agent surface**, not a tool humans drive directly. Humans install it
once (`fuse mcp install`); agents call MCP tools for the rest. The product bar is **it just
works**:

- No env-var archaeology for a normal Cursor or Claude Code session.
- Upgrade, schema drift, and corrupt derived data self-heal without asking the user to delete
  `.fuse` or read a stack trace.
- Operational states (`index_busy`, `upgrade_pending`) are returned to the agent as structured
  text the model can retry on, not opaque MCP failures.
- Breaking Fuse or index contract changes rebuild derived state automatically; source is never
  touched.

CLI and operator docs remain for maintainers and CI; defaults and `fuse mcp install` target the
agent path first.

## Execution protocol

Every checklist item (R1 through R65) follows the same contract. An item is `[x]` only when
**all** of the below are true. `[>]` in progress; `[!]` blocked (progress log names blocker);
`[-]` descoped only via an explicit written decision in this file's progress log, never by
shipping half an item.

### Per-item deliverables (all required)

1. **Engine** - the full behavior described in the item body; no "phase 1" or "MVP" left for a
   later row unless that later row is explicitly a separate checklist item.
2. **Tests** - automated tests that fail when the behavior is removed or regresses. Confirm the
   test command discovers and runs them (the count goes up or an existing test is strengthened).
   No item ships with "tests: none" except pure doc-deletion items (R2), which still carry a
   scripted verification step.
3. **Docs** - every user-visible or agent-visible behavior is documented on `fuse.codes` (MDX
   under `site/content/docs`) and named in `CHANGELOG.md` when behavior or defaults change.
   Maintainer-only items (R2 `briefing.md`, R8 profile artifact) still update the cited doc
   paths. `AGENTS.md` design invariants when the item touches architecture.
4. **Three gates** - `dotnet build Fuse.slnx -c Release`, `dotnet test Fuse.slnx -c Release
   --no-build`, `dotnet format Fuse.slnx --verify-no-changes`, plus the item's Validation
   commands and Gate criterion.
5. **Progress log** - append an entry (format in [v4-plan.md](v4-plan.md) Execution protocol):
   Preconditions, Shipped, Commands (pasted output), Tests (file paths and what each proves),
   Docs (paths touched), Deviations (must be empty for a PASS), Gate (criterion -> PASS).

### Canonical decisions and metrics

The settled Decision records and Metrics dictionary remain in [v4-plan.md](v4-plan.md) under
"Decision records" and "Metrics dictionary." They bind this addendum too. In particular, the
new performance work retains diffs-out tree ownership (D2), build capture as the semantic spine
(D7), graded verification honesty (D11), and one daemon per root with daemon-less fallback (D13).
This addendum does not change the canonical suite definitions.

### Performance metrics for R59-R65

- `parallel_gate_failures`: failures in each required full parallel test-suite run. Required value:
  zero.
- `multi_tfm_union_loss`: declarations in the full-TFM union absent from the canonical-TFM index.
  Required value: zero.
- `project_semantic_ready_ms`: product-entry elapsed time until the requested project-reference
  closure is semantic-ready.
- `compiler_owner_count` and `compiler_owner_rss_bytes`: root-level compiler-state ownership-domain
  count and aggregate host-plus-bounded-worker process-tree working set after refactor, doctor,
  and check paths have been exercised.
- `semantic_projection_lag_ms`: time from a settled watcher batch to matching persisted semantic
  query output.
- `syntax_artifact_hit_rate`: materialized syntax artifacts served from the same-repository cache
  divided by eligible artifacts; record second-worktree syntax-index elapsed time beside it.
- `hook_end_to_end_ms`: process start through hook output, reported as P50/P95 for hosted and
  no-host paths.

Except for test counts and failures, these measurements are environment-dependent. Record their
unrounded before/after samples in the progress log, never in canonical result files unless an
item's gate explicitly re-records one.

### Hard bans

- **No partial implementation.** If scope must split, add a new checklist row in the same
  session; do not mark the original row done with a tail.
- **No deferral inside an item.** Optional tests, optional docs, and "future work" paragraphs
  in item bodies are forbidden. Either the item includes them or the item is descoped in writing.
- **No silent behavior changes.** Defaults, throw behavior, tool output shape, and on-disk
  formats are named in `CHANGELOG.md` with migration (rebuild, reconnect, reconfigure).
- **No untested code paths** on the shipped default. Opt-in flags may stay off by default but
  their code path still has tests when the item introduces or changes them.

## Post-4.1 dogfood hardening (R22-R29)

These items come from dogfooding the 4.2 build end to end: bumping the version, installing the

## R22. Decouple index freshness from the product version

Reproduction: after `build/set-version.ps1 4.2.0`, the 4.2 daemon opened the repo's 4.1-built
`.fuse/fuse.db` and every read tool returned empty (`fuse_find` 0 symbols, 0 paths; status
"index mode: unknown"). Cause: `FuseBuildInfo.IsCompatible` compares the stamped `fuse_version`
by major.minor, so every minor product bump marks the index incompatible even when the extraction
contract is unchanged. With `FUSE_AUTO_UPDATE` on by default, a routine minor bump silently
discards a good index for every user. The schema version (`WorkspaceIndexSchema.TargetVersion`)
already gates structure; it does not gate extraction content, which is why the product-version
stamp was pressed into service as the catch-all. This supersedes R21's "keep `IsCompatible`
major.minor rebuild" line: dogfood showed that policy over-rebuilds on cosmetic bumps.

- Work: introduce an explicit extraction-contract version (an integer constant, for example
  `WorkspaceIndexSchema.ExtractionContractVersion`, stamped as `index_extraction_version` in
  `index_meta`), decoupled from the product version. Gate index reuse on schema-version match AND
  extraction-version match only. Remove the `FuseBuildInfo.IsCompatible` product-version gate from
  the index-open path (keep the `fuse_version` stamp for diagnostics, but it no longer forces a
  rebuild). Bump the extraction-contract version in the same change as any extractor behavior
  change (symbol/edge/chunk/route semantics); document that discipline so a forgotten bump is the
  only failure mode, never routine over-rebuilding.
- Tests: `Fuse.Indexing.Tests` - a store stamped current schema+extraction version but a different
  `fuse_version` (major or minor) is reused (no rebuild, non-empty reads); an older
  extraction-version stamp rebuilds; an older schema-version stamp rebuilds or migrates. A
  reflection/grep test that the extraction-version constant is read by the indexer so a bump is
  visible in review.
- Docs: `AGENTS.md` (amend the "ride the Fuse-version stamp" invariant to "ride the
  extraction-contract version"; record the decision), `internals/caching-internals.mdx`,
  `CHANGELOG.md` (behavior change: minor upgrades no longer force a reindex, with the migration note).
- Gate: a simulated minor bump (same schema+extraction version, different `fuse_version`) reuses
  the index and serves non-empty reads; an extraction-version bump forces a clean rebuild; both by test.
- depends: R23.

## Resilience milestone (R30-R37)

These enforce one invariant across every tool: a call returns its graded best, or an honest abstain
that defers to native and says why, never silent-empty and never worse than native. R30 (abstain and
defer to native when not semantic-ready, with an opt-in lexical fallback for fuse-only setups) and
R31 (self-verifying index) are the backbone: they make silent-empty and served-broken structurally
impossible. R32 is the standing guard; R33-R37 are supporting hardening. R30 and R31 depend on R23's
working rebuild path.

## R30. Use fuse only at its best: abstain and defer to native when not semantic-ready

Rationale: fuse's value is the graded semantic answer. When it cannot reach that grade (cold,
rebuilding, broken, or contended past the budget), it should abstain and defer to the agent's native
search rather than serve a diluted result, exactly as `fuse_check` abstains ("cannot verify, here is
what is missing") instead of guessing. This keeps every fuse tool consistent (graded best, or honest
abstain), keeps the brand undiluted (an agent never mistakes a lexical hit for a semantic one), and
does not make fuse maintain a grep it cannot beat. It satisfies never-worse-than-native by
construction: deferring instantly is not slower than native (the agent uses its own search, which is
native), and when fuse does answer it is better than native. The dogfood's silent-empty failures are
still impossible, because a not-ready read returns a fast, actionable signal, never silence.

- Work: a bounded cold-read ladder whose default is abstain-and-defer, never a silent block or empty.
  - Rung 1, serve semantic within a tight budget: on a cold or not-ready read, warm the queried scope
    (focus-first, R38) and wait up to a short deadline; if it completes, return the semantic answer.
    This is the common case once R38 is good.
  - Rung 2 (default), abstain and defer if the deadline passes: return a fast structured signal, not
    a result: the R20 index_state plus files_indexed and ETA (R37), plus an actionable hint ("index
    warming, use your native search meanwhile and retry fuse once it reports ready"), graded
    `deferred (not semantic-ready)`. Fuse steps aside; the agent uses its own ripgrep. Consistent with
    `fuse_check`'s abstain.
  - Rung 3, actionable error only when even the signal cannot be produced (workspace not found,
    permission denied), never a bare failure.
  - **Opt-in inline lexical fallback** (`FUSE_LEXICAL_FALLBACK=1`, default off): for fuse-only or CLI
    environments with no native search to defer to, rung 2 instead serves a scoped, ranked lexical
    result tagged `grade: lexical-fallback (raw text matches, not semantic; retry for semantic)`:
    candidate set from `git ls-files` minus the R25 ignores, narrowed by query kind, ranked, reduced
    top-N with signature or line context, first page emitted early. Honest bound: this does not
    out-scan ripgrep at raw bytes; its win is scope plus answer shape, and docs must say exactly that.
  - Agent steering: the `--rules` output (R21) tells the agent the protocol, use fuse_find for
    semantic search, and on a `deferred` or `index_rebuilding:` or `index_busy:` signal use native
    search and retry fuse once it reports ready. This is what makes an agent actually "use fuse only
    at its best" without being prompted each time. See the R37 rule adjustment.
- Tests: default path, a cold or not-ready read returns the deferral signal (state, ETA, use-native
  hint) fast, not a block and not empty and not a lexical result; rung 1 returns semantic when the
  scope warms within the budget; with `FUSE_LEXICAL_FALLBACK=1`, a not-ready read returns scoped
  ranked lexical hits tagged `lexical-fallback` and never fewer than a raw scan (property test on the
  opt-in path); rung 3 returns an actionable error when the workspace is missing; a healthy index
  serves semantic with no deferral or fallback.
- Docs: `reference/mcp-tools.mdx` (the not-ready deferral signal and the opt-in fallback),
  `concepts/claim-grades.mdx` (the `deferred` and `lexical-fallback` grades),
  `reference/configuration-keys.mdx` (`FUSE_LEXICAL_FALLBACK`), `project/performance.mdx` (the honest
  ripgrep bound for the opt-in path), `CHANGELOG.md`.
- Gate: a not-ready read defers fast with an actionable use-native-and-retry signal by default;
  `FUSE_LEXICAL_FALLBACK=1` serves scoped ranked hits and never fewer than grep; rung 1 serves
  semantic within the budget on a fast-warming fixture; never silent-empty, never a raw internal error.
- depends: R23, R25, R37, R38.

## R31. Self-verifying index: a store failing its invariants is never served as ready

Reproduction: the 0-chunk, mode="unknown" store was reported `index_state: ready` and served empty.
A store that is internally inconsistent must never present as ready.

- Work: define index invariants and check them cheaply after every build and on open: chunks > 0
  when symbols > 0 and FTS is available; index mode is set (never "unknown"); schema-version and
  extraction-version meta present; FTS availability agrees between its sources; a periodic
  `PRAGMA quick_check` and a symbols-vs-chunks sanity probe. A store failing any invariant is

- Tests: seed stores violating each invariant (0 chunks with symbols, unknown mode, missing version

## R32. Fault-injection resilience harness

- Work: add a resilience suite that deterministically injects: daemon killed mid-index; db corrupted

- Tests: the harness is the test; the count rises by one case per fault; a regression that removes a

- Docs: `project/contributing.mdx` (how to run the resilience suite), `CHANGELOG.md` if a public

## Warm-index automation and performance (R38-R41)

The index should warm itself the moment a repo is served, stay current on every change, and be fast
to build, with no user action and no manual reindex, ever. Today indexing is lazy (first tool call or
explicit `fuse index`) and freshness is reconcile-on-read. R38 makes it eager, focus-first, and
per-scope-ready; R39 keeps it live with a watcher; R40 is the opt-in always-on tier for warmth before
a session opens; R41 cuts the raw indexing time. R38 and R39 are default-on and degrade gracefully so
the automation never blocks or breaks a read; R40 is opt-in.

## R38. Eager, automatic warm-on-start indexing

Rationale: indexing is lazy, so the first query on a cold repo pays the full cold cost (tens of
seconds on a real repo). To be automatic, the index warms the instant the repo is served, before any
tool call, with no user action.

- Work:
  - On `fuse host` / `fuse mcp serve` start for a served root, immediately kick off a background
    syntax-first index (do not wait for the first tool call); the semantic upgrade continues in the
    background. The first tool call then hits a warm index or a bounded building header (R20/R27).
  - **Focus-first (priority-ordered) indexing**: do not warm the repo uniformly. Warm the hot set
    first, in priority order: (1) the files the first query just matched (the lexical fallback hits,
    R30), (2) git-active files (`git status` plus recent `git diff`), (3) the call's path and cwd,
    then (4) the rest of the repo. Single files re-index at about 21 ms, so the hot area goes semantic
    in about a second while the repo fills in behind.
  - **Per-scope readiness, not per-repo**: readiness is scoped to the queried area. Warm and serve
    the queried scope semantically even while the rest of the repo is cold, so a cold read costs
    "index the handful of files this query touches" (seconds), not "index the whole repo" (minutes).
    The availability header reports readiness for the served scope.
  - Ship default-on via the R21 agent-first install (no env tuning). Provide a `fuse warm <path>`
    action and `FUSE_EAGER_INDEX=0` opt-out for the rare case.
  - Keep the store warm across a session, and hold the session daemon warm for a generous, bounded
    idle timeout so quick successive sessions reuse it (this captures most cross-session warmth
    without a boot service; the boot-time always-on tier is R40). Idle-shutdown reopen is fast (the
    persistent store means open plus reconcile, never a full rebuild); configurable idle timeout and
    keep-warm policy.
  - Degrade gracefully: if eager indexing is slow or the toolchain is missing, the first read gets
    the syntax tier or the R30 grep fallback with a bounded header (R27), never a block.
- Tests: a freshly served cold repo is already building (or warm) before the first tool call (assert
  a build started at daemon spawn); focus-first warms the query and git-active files before the rest
  (assert order); a query on a large cold repo returns semantic for the queried scope within the
  budget while the repo is still indexing (per-scope readiness); the first `fuse_find` returns bounded
  (syntax or building header), not blocked; `FUSE_EAGER_INDEX=0` disables eager start; reopen after
  idle shutdown is warm (no full rebuild); the session daemon stays warm within the idle window.
- Docs: `internals/daemon.mdx`, `start/connect-your-ai.mdx` (zero-config warm),
  `reference/configuration-keys.mdx` (`FUSE_EAGER_INDEX`, idle timeout, keep-warm), `CHANGELOG.md`.
- Gate: on a cold repo the index is warming at daemon start, not at first query; the first read is
  bounded; default-on with a documented opt-out.
- depends: R23, R27.

## R39. Live incremental index via file-system and git watcher

Rationale: freshness is reconcile-on-read, so the first read after edits pays the reconcile and the
index lags between reads. To be automatic and always current, watch the tree and reconcile on
change, so reads are always fresh with no per-read cost and no manual reindex.

- Work:
  - A file-system watcher in the daemon (FileSystemWatcher on Windows, inotify on Linux) over the
    served root minus the R25 ignore set, debounced (a few hundred ms), incrementally re-indexing
    created, modified, deleted, and renamed files through the single-writer `IndexCoordinator`
    (single-file syntax re-index is about 21 ms per `performance.json`).
  - Watch `.git/HEAD` and `.git/index` to catch branch switches and pulls; reconcile the changed set
    from `git diff` rather than per-file events; a bulk change above the storm threshold triggers a
    scoped rebuild.
  - Default-on when the daemon is active, with a `FUSE_WATCH=0` opt-out. Degrade gracefully where
    watchers are unreliable (network drives, dropped events): keep reconcile-on-read as the backstop
    and run a periodic full reconcile as a safety net. The watcher and indexer are read-only on the
    tree; watcher-driven writes are debounced to avoid write storms.
- Tests: editing a file makes the change visible on the next read with zero read-triggered reconcile
  work; a branch switch reconciles the diff; a bulk change triggers a scoped rebuild, not thousands
  of events; with `FUSE_WATCH=0`, on-read reconcile still keeps freshness; a simulated dropped event
  is caught by the periodic safety reconcile.
- Docs: `internals/daemon.mdx` (live index), the freshness section of an internals page,
  `reference/configuration-keys.mdx` (`FUSE_WATCH`, debounce), `CHANGELOG.md`.
- Gate: an edit is reflected within the debounce window with no read-triggered reconcile; watcher-off
  falls back to on-read reconcile; the safety reconcile catches a dropped event.
- depends: R14, R25.

## R40. Opt-in always-on warm service

Rationale: R38 and R39 make a repo warm within an agent session (the client spawns the daemon, which
self-warms and self-refreshes) and, with the generous idle timeout, across quick successive sessions.
The remaining tier is warm before a session even opens, across reboots: a background OS service that
keeps the daemon alive and pre-warms recently-used repos. This is a human-in-IDE convenience more than
an agent need, so it ships opt-in, with a documented path to promote it to opt-out once its idle cost
is measured and shown to be light.

- Work: an opt-in background service (Windows service, launchd agent, systemd user unit) installed by
  an explicit `fuse warm --service install` (never silently by `mcp install`). It keeps a store-backed
  daemon alive and re-warms a bounded LRU of recently-used repos so they are ready before a session
  opens. Guardrails, all required: store-backed only, never a resident Roslyn compilation (that stays
  opt-in per repo); a hard LRU cap on warm repos (for example the last 3 to 5 used) and no pre-warming
  of repos not opened this cycle; battery and load aware (pause the watcher and pre-warm on battery or
  under system load); idle-evict; clean uninstall (`fuse warm --service uninstall`); a first-run notice
  naming how to disable it.
- Promotion gate to opt-out: only after measuring idle RSS, idle CPU, and battery impact on a laptop
  and showing they are negligible, and only for the human-IDE target, may the default flip to opt-out
  with a first-run notice. For the agent-first default it stays opt-in (agents get warmth from R38/R39).
- Tests: install/uninstall are clean (no residue); the service re-warms the LRU set and respects the
  cap; it pauses on simulated battery/low-power; it evicts idle repos; it never installs from
  `mcp install`.
- Docs: `internals/daemon.mdx` (the always-on tier), `reference/commands.mdx` (`fuse warm --service`),
  `start/connect-your-ai.mdx` (when to enable), `reference/configuration-keys.mdx`, `CHANGELOG.md`.
- Gate: opt-in install works and uninstalls cleanly; the LRU cap and battery pause hold; `mcp install`
  never installs the service; idle cost is recorded so the opt-out promotion can be decided on data.
- depends: R38.

## R41. Faster indexing

Rationale: syntax-first is already seconds, but the semantic tier is tens of seconds to minutes
(MSBuild plus Roslyn), and the `profile-v42.json` hot path shows a large, avoidable cost. The goal is
to cut real indexing time, so warm-on-start (R38) and every reindex are cheaper. Use the recorded
profile as the precondition: re-run `fuse eval profile-v42 --repo NodaTime` and confirm the hotspots
before cutting.

- Work:
  - **Gate default-off work off the index path**: the current `profile-v42.json` hotspots are
    dominated by `GitCoChangeCollector` (`RunGitLogAsync`, the `git log` walk) at a large share of
    self time, yet the git co-change prior is default-off (Decision D6, discharged as net-negative).
    Collecting and storing co-change data on every index while the prior is off is wasted work. Gate
    the collector behind the same flag as the prior so the default index path does not run it. This is
    the single highest-value, measured cut.
  - **Parallelize syntax extraction** across cores (it is embarrassingly parallel: parse and extract
    per file with no cross-file binding), bounded by a worker cap.
  - **Focus-first, selective semantic load**: load the project that owns the hot files first and defer
    the rest of the solution, rather than a whole-solution MSBuild load up front (pairs with R38's
    per-scope readiness); upgrade the remaining projects lazily in the background.
  - **Prefer tier-1 build-capture over a live MSBuild design-time build** for the semantic graph where
    a capture is available (C2), avoiding the MSBuild workspace-load cost.
  - **Batch SQLite and FTS writes** (chunked transactions, prepared statements) and skip or sketch
    generated code (EF migrations) and oversized files (with R35) so extraction does less work.
- Tests: `fuse eval profile-v42` after the change shows the co-change frames gone from the default
  index path and a lower cold-index time; parallel syntax extraction indexes a fixture faster than the
  serial path (recorded); selective semantic load makes the hot project semantic before the whole
  solution; the ranking and semantics suites are unchanged (co-change was already default-off, so
  gating its collection must not move `ranking.json` or `semantics.json`).
- Docs: `project/performance.mdx` (new indexing numbers), `internals/caching-internals.mdx`,
  `CHANGELOG.md`.
- Gate: recorded cold-index time on NodaTime drops versus the current `profile-v42.json` with no change
  to `ranking.json` or `semantics.json`; the co-change collector does not run on the default index path.
- depends: R25.

### 2026-07-15 R46 Stabilize parallel test isolation (per-test SQLite store paths)

Preconditions (reproduced): during R42's first full-suite run, `Fuse.Indexing.Tests`
(`WorkspaceIndexStoreUpsertTests.CoChangesRoundTripAndQueryMatchesEitherColumn`) failed with a SQLite "database is
locked" at `WorkspaceIndexConnectionFactory.OpenAsync`, yet passed 129/129 in isolation - the exact R46 flake. Root
cause found: every SQLite test already uses a unique Guid store path (confirmed by auditing `_databasePath`/temp-dir
assignments across the test tree - no shared fixed path), but the teardowns cleared the wrong connection pool. The
factory's real connection string is `Data Source=<path>;Pooling=True;Default Timeout=30`
(`WorkspaceIndexConnectionFactory`), while all 50 teardown sites cleared
`SqliteConnection.ClearPool(new SqliteConnection($"Data Source={path}"))` - a DIFFERENT pool key. So the pool was
never actually cleared: pooled SQLite connections (each holding open `-wal`/`-shm` file handles) accumulated across
the ~1500 test methods, and under heavy parallel load that handle/memory pressure produced the transient
open-time lock (`busy_timeout` is set by a PRAGMA only AFTER the connection opens, so the open itself is unprotected).

Shipped (test-only, no product change): replaced the mismatched per-path `ClearPool(...)` with
`SqliteConnection.ClearAllPools()` in the teardown of all 50 test/eval-suite files. `ClearAllPools` disposes every
idle pooled connection regardless of the connection-string key, so a disposed test's handles are released promptly
instead of leaking; it only touches idle (returned) connections, never a concurrent test's in-use one, and with
unique per-test paths there is no cross-test file collision. The suite is NOT serialized (the item's do-not).

Tests: the item is the teardown change itself; the gate is the full parallel suite passing repeatedly.

- Run 2: all pass, 0 failures.
- Run 3: all pass, 0 failures.
No non-zero `Failed!` line in any of the three runs.
