This page is for operators running Fuse as an MCP server, a host, or in CI: where the index lives, how to reset it, and which environment variables affect behavior. For the store layout, see [Caching internals](/docs/internals/caching-internals).

## fuse.db lifecycle

All persistent index data for a workspace lives in one SQLite file named `fuse.db`. Placement depends on whether the workspace is inside a git repository:

| Context | Behavior and path |
|---------|-------------------|
| Inside a Git repo | `{repoRoot}/.fuse/fuse.db`. The nearest enclosing `.git` directory or file defines the canonical root, so every nested call shares one warm index, daemon, and writer lock. |
| Outside a Git repo | Workspace-scoped MCP tools return `workspace_identity_unresolved:` and do not start a daemon or create an index. One-shot non-MCP paths retain a per-directory fallback under `~/.fuse/roots/{hash}/fuse.db` (override the base with `FUSE_USER_DATA`). |

The database is created and populated on the first index build. MCP serving starts eager warming after repository identity resolves, and every indexed MCP read validates the warm manifest before answering. `fuse index` builds it explicitly. The MCP server and the host keep the store open across calls, so a multi-call task pays the indexing cost once and reads a warm graph afterward.

## Resetting the index

| Action | Effect |
|--------|--------|
| `fuse index --force` | Rebuilds the index from scratch for the workspace |
| Delete the store file (`{repoRoot}/.fuse/fuse.db` or `~/.fuse/roots/{hash}/fuse.db`) | Full reset; the next eligible read tool rebuilds it |

The index is entirely derived data: deleting it loses nothing that cannot be rebuilt from the source.

## Freshness after an edit

The read tools do not serve an index frozen at first call. When the store is already warm, opening it for a read scans the complete current inventory, re-indexes files that were added or edited, and hard-deletes files that were removed. A new file's symbol is found on the next read, and a deleted file no longer contributes to the stored file count or any derived table.

The reconcile refreshes a file's own rows (symbols, chunks, full-text, routes); cross-file semantic graph edges (DI, routes, MediatR, EF) are recomputed by a full index. A bulk change (a branch switch or a large pull that changes more than 300 files) is not processed one file at a time. The reconcile records a stale stamp, then the coordinator starts a full replacement automatically. No manual `fuse index` is required for recovery.

## Version control

Add `.fuse/` to your repository `.gitignore`. The directory holds derived index data (like `bin/` or `obj/`): it can grow, varies by machine, and is rebuilt on the next run. The Fuse repository ignores `.fuse/` itself; your project should too. Running `fuse init` inside a git repo appends `.fuse/` to `.gitignore` when the entry is missing.

## Schema migration and corrupt recovery

The store carries a schema version. When Fuse opens a database older than the current schema (for example a cache from an earlier major version), it drops the Fuse-owned tables and rebuilds at the current version rather than attempting an in-place migration, so an older index is replaced cleanly on the next run. MCP read tools return `index_rebuilding:` while the empty store is being repopulated; retry the tool call after a few seconds or run `fuse index` explicitly.

If SQLite reports a malformed `fuse.db` on open, Fuse deletes the derived index file and recreates an empty database (the same policy as `fuse-cache.db` recovery). The next index pass repopulates it from source. Corrupt recovery is serialized per repository root so concurrent MCP sessions do not race the delete. If corruption persists, check filesystem permissions on `.fuse/` and that no other process holds a stale lock on the WAL files (a leaked `fuse mcp serve` child, for example).

Every rebuild lands on a store that can actually search. The rebuild re-probes FTS5, re-creates the `chunk_fts` table, and stamps the index mode, so a rebuilt index is fully functional rather than a store with indexed files but no full-text search. If a store is ever found internally inconsistent (its full-text stamp says available but the table is gone, or it has symbols but zero chunks on an FTS-capable runtime), it is not reported `ready`; it reports `index_rebuilding` and the read path repairs it. A search against a store missing `chunk_fts` returns the `index_rebuilding:` prefix and never a raw `internal_error: SQLite Error`.

A completed warm index also carries its repository identity and complete inventory count and hash. Missing metadata, an interrupted `building` state, a wrong root, or stored rows that do not match the manifest triggers automatic replacement even when `files` is nonzero. This prevents a small accidental scan, such as six runtime JSON files from `bin/Release`, from being accepted as the repository index.

## Observability of degraded states

Fuse makes a degraded state loud rather than silent (R37):

- **Counts**: each degraded path increments the `fuse.degraded.state` metric (tagged by kind: `index_busy`, `index_rebuilding`, `integrity_failed`, `deferred`, `lexical_fallback`, `verify_abstained`) when `FUSE_METRICS` is on, and is tracked in-process regardless so `doctor` can report it.
- **Header hint**: every not-ready availability header (`building_syntax`, `upgrade_pending`, `index_busy`, `index_rebuilding`, `stale_as_of`) carries an actionable `hint:` line telling the agent to use its native search meanwhile and retry once `index_state` is `ready`, plus a `progress: building for ~Ns` line during a cold build.
- **Doctor**: `fuse_workspace action=doctor` reports store size and last-written time, index integrity (R31), the files skipped during indexing (R35), the running `fuse host` daemons with their served roots and versions (R28), and the degraded-state counts for this session.
- **Daemon log**: `fuse host` writes a size-rotating log at `{user-data}/logs/fuse-host.log` (rotated to `.1` past 1 MB), so lifecycle and degraded events survive a detached run.

The pull-based header remains the source of truth for progress; MCP `notifications/progress` push is a best-effort future add for clients that surface it, not required for the signal to be visible.

## Environment variables

| Variable | Effect |
|----------|--------|
| `FUSE_USER_DATA` | Base directory for machine-wide Fuse data (default `~/.fuse`). One-shot non-MCP stores are per-directory under `{FUSE_USER_DATA}/roots/{hash}/fuse.db`; workspace-scoped MCP tools refuse non-Git folders. Inside a Git repo the store stays at `{repoRoot}/.fuse/fuse.db` regardless of this variable. |

When driving an agent against the Fuse MCP server (for example in the agent evaluation suite), the Claude CLI variables `MCP_TOOL_TIMEOUT` and `MCP_TIMEOUT` bound per-tool-call and startup time; these are read by the agent client, not by Fuse.

Session-delta state (`sessionId` on `fuse_context` and `fuse_review`) is held in memory inside the running server only; it is not stored in `.fuse/fuse.db` and is lost when the process exits.
