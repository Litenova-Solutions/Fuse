This page reports what Fuse does to a real .NET codebase in numbers: how far it cuts the token count, how much of the public API surface survives that cut, and how well its scoping finds the files a task needs. Every number here comes from a harness that anyone can rerun against the same pinned commits. The goal is a credible claim backed by a script, not a marketing figure.

Three readers are served below. A stakeholder can read the headline findings and the fidelity table. A new engineer can follow the reproduction steps and run the suite. A maintainer can read the method, the corpus manifest, and the per-mode results, including the arms where Fuse ties or loses.

The benchmark is built in layers by cost and persuasiveness. Layer 1 is intrinsic and deterministic: token reduction, speed, memory, and fidelity. Layer 2 tests retrieval: scoping recall against real merged-PR change sets, and a single-turn localization task. Layer 4 is the context-acquisition scenario: Fuse, no Fuse, and Repomix on the two axes of round-trips and input tokens for the same task. A full end-to-end task-resolution harness (driving a programmatic agent across arms and scoring patches) is not built here and is listed under Blocked And Out Of Scope.

## Reproduce First

Prerequisites: the .NET SDK 10.0 or later, Git, and Node.js (the competitor packer Repomix runs through `npx`). Network access is needed once to clone the corpus.

One command runs everything from a clean checkout:

```bash
pwsh -File tests/benchmarks/harness/run-all.ps1
```

That script builds the CLI and two measurement tools, clones the corpus at the pinned commits, regenerates the PR ground truth, and runs all three layers. Results are written as JSON, CSV, and Markdown tables under the results directory. To run one layer at a time, call the per-layer scripts in the harness directory after the corpus is in place.

The inputs that make the run reproducible are committed alongside the harness: the corpus manifest pins each repository to an exact commit, the PR ground truth records base and head commits per change set, and the localization question set records each question with its known answer file.

## What Is Measured, and How It Stays Fair

Token counts use the `o200k_base` encoding through `Microsoft.ML.Tokenizers`, the same encoding Fuse uses by default. Reduction ratios transfer across models because they are ratios; absolute token counts do not, because each model tokenizes differently. Every arm in a comparison is counted with the one tokenizer, so raw concatenation, Fuse output, and Repomix output are measured on the same scale.

Three rules keep the comparison honest:

1. One file set per repository. Each tool sees a C#-only mirror of the repository, so raw concatenation, Fuse, and Repomix all process the same files. This isolates C# reduction, which is the feature under test.
2. An independent fidelity oracle. The fidelity tool parses the raw source with Roslyn to build the ground-truth set of public and protected types, public methods, and ASP.NET route templates. It does not reuse Fuse's own regex extraction, so the measure cannot flatter Fuse by sharing its blind spots.
3. No tuning for the result. Fuse runs with documented flags only. Runtime behavior was not changed to improve any number on this page.

Two regression guards back these rules:

- A body-integrity check parses each raw source file with Roslyn to collect the verbatim text of every non-trivial string literal, then verifies each survives byte-identical in the reduced output. It is the guard that a reduction never alters a literal's contents (embedded JSON, SQL, or templates), and for the body-preserving levels it also confirms the output still parses. It is measured with redaction disabled, since redaction deliberately replaces secret literals. Layer 1 records it as the Literals and Parses columns.
- A compare gate (`run-all.ps1 -Compare baseline.json`) diffs a fresh layer-1 run against a committed baseline and fails on any drop in reduction ratio, type/method/route fidelity, or body-integrity beyond a small tolerance, and on any parses true-to-false flip. A SampleShop-only smoke (`smoke.ps1`) exercises this gate without cloning the corpus.

The corpus is four open-source .NET libraries pinned by commit, plus the in-repo sample fixture for route coverage:

| Repository | Size | C# files |
|------------|------|---------:|
| MediatR | small | 152 |
| FluentValidation | small to mid | 218 |
| AutoMapper | mid | 511 |
| Newtonsoft.Json | large | 945 |
| SampleShop | micro, in-repo | 7 |

### Layer 1 Findings, Including Losses

- The no-flag default and `--level aggressive` preserve the public API. Across the four libraries, the default and `--level aggressive` keep 99 to 100 percent of public types and methods while removing 7 to 40 percent of tokens. This is the headline result: Fuse cuts tokens without dropping the surface an agent reads for.
- `--level aggressive` is the dependable cut. It removes 21 to 40 percent of tokens at full type and method fidelity. The cut is larger on repositories with heavy comments and usings (FluentValidation at 36 percent, Newtonsoft.Json at 40 percent).
- Skeleton trades signature completeness for the deepest cut, and it degrades on large complex code. Skeleton removes 66 to 93 percent of tokens. On the three smaller libraries it keeps 91 to 100 percent of types and 84 to 99 percent of method signatures. On Newtonsoft.Json it keeps only 71 percent of types and 4 percent of method signatures. That collapse is real: the regex skeleton extractor loses most signatures in a codebase with heavy conditional compilation and partial classes. Skeleton is an architecture map, not a substitute for the source, and this case is the clearest example of its limit.
- Skeleton drops route attributes. On the sample fixture, route fidelity falls to zero under skeleton because skeleton removes attributes. The default and `--level aggressive` keep all four routes. Use the default or `--level aggressive` when route templates matter.
- The default reduction loses to raw on tiny inputs. SampleShop at `--level none` produces more tokens than raw concatenation because the structured wrapper and manifest cost more than whitespace normalization saves on a 7-file project. The cut turns positive once the project is large enough for reduction to dominate the wrapper cost.
- Fuse beats Repomix in every mode, including the default. Repomix output is 1.3 to 3.9 percent larger than raw concatenation on these repositories, because its preamble and structure add tokens. Fuse at the default is smaller than raw and smaller than Repomix; `--level aggressive` and `--level skeleton` widen the gap. Both tools preserve the API surface at full output, so the difference is token cost. Fuse cold wall-clock is also lower here, though Fuse pays a fixed startup cost that dominates on tiny inputs.

## Layer 2A: Scoping Recall Against Real Pull Requests

Layer 2A measures whether a scoping mode includes the files a real change touched. The ground truth is the set of C# files changed by 24 real merged pull requests across the four libraries. For each pull request the harness reconstructs the head state in a Git worktree and runs three Fuse modes plus a grep baseline. Recall@budget is measured at several token budgets (10,000, 25,000, and 50,000) so a retrieval change can be read across the budget curve rather than at one point; the 50,000 token budget remains the headline reported below. The grep baseline ranks files by query-term hit count and fills the budget, which is what an agent does when it greps and reads the top results. Recall (the necessary files included) is the headline; precision is the fraction of emitted files that were necessary.

| Mode | Mean recall | Mean precision | Mean tokens |
|------|------------:|---------------:|------------:|
| changes | 87% | 50% | 29,607 |
| focus | 71% | 5% | 46,543 |
| query | 51% | 2% | 46,366 |
| grep (baseline) | 38% | 11% | 41,452 |

Recall varies widely by repository, and the aggregate hides that:

| Repo | changes | focus | query | grep |
|------|--------:|------:|------:|-----:|
| MediatR | 100% | 100% | 94% | 94% |
| FluentValidation | 100% | 74% | 51% | 23% |
| AutoMapper | 92% | 88% | 29% | 29% |
| Newtonsoft.Json | 56% | 21% | 30% | 5% |

Findings, including losses:

- Change scoping is the strongest mode. It leads recall at 87 percent at 50 percent precision, because following reverse edges one hop reaches the files that use a changed file without pulling in a wide neighborhood. On MediatR and FluentValidation it recalls every changed file. Its mean emitted tokens (29,607) sit well under the 50,000 budget because emission now stops before the one entry that would cross the cap rather than overrunning it by that entry (strict total-token accounting); this raised precision and trimmed the mean tokens, at the cost of about one budget-crossing file per large change set (Newtonsoft.Json change recall 58 to 56 percent).
- All three scoping modes beat the grep baseline at the headline budget. Focus (71 percent) and query (51 percent) both clear grep (38 percent). Focus gains because it pulls a seed's dependents as well as its dependencies, so a changed test that references the seed is reached. Query gains from fielded BM25F ranking, query normalization, and a pseudo-relevance feedback pass that rewrites a sparse title in the codebase's declared-symbol vocabulary before selecting files.
- Query expansion lifts the repositories where the title alone is too sparse. The second pass raised FluentValidation query recall from 47 to 51 percent and Newtonsoft.Json from 27 to 30 percent at the headline budget, with MediatR (94 percent) and AutoMapper (29 percent) unchanged, so the gain does not come at the cost of a per-repository regression. It is held to a low blend weight so an off-topic title does not displace an incidental first-pass hit. The effect is smaller at tight budgets (the 10,000 and 25,000 token query rows are slightly below the single-pass values) because the extra admitted seeds compete for a small budget; at the 50,000 token headline it is a clear gain.
- Precision stays modest under forward expansion. Seeding and graph expansion still surface more files than the task strictly needed; the relevance-ordered budget cut keeps the most relevant within the 50,000 tokens rather than the largest.
- Newtonsoft.Json remains the hardest repository, but it is no longer a near-total miss. Change recall rose from 17 to 56 percent and query from 0 to 30 percent as comment and string stripping removed false graph edges, fielded ranking found concept files, and query expansion surfaced co-occurring symbol names. Deeply nested types and conditional compilation still defeat parts of the regex graph, so it trails the smaller libraries.

## Layer 2B: Single-Turn Localization

Layer 2B is a cheap, outcome-flavored signal. It asks twelve natural-language questions of the form "which file handles X," each with one known correct answer file, and checks whether a mode surfaces that file inside a 20,000 token budget. The question set is recorded so the task is reproducible.

| Mode | Accuracy | Hits | Mean tokens |
|------|---------:|-----:|------------:|
| grep (baseline) | 58% | 7/12 | 19,924 |
| focus | 42% | 5/12 | 8,078 |
| query | 42% | 5/12 | 19,203 |

Findings, including losses:

- Grep leads this task. The baseline lands 7 of 12 answer files inside the 20,000 token budget by ranking on query-term hit count. Query and focus each reach 5 of 12 (42 percent). Fielded BM25F ranking and query normalization help when the question shares vocabulary with the target file, but they do not beat a straight grep on this set.
- Focus ties query on accuracy at lower cost. Focus accuracy rose from 25 to 42 percent now that it follows dependents as well as dependencies, and it spends about 8,100 tokens against query's 19,100. When the seed is a concept rather than a type name both modes still miss.
- A lexical ceiling remains. Ranking rewards shared vocabulary, so a question that shares no words or stems with the target file still misses regardless of mode. Layer 2B is the regression guard for future retrieval improvements.

## Layer 4: Context Acquisition (Fuse VS No Fuse VS Repomix)

This layer is the proof behind the front-door claim that Fuse collapses the explore phase
into one scoped call; the concept is in [The explore phase](/docs/concepts/the-explore-phase).

Layers 1 and 2 measure Fuse in isolation. Layer 4 puts three approaches side by side on the two axes that decide the real cost of a task: the round-trips it takes to acquire the context, and the input tokens that context costs. The task is the same one Layer 2A uses, "assemble the context this change needs," and the needed files are the C# files each of the 24 real merged pull requests touched. For every pull request the harness reconstructs the head state in a Git worktree and measures three arms at the same 50,000 token budget (with 10,000 and 25,000 token points for the curve). All counts use `o200k_base`.

The three arms answer one question in three ways:

- **no-fuse** is what an agent pays with no tool: it reads the files the task needs, one at a time. Its round-trip count is the number of files the change touches, which is a structural lower bound, because a blind agent must read each needed file at least once and in practice reads more while exploring to find them. Its recall is 1.00 by construction (it is exactly the needed set). The harness also records the whole-repo token total as the "explore blind" ceiling, what an agent pays if it cannot find the right files and reads broadly.
- **Repomix** is the generic-packer answer: one dump of the whole repository. One call, recall 1.00 by construction (it contains everything), and the token cost of the entire repository.
- **Fuse** is one scoped call, `fuse --query "<PR title>" --depth 2 --level standard --max-tokens 50000`. One call, and its recall is the fraction of the changed files the scoped output actually contains. Query is the honest default here because it needs no prior knowledge of the change; the change-scoping mode in Layer 2A, which seeds on a Git base, reaches higher recall when one is available.

Means across the 24 pull requests at the headline budget:

| Arm | Round-trips | Input tokens | Recall of needed files |
|-----|------------:|-------------:|-----------------------:|
| no-fuse (blind, whole repo) | >= 5.8 | 493,661 | 1.00 |
| no-fuse (relevant set) | >= 5.8 | 27,473 | 1.00 |
| Repomix (one dump) | 1 | 511,574 | 1.00 |
| Fuse (`--query`) | 1 | 44,694 | 53% |

The round-trip number for no-fuse is a lower bound, not a measured agent: it is the mean count of files the change touched (5.8), and a real agent reads more while discovering them. Recall for no-fuse and Repomix is 1.00 by construction, so this is not a recall contest; the contest is tokens at one call (against Repomix) and round-trips against blind exploration. Always read Fuse's token number together with its recall: 44,694 tokens at 53 percent recall is the honest pairing.

Recall varies widely by repository, and the mean hides that. The per-repository view at the headline budget:

| Repo | no-fuse round-trips (>=) | no-fuse relevant set | whole repo | Repomix | Fuse | Fuse recall |
|------|-------------------------:|---------------------:|-----------:|--------:|-----:|------------:|
| MediatR | 2.7 | 3,326 | 79,282 | 84,894 | 39,398 | 94% |
| FluentValidation | 3.5 | 5,711 | 256,538 | 268,083 | 43,639 | 55% |
| AutoMapper | 2.3 | 3,342 | 456,599 | 473,904 | 47,990 | 29% |
| Newtonsoft.Json | 14.5 | 97,514 | 1,182,224 | 1,219,416 | 47,749 | 34% |

Recall rises with the budget, as expected, since a larger budget admits more of the neighborhood:

| Budget | Fuse mean recall | Fuse mean tokens |
|-------:|-----------------:|-----------------:|
| 10,000 | 35% | 9,712 |
| 25,000 | 43% | 23,383 |
| 50,000 | 53% | 44,694 |

Findings, including the honest boundary:

- One Fuse call replaces many blind round-trips. A blind agent must read at least the files a change touches, a mean of 5.8 and as many as 14.5 on Newtonsoft.Json, before it has the context; Fuse and Repomix each acquire it in one call. Fuse does not beat Repomix on round-trips, it ties it at one call.
- At that one call, Fuse costs far fewer tokens than the generic packer. Fuse delivers the scoped context in 44,694 tokens on average against Repomix's 511,574, about 11.4 times fewer, and against the 493,661 tokens of reading the whole repository blind. The Fuse number is the reduced, scoped set the task needs, not the whole repository.
- The token win is paired with partial recall, and that is the trade. Fuse query reaches 53 percent of the changed files at the headline budget, against 1.00 for Repomix and the relevant set, which include everything by construction. Where the query carries real domain words and the repository is not dominated by conditional compilation, recall is high (MediatR 94 percent); on Newtonsoft.Json and on broad changes it is lower. For pull-request review, where a Git base is available, the change-scoping mode in Layer 2A reaches 87 percent recall. The defensible claim is coupled: Fuse reaches recall R of a task's files in one call at far fewer tokens than a generic packer needs for the same one call, while blind exploration needs at least K round-trips.

Reproduce this layer with the full suite or on its own:

```bash
pwsh -File tests/benchmarks/harness/run-all.ps1
pwsh -File tests/benchmarks/harness/layer4-scenario.ps1
```

Results land in `results/layer4-scenario.{json,csv,md}`, one row per pull request, arm, and budget, so the curve and the per-repository view are reconstructable. The Repomix arm needs `npx`; when it is unavailable that arm is omitted (never stubbed), and every Fuse-versus-no-fuse number still stands.

## Retrieval and Output Changes: Before and After

This section records the effect of the retrieval work and the output and trust work against the same pinned corpus. All token counts use `o200k_base`. Layer 1 reduction and fidelity did not change: the default and `--level aggressive` still keep 99 to 100 percent of public types and methods, and skeleton still collapses on Newtonsoft.Json, so the cut was not bought by dropping API.

### Retrieval and Scoping

The six retrieval items change one scoring and expansion pipeline, so the harness measures their combined effect rather than isolating each. The before column is the previous release; the after column is this release at the same budgets and depths.

| Metric | Before | After |
|--------|-------:|------:|
| Layer 2A recall, changes | 71% | 87% |
| Layer 2A recall, focus | 26% | 71% |
| Layer 2A recall, query | 37% | 51% |
| Layer 2A precision, changes | 21% | 50% |
| Layer 2B accuracy, focus | 25% | 42% |
| Layer 2B accuracy, query | 25% | 42% |
| Grep baseline (unchanged) | 38% / 58% | 38% / 58% |

What each item contributes to that combined movement:

1. Reverse edges and dependents. Focus and changed-since now pull files that reference a seed's declared types, not only the files it references. This is the main driver of the focus recall gain and the changes precision gain (dependents replace a wider forward expansion).
2. Fielded ranking (BM25F). Declared type and member names and path tokens are weighted above the body, which moves the file that declares a concept above files that mention it. This contributes to the Layer 2A query recall gain and lifts Layer 2B query accuracy from 25 to 42 percent, though grep still leads localization at 58 percent.
3. Comment and string stripping. Removing type names that appear only in comments or strings before graph extraction cut false edges; this is the largest contributor to the Newtonsoft.Json improvement, where prose and string mentions previously polluted the graph.
4. Budget-aware, rank-decayed expansion. Best-first expansion with per-hop decay and a budget stop keeps the seed neighborhood and drops distant files first, which holds precision as recall rises.
5. Query normalization. CamelCase splitting, stopword removal, and light stemming align a natural-language question with code identifiers, contributing to the query gains in both layers.
6. Relevance-ordered truncation. Under a token budget, emission now writes most-relevant first instead of largest first, so the seed file survives the cut. This protects recall and accuracy at the 50,000 and 20,000 token budgets used here.
7. Pseudo-relevance feedback query expansion. A second BM25F pass seeded with recurring declared-symbol terms from the first pass's top files rewrites a sparse title in the codebase's vocabulary. It lifts Layer 2A query recall from 49 to 51 percent at the headline budget, concentrated on FluentValidation (47 to 51) and Newtonsoft.Json (27 to 30), with MediatR and AutoMapper unchanged.

### Output and Trust

These features are opt-in and do not change the default output, so they do not move the Layer 1 or Layer 2 arms above. Their effect is measured directly.

- Compact envelope (`--format compact`). Measured on the corpus under default reduction against XML: MediatR 85,431 to 84,732 tokens (0.8 percent), FluentValidation 277,453 to 276,389 (0.4 percent), AutoMapper 456,992 to 454,740 (0.5 percent), Newtonsoft.Json 3,155,917 to 3,151,980 (0.1 percent). The saving is the per-file envelope, so it grows with file count and shrinks as a fraction when files are large.
- Header dedup (`--dedup-headers`). On the pinned corpus the saving is negligible, because these projects do not prepend an identical comment header to every file (Newtonsoft.Json wraps its license in a `#region`, which the dedup correctly leaves alone). On a synthetic 40-file project where each file carries the same three-line banner, dedup cut 3,131 tokens to 1,989 (36 percent). The feature pays off on header-heavy codebases and is honestly near-zero here.
- Tokenizers. The Anthropic and Gemini estimators are deterministic. On a small sample the run report counted 49 tokens under `o200k_base`, 60 under `claude`, and 59 under `gemini`, reflecting their published characters-per-token ratios. These are estimates; OpenAI encodings remain exact.
- Verify. On MediatR excluding tests, `fuse verify` reports 86 of 86 public types and 36 of 36 public methods preserved under the default and `--level aggressive`. It is a trust check, not a reduction metric.
- Explain and the JSON run report. Both surface the run without changing it: explain lists included and excluded files with a token estimate, and `--report` emits a machine-readable summary that names the tokenizer. They have no token-reduction figure to report.

### Round-Trips and the Precision Tier

The round-trip features (table-of-contents survey, the `fuse_ask` auto-scope tool, review-shaped change emission, session-delta emission) and the precision tier (Roslyn analysis, symbol-level scoping, the persistent index, cross-language reduction, and generated-code collapse) are all opt-in. None of them change the default reduction or scoping path. The harness was rerun over the same pinned corpus to confirm that, and the Layer 1 token counts and fidelity and the Layer 2 recall and precision above are byte-identical to the previous release. The gains are not bought by changing the default, because the default did not change.

Each item is measured directly where the harness exercises it:

- Table-of-contents (`--toc`, `fuse_toc`). On the SampleShop fixture the table of contents is 221 tokens against a 624-token cost to read every listed file in full under the default reduction, so the survey costs about a third of fetching the files it maps. The saving grows with file count, since the table of contents lists one line plus a symbol outline per file regardless of file size.
- Round-trips: the measured bound and an illustrative model. The measured round-trip result is the Layer 4 bound above: a blind agent must read each file a task needs at least once (a structural lower bound from real change sets, a mean of 5.8 files), while Fuse and Repomix each acquire the context in one call. Subordinate to that bound, layer 3 of the harness keeps an illustrative prefill-growth model: a simulated stateless agent re-sends its whole context every turn, so naive file-by-file exploration pays a cost that grows with the square of the turns, against a `fuse_toc` survey plus one budgeted `fuse_search` (K=2) and a single budgeted `fuse_ask`-style retrieval (K=1). Those token counts come from real Fuse runs, but the agent is a model, not a measured agent, so its K and cumulative prefill are labeled illustrative of the round-trip structure, not benchmarks. A downstream task-success layer (feeding fused context to a fixed model and scoring answers) is gated behind an API key and is out of scope for the keyless harness run.
- Generated-code collapse (`--collapse-generated`). The corpus libraries contain no EF Core migrations, so this item does not move the Layer 1 numbers; it is unit-tested on a migration fixture, where the generated `Up`, `Down`, and `BuildModel` bodies are dropped while the class and method signatures are kept. It is reported here as not exercised by the corpus rather than as an improvement.
- Cross-language reduction (TS, JS, SQL). The corpus is a C#-only mirror, so this item does not move the Layer 1 numbers either; the TypeScript-family and SQL reducers are unit-tested for comment and whitespace removal. Extending the corpus with a TypeScript or SQL repository is the way to confirm a ratio, and is noted as future work rather than claimed here.
- Roslyn structural analysis. This is the accuracy result of the whole effort. Running `--level skeleton` over the same C#-only mirrors the harness uses, with the same independent Roslyn fidelity oracle, Roslyn keeps the signatures the retired regex skeleton dropped:

  | Repo | Regex skeleton | Roslyn skeleton |
  |------|----------------|-----------------|
  | MediatR | types 100%, methods 84% | types 100%, methods 100% |
  | FluentValidation | types 98%, methods 99% | types 100%, methods 100% |
  | AutoMapper | types 91%, methods 86% | types 99%, methods 100% |
  | Newtonsoft.Json | types 71%, methods 4% | types 100%, methods 100% |

  The Newtonsoft.Json row is the headline: regex skeleton kept 4 percent of method signatures, Roslyn keeps 100 percent. The cost is honest output size. Because the regex skeleton dropped most signatures, its deep cut was partly a fidelity failure, not real saving. The Roslyn skeleton is a faithful signatures-only cut and is therefore larger: Newtonsoft.Json moves from 96,953 tokens (regex, 93.4 percent reduction but 4 percent method fidelity) to 694,862 tokens (Roslyn, 52.7 percent reduction at 100 percent fidelity). MediatR moves from 26,405 to 39,865 tokens, FluentValidation from 51,890 to 115,635, AutoMapper from 159,542 to 258,427. This is the intended trade: the precision tier does not weaken fidelity to inflate reduction. Symbol-level scoping rides on the same tier and is unit-tested rather than measured over the corpus, since the corpus carries no god-class for a representative member slice.

### The Persistent Index: Cold versus Warm

The persistent index (`--index`, on by default in watch and serve) caches per-file analysis on disk so repeated scoped calls reuse it. Measured on MediatR (152 C# files) with a query-scoped run, wall-clock cold (a cleared index) versus warm (the index populated by a prior run), counted on one Windows machine and including the fixed process startup cost:

| Tier | Cold | Warm |
|------|-----:|-----:|
| Regex (default) | 1,032 ms | 626 ms |
| Roslyn | 1,127 to 1,217 ms | 611 to 644 ms |

The warm run is about half the cold run. The Roslyn tier is more expensive cold because it parses each file, and the index is what amortizes that parse: warm, the two tiers converge because neither re-analyzes an unchanged file. These figures are directional and machine-specific, like the other wall-clock numbers on this page; the index's correctness (a hit returns the stored analysis, a changed file misses) is covered by unit tests.

A note on the competitor arm: the Repomix comparison in the Layer 1 table and the Layer 4 scenario was measured live in this environment through `npx`. When `npx` is unavailable, the harness does not publish a broken stub: Layer 1 carries the prior committed Repomix row with a clear notice, and Layer 4 omits the Repomix arm for that run. The Fuse-versus-raw and Fuse-versus-no-fuse numbers do not need `npx` and stand regardless.

## How to Read These Results Together

Layer 1 is the strong, clean claim: Fuse cuts 7 to 40 percent of tokens at full public-API fidelity with the default and `--level aggressive`, cuts far more with skeleton at signature level, and beats Repomix on token cost in every mode. That claim is deterministic and holds across all four repositories.

Layer 2 is the honest boundary, and it moved this release. Change scoping is reliable because it rests on Git, and it carries moderate precision through one-hop dependents. Focus and query rest on a best-effort dependency graph and a lexical BM25F index; with reverse edges, fielded ranking, comment and string stripping, and query normalization they clear the grep baseline on the pull-request recall task. They help most when the query carries real domain words and the codebase is not dominated by conditional compilation, which is why Newtonsoft.Json still trails. On single-turn localization, grep still leads and a lexical ceiling remains when the question shares no vocabulary with the target file.

## Blocked and Out of Scope

- Layer 3, task resolution. A SWE-bench-style harness that drives a programmatic agent across arms and scores patches with a test oracle is not built here. It needs provisioned model credentials and compute, and it is confounded and scaffold-specific. It remains aspirational, as the roadmap states.
- Cross-machine timing. Wall-clock and memory were captured on one Windows machine. They are directional. Token counts and fidelity are machine-independent.
- Tokenizer scope. All counts use `o200k_base`. Absolute counts differ under other encodings; the ratios are what transfer.
