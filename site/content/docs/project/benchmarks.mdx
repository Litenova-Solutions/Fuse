title: Benchmarks
description: Reproducible measurements of Fuse as the Roslyn-backed compiler oracle for .NET agents, scored on suites over a pinned corpus of real .NET repositories, with honest reporting of where Fuse loses.
---

Fuse is the Roslyn-backed compiler oracle for .NET coding agents, working from a warm persistent index. This page reports what it does on real .NET codebases, judged by one question: can it verify an edit honestly, resolve .NET wiring, scope a change with precision, help an agent gather the right files, and avoid returning junk on a low-signal query. The yardstick is the compiler-grade engine, not tokens saved; token efficiency is a property of the scoped output (a pull request's context arrives in a median 958 tokens), not the headline.

Three readers are served below. A stakeholder can read the suite headlines and the honest-reporting section. A new engineer can follow the reproduction steps and run a suite. A maintainer can read each suite's method, sample, index-mode distribution, and the cases where Fuse is weak.

The benchmark is seven suites run by a C# eval driver in `tests/benchmarks/Fuse.Benchmarks`, each reproduced with `fuse eval <suite>` and each writing its scorecard to `tests/benchmarks/results/<suite>.json`:

- Suite A, semantic resolution: does the extracted graph match a hand-built edge ground truth. Deterministic.
- Suite B, change impact and review: does `fuse review` scope a real merged PR to the files it touches plus their semantic blast radius. Deterministic.
- Suite C, open-ended localization: given only a PR title and no Git base, does `fuse localize` find the changed files. Deterministic, and the weakest mode.
- Suite D, agent context sufficiency: under a real model, does an agent gather the task's files with the Fuse MCP tools. Model-dependent and not byte-reproducible.
- Suite E, token reduction and fidelity: how far each reduction level cuts tokens, and how much of the public API the skeleton keeps. Deterministic and offline.
- Suite F, `fuse_check` honesty: over known-good and known-bad single-file edits, how often the speculative typecheck reports a broken edit clean (a false green) or a valid edit broken (a false red). Deterministic and offline.
- Suite R4, the loop metric: how many build-gated turns an agent takes to reach green, per toolbox. Model-dependent; the harness and its deterministic turn classifier are the deliverable, and the numbers are recorded from an opt-in provisioned run.

## An honest caveat on index mode

Fuse's semantic graph is strongest when the corpus loads with a full MSBuild semantic workspace. A full load needs a restored workspace (a `project.assets.json` from `dotnet restore`); without one, indexing falls back to syntax mode and every semantic suite reads a thin graph. The harness restores each checkout before indexing with the `--restore` flag (see Reproduce) and records the achieved index mode per repository, so each number can be read in light of how much structure was available. Where a repository restores, it loads semantically; the deterministic Suite A is the semantic ceiling, and the corpus suites run mostly in partial or syntax mode, so read them against that ceiling.

## The corpus

The corpus is three open-source .NET libraries and one open-source ASP.NET Core application, all pinned by commit and chosen to restore on the .NET 10 SDK, plus the in-repo SampleShop fixture, counted with the `o200k_base` encoding. The application (eShopOnWeb) is the only non-library: it carries DI wiring, MVC and Razor Pages, EF Core, and config sprawl that the clean library APIs do not exercise, so it tests external validity beyond tidy public surfaces. PR ground truth (`prs.json`, 53 PRs reconstructed from merge history) drops merged PRs whose titles are misleading maintenance (a CI tweak or a dependency or version bump over an unrelated C# diff), since a title-only scope cannot find their change sets.

| Repository | Size | C# files | PRs |
|------------|------|---------:|----:|
| Scrutor | micro | 45 | 3 |
| Ardalis.Specification | small to mid | 233 | 14 |
| NodaTime | mid | 488 | 18 |
| eShopOnWeb (application) | mid | 254 | 18 |

One corpus-pinning caveat is recorded honestly: Scrutor uses central package management with two NuGet sources, and nested under Fuse's repository NuGet config it triggers a deterministic NU1507 source-mapping conflict and restores to syntax. NodaTime and Ardalis.Specification restore clean; eShopOnWeb restores per-PR-worktree (Suite B) but its main checkout (Suite C) loads syntax, so the localize index-mode distribution is lower than review's.

## Reproduce

Prerequisites: the .NET SDK 10.0 or later and Git. The corpus-bound suites (review, localize, agent) skip gracefully when the pinned corpus is absent, so a bare invocation stays offline; the semantics suite runs on the in-repo fixture with no corpus.

Run a suite by name:

```bash
fuse eval semantics
fuse eval review
fuse eval localize
fuse eval ranking
fuse eval checkgate
fuse eval loop
fuse eval agent
fuse eval reduce
```

There is one harness: the `Fuse.Benchmarks` C# library behind `fuse eval`. The legacy PowerShell layer scripts were retired (v4 N5); the one documented exception is the peer comparison (`harness/layer6-peers.ps1`), which orchestrates external MCP servers.

To index the corpus semantically, add `--restore`, which runs `dotnet restore` on each checkout (or, for review and agent, each PR worktree) before indexing so MSBuild and Roslyn can load it:

```bash
fuse eval review --restore
fuse eval localize --restore
```

A repository that cannot restore on the installed SDK stays in syntax mode and is recorded as such. Add `--require-semantic` to skip (rather than silently score at the syntax fallback) any checkout that does not reach semantic mode. Each suite writes its scorecard to `tests/benchmarks/results/<suite>.json` with the per-task hits, misses, and the index-mode distribution that produced the numbers. The agent suite additionally requires a model and the Claude Code CLI, and is not byte-reproducible.

## Suite A: semantic resolution (the deterministic moat)

Suite A checks the extracted semantic graph against a hand-built edge ground truth on the in-repo OrderingApp wiring fixture. The fixture covers the .NET wiring kinds that a regex or a tree-sitter scan cannot follow: DI registration and constructor injection, MediatR request-to-handler dispatch, ASP.NET route-to-action mapping, options binding and consumption, hosted-service workers, MediatR pipeline behaviors, EF Core DbContext-to-entity and entity-to-configuration wiring, Scrutor decoration, factory-lambda registration, minimal-API endpoints, gRPC services, and SignalR hubs. It also includes the precision edge cases: an explicit open generic registration, a `TryAdd` registration, and a multiple-implementation ambiguity where only the registered implementation resolves and a second, unregistered implementation of the same interface is correctly not resolved.

Result: 100 percent recall and 100 percent precision, 22 of 22 edges matched with zero false positives. This is the deterministic moat: when the workspace loads with semantics, Fuse resolves .NET wiring exactly, and nothing in the result depends on a model or a token budget.

A corpus sampled-adjudication mode (`fuse eval semantics --corpus-sample N`) indexes the corpus repositories, extracts the predicted graph edges, samples a fixed number per edge type with a seeded shuffle, and writes them to `semantics-corpus-sample.json` for adjudication. It produces a reproducible sample for a larger adjudicated run; corpus-wide per-resolver precision is bounded by how much of the corpus reaches semantic mode (a repository that loads syntax emits no graph edges to sample). One honest nuance: the constructor-injection edge captures any constructor-parameter dependency, so it includes value and enum parameters, not only DI-container services.

## Suite B: change impact and review

Suite B measures whether `fuse review` scopes a real merged PR to the files it touches plus the semantic blast radius around them. The sample is 53 real merged pull requests across the four repositories, at a 25,000-token budget with `--restore` on (each PR's worktree is restored before indexing). Changed files are seeded as must-keep, so the question this suite answers is precision: beyond the files the PR changed, what else does review return, and is it relevant.

| Metric | Value |
|--------|------:|
| Changed-file recall | 100% |
| Precision | 79.8% |
| F1 | 0.89 |
| Median returned tokens | 958 |
| Mean returned tokens | 1,095 |

Index modes over the 53 PRs: partial 27, semantic 1, syntax 25. Because review restores each PR's worktree (outside the repository tree, so no nested NuGet-config conflict), a majority of PRs load in partial or semantic mode, so the moat is exercised rather than hidden in syntax. Versus a grep baseline (rank the repository's C# files by title-token matches, admit to the budget), review wins decisively: grep reaches 53 percent changed-file recall at 14 percent precision.

How to read the precision:

- Changed-file recall is 100 percent by construction, since the changed files are seeded as must-keep. The signal in this suite is precision (79.8 percent) and F1 (0.89): review rarely returns a file the change did not touch or pull in.
- Review delivers a PR's context compactly. The median PR is scoped in 958 returned tokens, the mean in 1,095. This is where token efficiency lives: a property of the scoped review output, not a separate reduction claim.
- The semantic blast radius shows up on the application, not the libraries. On eShopOnWeb (partial mode) review adds files beyond the changed set through the semantic graph; where those extra files are the wiring and support a reviewer would want, precision on that PR reads lower while the wider set is the point. The library repositories that load syntax scope tightly to the changed files, which keeps aggregate precision high.

Against an adjudicated reading set (the support files a reviewer must read beyond the changed files, curated from the diff for a small pilot of PRs), review reaches a majority of the changed-plus-support files at high precision and beats the grep baseline on both axes. The remaining gap to full support recall is bounded by how much of the corpus loads semantically; scaling the adjudication and the semantic-mode coverage is the path to closing it.

## Suite C: open-ended localization (the weakest mode)

Suite C is the open-ended floor. It gives `fuse localize` only a PR title, with no Git base, and asks it to find the changed files from the title text alone. This is the hardest mode and the one where Fuse is weakest. Retrieval is the lexical channel (the dense embedding channel was retired in v4.1; see the note below). The retrieval mechanism (subword identifier indexing, stemming, a comment bridge, and the structural priors) is described in [Retrieval Internals](/docs/internals/scoping-internals).

Over the 53 PRs (title only, restore on):

| Metric | Value |
|--------|------:|
| Changed-file recall | 15.0% |
| Precision | 8.1% |
| Median returned tokens | 1,049 |
| Low-signal detection F1 | 1.00 |
| False-rejection on answerable | 0.0% (0 of 52) |
| Precision when confident | 5.6% (9 tasks) |

By bucket: identifier-rich 19 percent, natural-language domain 17 percent. The localize main-checkout index modes are partial 2, syntax 2 (NodaTime and Specification partial; Scrutor's NU1507 conflict and eShopOnWeb's main checkout load syntax), so the recall ceiling here is index mode, not the retrieval levers.

### The dense channel was retired (v4.1)

Earlier releases ran a dense embedding channel (a small local all-MiniLM-L6-v2 model) on by default. Its recorded lift did not pay for a 23 MB model download, an index-time embedding step, and a supply-chain surface: on this corpus dense reached 14.9 percent overall recall against the lexical fallback's 13.3 percent, and its only clear win was removing the lexical path's false rejections. In v4.1 the channel was deleted and the tool carries no model. The false rejections were recovered instead by lowering the signal-sufficiency floor (the lexical-only path now grades those three answerable queries as partial rather than refusing them), which held overall recall at 15.0 percent with false rejection back to 0.0 percent, on the lexical channel alone.

The signal-sufficiency contract is why the precision metrics hold despite the low recall. Because Fuse runs as an MCP server in a loop with a model, the answer to a low-signal request is not a low-precision list, it is to refuse and route: every request is graded **confident** (a candidate stands clear; return the tight set), **partial** (some signal, no clear winner; return a small flagged set plus a navigation map), or **insufficient** (no usable anchor; return only a navigation map and an explicit ask for a symbol, route, service, request, config section, or git base). The grade is computed from the candidate score distribution with no model. Correct-refusal on no-signal titles is 100 percent (low-signal detection F1 1.0), and the false-rejection rate on answerable queries stays at or under the 0.10 target. With a Git base available, the routed mode is change impact (Suite B), which is where the recall and precision are; Suite C is what Fuse can do from a sentence with no base, and it is reported as the weakest mode on purpose.

## Ranking regression (the ranking gate)

A field-weight inversion once shipped on the localize hot path (the FTS5 `path` column was weighted highest, so a folder-name match outranked a symbol-name match) because nothing measured ranking. The ranking suite closes that gap. Run with `fuse eval ranking`, it scores the retrieval ranking with MRR, recall@k, and nDCG@k against changed-file ground truth in three configurations over the same index: the base channels in isolation (no embedder, both structural priors off), the shipping default (embedder if present, both priors on), and the shipping default with the git co-change prior off (isolating that prior's ranking effect, the A6 re-adjudication). It writes `results/ranking.json` and is the required gate on any change to field weights, tokenization, query expansion, or priors. The corrected weight ordering (name and signature and symbols above path, path above comments and body, subtokens below the exact name but above the body, stems lowest) is documented in [Retrieval Internals](/docs/internals/scoping-internals). Like the other corpus suites, the ranking numbers are bounded by index mode; the deliverable is the guarded metric, so a future reweighting cannot regress ranking unmeasured.

## Suite R4: the loop metric (model-dependent, harness-first)

Suite D showed that per-payload token reduction does not change the cumulative token cost of a session, so the metric that matters is not the payload but the loop: how many build-gated turns an agent takes to reach a green build. Suite R4 measures that. One Claude Code CLI driver resolves each task (edit, verify, repeat) in two arms, `native` (filesystem plus `dotnet build`/`dotnet test`) and `fuse` (the fuse MCP tools, so a verify can be a speculative `fuse_check` instead of a `dotnet build` round-trip), and the run records, per arm, whether it reached green, the number of build-gated turns to green, and the total build invocations. The claim under test is that the fuse arm reaches green in fewer build-gated turns.

Following the plan, R4 is harness-first: the harness is the deliverable, and the numbers are recorded from an explicit, provisioned run. The deterministic core, turn classification (`LoopTranscriptClassifier`, which maps a stream-json transcript to the ordered build, test, edit, and read turns) and the metric computation (`LoopMetrics`), is unit-tested and runs offline. The model-driven arms are opt-in: a bare `fuse eval loop` records the harness state and the task set it would sample without spending model time, and only `FUSE_LOOP_RUN=1` (with the `claude` CLI present) launches the rollouts. This is deliberate, because a loop rollout drives a real model per task per arm and must never fire silently, especially inside a host where the CLI is already on PATH. Run with `fuse eval loop`; it writes `results/loop.json`.

### Recorded run (`loop.json`)

The first provisioned run used `claude-sonnet-4-6` over 4 PRs (one per corpus repository, `--limit 1 --restore`), producing 7 scored rollouts; one fuse rollout (eShopOnWeb#949) wedged and was omitted rather than stubbed.

| Arm | Tasks (n) | Reached green | Median build-gated iterations to green | Mean build invocations |
|-----|----------:|--------------:|---------------------------------------:|-----------------------:|
| native | 4 | 25% (1/4) | 1.0 | 0.8 |
| fuse | 3 | 0% (0/3) | 0.0 | 0.3 |

Read this honestly: on this run the fuse arm does **not** collapse the loop. It reached green on none of its tasks, while native reached green on one (eShopOnWeb#949, in a single iteration and two builds). The dominant cause is that the corpus does not build in this environment: two of the four repositories restored zero packages, so most tasks cannot reach a green build in either arm regardless of tooling, which is the same build and index-mode ceiling that bounds every other corpus suite here. The fuse arm's lower mean build count (0.3 versus 0.8) is not a win; it ran fewer builds and still never reached green.

The sample is small (4 PRs, one rollout per arm, one model) with a wide confidence interval and one wedged rollout, so this is a directional null, not a settled measurement: the oracle-collapse thesis is neither confirmed nor refuted by it. Confirming or refuting it needs a larger provisioned run over a task set that actually builds in the eval environment. This result is reported as it landed and the launch messaging is positioned against it, not around it.

## Suite D: agent context sufficiency (model-dependent)

Suite D replaces a structural bound with a real agent. It runs one Claude Code CLI driver, model `claude-sonnet-4-6`, over 12 PRs (two per repository) in two arms (native filesystem Read/Grep/Glob, and the Fuse MCP tools), one rollout each, for 24 rollouts. The run goes through the real `fuse.exe` so the fuse arm's MCP server is correctly wired, with `--restore` so the fuse arm indexes semantically where a repository restores. It measures the file recall of the set the agent gathered and the cumulative tokens it spent.

This suite is model-dependent and not byte-reproducible. The numbers depend on the model, its sampling, and the day; read them as a small, wide-CI sample, not a fixed measurement, and always read recall together with tokens.

| Arm | Mean file recall | Mean precision | Median cumulative tokens |
|-----|-----------------:|---------------:|-------------------------:|
| fuse | 30% | 44% | 211,502 |
| native | 26% | 43% | 209,182 |

On this 12-PR sample the fuse arm reached 30 percent mean recall to native's 26 percent, at comparable precision (44 versus 43 percent) and comparable token cost. With the MCP server correctly wired and semantic indexing where the repository restores, the Fuse tools gather a slightly more complete set without spending more. The sample is small (12 PRs, one rollout per arm, one model), so the confidence interval is wide and neither mean is settled; this is a directional signal, not a fixed benchmark. A full run of 50 to 100 tasks at three rollouts needs a larger agent-compute budget.

### Peer comparison (Fuse versus CodeGraph, coa-codesearch, and Serena)

The peer layer gives each tool the PR title and scores the returned file set against the changed files. Fuse runs through its index-then-localize path; CodeGraph (the offline tree-sitter graph peer) runs `codegraph init` then `codegraph explore`; coa-codesearch (anortham/coa-codesearch-mcp, a .NET Lucene MCP server, built from source) and Serena (oraios/serena, an LSP-backed symbol-search MCP server, via uvx) are each driven by one headless `claude` rollout restricted to that server's search tools. All four arms are present, none stubbed. At a 50,000-token budget (`layer6-peers.json`):

| Arm | Sample (PRs) | Mean recall | Mean precision | Mean tokens |
|-----|------------:|------------:|---------------:|------------:|
| fuse | 12 | 19% | 19% | 10,717 |
| codegraph | 12 | 9% | 11% | 3,582 |
| coa | 4 | 9% | 1% | 3,382 |
| serena | 4 | 34% | 27% | 1,538 |

Read this carefully, because the aggregate hides a sampling difference. The deterministic arms (fuse, codegraph) ran all 12 sampled PRs; the model-driven arms (coa, serena) ran one `claude` rollout per PR and were bounded to one PR per repository (4 PRs total) to keep the rollout compute affordable, so their column is a smaller, different sample. On the deterministic 12-PR comparison, Fuse leads CodeGraph decisively on recall (19 versus 9 percent) at higher precision. On the four shared model-driven PRs, Serena's higher aggregate is dominated by a single tiny-repo outlier (on a 2-file change in a 45-file repository its symbol search returns the whole repository and scores 100 percent recall), while on the two substantive PRs Fuse leads or ties it. coa returns a long ranked path list, so its recall is moderate but its precision is near zero. The token columns are not directly comparable: Fuse and CodeGraph return source the agent reads directly, while coa and Serena return path or snippet lists the agent would still open. This is a small sample with Fuse on its C# home field, reported straight, and no single arm-versus-arm point should be over-read.

Reproduce the head-to-head (the deterministic arms run with no extra setup; the model-driven arms are env-gated and run only when their binary is present, otherwise they are recorded as omitted, never stubbed):

```bash
# Fuse and CodeGraph only (deterministic, all sampled PRs)
pwsh -File tests/benchmarks/harness/layer6-peers.ps1

# Add coa-codesearch and Serena (model-driven, one claude rollout per PR, bounded per repo)
COA_CODESEARCH_EXE=/path/to/COA.CodeSearch.McpServer.exe \
SERENA_CMD=uvx \
pwsh -File tests/benchmarks/harness/layer6-peers.ps1 -PerRepo 4 -ModelPerRepo 1
```

The run writes `tests/benchmarks/results/layer6-peers.json` with every arm's per-PR rows, sample size, and setup time. Where Fuse does not lead is stated above: on a tiny repository a whole-repo symbol dump can out-recall a scoped result, and a pure path-list search returns fewer tokens because it returns paths, not source. A full 50-to-100-PR model-driven run is the remaining work; until it exists, no broader ranking is claimed.

## Suite E: token reduction and fidelity

Suite E counts, for each repository, the raw `o200k_base` tokens of the C# files, reduces them at each level through the shipped reduction path, counts the reduced tokens, and at skeleton level checks how many public and protected types and methods survive. Fidelity is counted by parsing the raw source with Roslyn as an independent ground truth, so the measure is not circular. Run with `fuse eval reduce`.

## Suite F: the fuse_check honesty gate

`fuse_check` is only useful if it does not lie. There are two ways it can. A false green reports a broken edit clean, and that is the dangerous one: it would let an agent commit a change that does not build. A false red reports a valid edit broken, which is cheaper but wastes an agent's turn chasing a phantom error. Suite F measures both. It builds a small self-contained compilation with raw Roslyn (no MSBuild, so it runs everywhere), then applies a battery of single-file edits each with a known-correct verdict: equivalent rewrites, an added valid overload, and a comment-only change that must stay clean; a call to a missing member, a wrong return type, an undefined type, a syntax error, and an assignment to an init-only property that must each be flagged. Each edit replaces one document's syntax tree and is classified with the exact `CheckResult.IsClean` rule the shipped worker applies, so the gate measures the classification contract the tool ships. An abstention (the compiler could not run) counts as neither a false green nor a false red, because abstaining is the honest answer when the oracle cannot know. Run with `fuse eval checkgate`; it writes `results/checkgate.json`. The eight curated edits are all classified correctly, zero false green and zero false red. The tier-1 end-to-end path (the out-of-process build-capture worker) is exercised by `BuildCaptureCheckTests` when a worker is provisioned; the offline gate is the in-process classification measure and records the tier-1 arm as skipped when no worker is configured, rather than fabricating a number for it.

### Mutation-derived honesty at scale

Eight hand-built cases are too few to carry the "never lies" claim, so `fuse eval checkgate --mutations N` generates the ground truth mechanically. Roslyn syntax rewriters produce two classes of single-file edit over an in-repo fixture: breaking (rename a bound member access, replace a used type with an undefined one, change an expression to an incompatible literal, delete a referenced private member) and neutral (a consistent reorder, redundant parentheses, comment and whitespace insertions). The verdict is not asserted by the operator, it is confirmed by the compiler: a breaking mutant is kept only when the compilation actually reports an error located in the edited file, and a neutral mutant only when the compilation stays clean, so the labels are mechanical rather than hand-assigned. Generation is deterministic from a recorded seed. Operators are single-file because the shipped `fuse_check` contract is a single-file edit; a breaking mutant is credited only when the error lands in the edited document, so the gate fairly tests the single-file classifier.

The recorded run (`--mutations 500`) produced 1,000 compiler-verified cases (500 breaking, 500 neutral) over the OrderingApp fixture: zero false green and zero false red (a false-green rate of 0.00 percent and a false-red rate of 0.00 percent over 1,000 verified cases), with the eight curated cases retained as a named subset. The SampleShop fixture is a two-project web solution (a core library plus an MVC web app) that does not bind in the suite's flat in-process compilation (it reports CS0234/CS0246/CS0103 without its full per-project references), so it is recorded skipped rather than scored; compiling it with per-project references is a follow-up that would add a second fixture to the run. The gate is false green 0 and false red under 1 percent, and the recorded run meets it.

### Oracle-versus-build agreement

`fuse_check` answers at the best available [verification grade](/docs/concepts/verification-grades): a sub-second speculative typecheck against the tier-1 build-captured compilation (`oracle`), or, when no tier-1 substrate is available, a real `dotnet build` scoped to the owning project (`build`). Build-grade is ground truth by construction (the compiler itself answered), so the honesty question is whether the fast oracle path agrees with it. `fuse eval checkgate --verify-agreement N` (with a build-capture worker provisioned) runs the same proposed content through both paths for a sample of mutants and compares the set of error diagnostics attributed to the changed file. The recorded run (`--verify-agreement 24`) sampled 24 OrderingApp mutants, all 24 comparable with no abstention on either side, and recorded diagnostic-id agreement of 24 of 24 (100.0 percent) and clean/red verdict agreement of 24 of 24. The gate is at least 99 percent agreement, met at 100 percent on this sample. The sample is bounded because each mutant runs two real builds; a larger provisioned sweep is the natural follow-up. When no worker is configured the arm is recorded skipped, never fabricated.

| Repository | Raw tokens | Standard | Aggressive | Skeleton | Public API | Skeleton fidelity (types / methods) |
|------------|-----------:|---------:|-----------:|---------:|-----------:|-------------------------------------|
| NodaTime | 823,655 | 41% | 46% | 38% | 60% | 336/336, 1696/1708 |
| Scrutor | 40,868 | 33% | 37% | 42% | 53% | 107/107, 101/105 |
| Specification | 133,415 | 25% | 30% | 44% | 49% | 188/188, 336/349 |
| eShopOnWeb | 65,569 | 11% | 16% | 42% | 47% | 226/226, 186/189 |

The Roslyn skeleton keeps every public and protected type and 99 to 100 percent of public methods while removing 38 to 44 percent of tokens; public-API level removes 47 to 60 percent. The percentages are reduction against the raw C# token count.

## What is not measured at scale

These are honest gaps, not claims; do not read an absence here as a result:

- The model-driven peer comparison at full scale. All four arms run head-to-head (see Peer comparison above), but the model-driven arms run a bounded 4-PR sample; a full 50-to-100-PR model-driven run needs a larger compute budget.
- Full task resolution at scale: driving an agent across arms and scoring its patch against a test oracle over many tasks. The deterministic core of that harness (materialize an isolated worktree at the base commit, apply a candidate patch, run a test oracle, score pass@1) is built and tested; the agent-driven, at-scale run needs provisioned model credentials and a large compute budget.
- Cross-machine timing. The warm-index latency model is on [Performance and Benchmarking](/docs/project/performance); no fixed cross-machine number is published.

## How to read these results together

Suite A is the clean, deterministic claim: with a semantic workspace, Fuse resolves .NET wiring exactly (22 of 22 edges across the wiring catalog). Suite B is the practical strength: review scopes a real PR at high precision and 100 percent changed-file recall in around a thousand tokens, beating a grep baseline, with the semantic blast radius exercised on the application. Suite C is the honest floor: from a title alone with no base, recall is low and the engine abstains on a no-signal title rather than returning junk. Suite D is a small, model-dependent signal that the Fuse MCP tools can gather the right files cheaply when the task names the wiring. The recurring caveat across the corpus suites is index mode: most repositories load in syntax or partial mode in this environment, so the corpus numbers sit below the semantic ceiling that Suite A and the application demonstrate.
