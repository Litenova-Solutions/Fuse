---
title: Benchmarks
description: Reproducible measurements of Fuse V3 as a Roslyn-backed .NET semantic context engine, scored on four suites over a pinned corpus of real .NET repositories, with honest reporting of where Fuse loses.
---

Fuse V3 is a Roslyn-backed .NET semantic context engine with a warm persistent index. This page reports what it does on real .NET codebases, judged by one north star: can it resolve .NET wiring, scope a change with precision, help an agent gather the right files, and avoid returning junk on a low-signal query. The yardstick is the semantic engine, not tokens saved. Token efficiency is now a rendering and transport feature (a PR's context arrives in a median 874 tokens), not the headline claim.

Three readers are served below. A stakeholder can read the four suite headlines and the honest-reporting section. A new engineer can follow the reproduction steps and run a suite. A maintainer can read each suite's method, sample, index-mode distribution, and the cases where Fuse is weak.

The benchmark is four suites run by a C# eval driver in `tests/benchmarks/Fuse.Benchmarks`, each reproduced with `fuse eval <suite>` and each writing its scorecard to `tests/benchmarks/results/<suite>.json`:

- Suite A, semantic resolution: does the extracted graph match a hand-built edge ground truth. Deterministic.
- Suite B, change impact and review: does `fuse review` scope a real merged PR to the files it touches plus their semantic blast radius. Deterministic.
- Suite C, open-ended localization: given only a PR title and no Git base, does `fuse localize` find the changed files. Deterministic, and the weakest mode.
- Suite D, agent context sufficiency: under a real model, does an agent gather the task's files with the Fuse MCP tools. Model-dependent and not byte-reproducible.
- Suite E, token reduction and fidelity: how far each reduction level cuts tokens, and how much of the public API the skeleton keeps. Deterministic and offline.

## An Honest Caveat on Index Mode

Fuse's semantic graph is strongest when the corpus loads with a full MSBuild semantic workspace. A full load needs a restored workspace (a `project.assets.json` from `dotnet restore`); without one, indexing falls back to syntax mode and every semantic suite reads a thin graph. The harness can now restore each checkout before indexing with the `--restore` flag (see Reproduce), and records the achieved index mode per repository so the reader can see which mode produced which number.

In the current benchmark environment, restore lifts only the two repositories whose pinned commit still restores on the installed SDK: NewtonsoftJson and the eShopOnWeb application reach partial mode (the semantic graph), while AutoMapper, FluentValidation, MediatR, and Serilog fail to restore on this SDK because their pinned-era build (central package management with an inline-versioned SourceLink, or an older target framework) does not resolve against the newer SDK. Those four stay in syntax mode. This is a corpus-pinning limit, not an engine limit: where a repository restores, it loads semantically. The full semantic graph is exercised on eShopOnWeb (partial mode) and on the in-repo wiring fixture (Suite A). Read the deterministic moat (Suite A) as the semantic ceiling and the corpus suites as what the achieved modes deliver today.

## The Corpus

The corpus is five open-source .NET libraries and one open-source ASP.NET Core application, all pinned by commit, counted with the `o200k_base` encoding. The application (eShopOnWeb) is the only non-library: it carries DI wiring, MVC and Razor Pages, EF Core, and config sprawl that the clean library APIs do not exercise, so it tests external validity beyond tidy public surfaces. PR ground truth (`prs.json`) is reconstructed from merge history, dropping merged PRs whose titles are misleading maintenance (a CI tweak or a dependency or version bump over an unrelated C# diff), since a title-only scope cannot find their change sets.

| Repository | Size | C# files |
|------------|------|---------:|
| MediatR | small | 152 |
| FluentValidation | small to mid | 218 |
| AutoMapper | mid | 511 |
| Newtonsoft.Json | large | 945 |
| Serilog | small to mid | 216 |
| eShopOnWeb (application) | mid | 254 |

## Reproduce

Prerequisites: the .NET SDK 10.0 or later and Git. The corpus-bound suites (review, localize, agent) skip gracefully when the pinned corpus is absent, so a bare invocation stays offline; the semantics suite runs on the in-repo fixture with no corpus.

Run a suite by name:

```bash
fuse eval semantics
fuse eval review
fuse eval localize
fuse eval agent
```

To index the corpus semantically, add `--restore`, which runs `dotnet restore` on each checkout (or, for review and agent, each PR worktree) before indexing so MSBuild and Roslyn can load it:

```bash
fuse eval review --restore
fuse eval localize --restore
```

A repository that cannot restore on the installed SDK stays in syntax mode and is recorded as such. Add `--require-semantic` to skip (rather than silently score at the syntax fallback) any checkout that does not reach semantic mode, so a restored-workspace run reports only semantic numbers.

Each suite writes its scorecard to `tests/benchmarks/results/<suite>.json` (one file per suite: `semantics.json`, `review.json`, `localize.json`, `agent.json`), with the per-task hits, misses, and the index-mode distribution that produced the numbers. A `--restore` run is recorded alongside as `<suite>.restore.json`. The agent suite additionally requires a model and the Claude Code CLI, and is not byte-reproducible.

## Suite A: Semantic Resolution (the deterministic moat)

Suite A checks the extracted semantic graph against a hand-built edge ground truth on the in-repo OrderingApp wiring fixture. The fixture covers the four .NET wiring kinds that a regex or a tree-sitter scan cannot follow: DI registration and constructor injection, MediatR request-to-handler dispatch, ASP.NET route-to-action mapping, and options binding and consumption.

Result: 100 percent recall and 100 percent precision, 10 of 10 edges matched with zero false positives. This is the deterministic moat: when the workspace loads with semantics, Fuse resolves .NET wiring exactly, and nothing in the result depends on a model or a token budget.

This is the suite that runs on the full semantic graph (the fixture loads with a complete workspace). The corpus suites below run mostly in partial or syntax mode, so read them against this ceiling.

## Suite B: Change Impact and Review

Suite B measures whether `fuse review` scopes a real merged PR to the files it touches plus the semantic blast radius around them. The sample is 48 real merged pull requests, 8 per repository across the six repositories, at a 25,000-token budget. Changed files are seeded as must-keep, so the question this suite answers is precision: beyond the files the PR changed, what else does review return, and is it relevant.

| Metric | Value |
|--------|------:|
| Changed-file recall | 100% |
| Precision | 89.6% |
| F1 | 0.95 |
| Median returned tokens | 874 |
| Mean returned tokens | 3,196 |

Index modes over the 48 PRs: partial 12, syntax 36.

With `--restore` on the application (eShopOnWeb), the per-PR worktree restores and review runs in partial mode on 15 of 18 PRs (`review.restore.json`): changed-file recall 1.0 at 64 percent precision in a median 587 returned tokens. The lower precision here is the semantic blast radius working as intended on an application: review pulls in the wiring and support files a reviewer wants, which the changed-files-only truth counts against it. This is the same effect the per-PR examples below show, now across the application's PR set in restored mode.

Findings, including the boundary:

- Changed-file recall is 100 percent by construction, since the changed files are seeded as must-keep. The signal in this suite is precision (89.6 percent) and F1 (0.95): review rarely returns a file the change did not touch or pull in.
- Review delivers a PR's context compactly. The median PR is scoped in 874 returned tokens, the mean in 3,196. This is where token efficiency lives in V3: a rendering and transport property of the scoped review output, not a separate reduction claim.
- The semantic blast radius shows up on the application, not the libraries. On eShopOnWeb (partial mode) review adds files beyond the changed set through the semantic graph, which lowers precision where the extra files are the wiring a reviewer would want: eShopOnWeb#322 scopes at 21 percent precision, eShopOnWeb#300 at 35 percent. The library repos load in syntax mode and scope tightly to the changed files, which is why aggregate precision stays high. The honest reading: the partial-mode application is where the blast radius is exercised, and there the wider set is the point, not noise.

## Suite C: Open-Ended Localization (the weakest mode)

Suite C is the open-ended floor. It gives `fuse localize` only a PR title, with no Git base, and asks it to find the changed files from the title text alone over 108 PRs. This is the hardest mode and the one where Fuse is weakest.

| Metric | Value |
|--------|------:|
| Changed-file recall | 27.3% (95% CI 21.5 to 33.4) |
| Precision | 8.4% |
| Median returned tokens | 2,658 |

By signal bucket (how much the title tells the engine where to look):

| Bucket | Mean recall |
|--------|------------:|
| identifier-rich | 40% |
| test-only | 33% |
| no-signal | 33% |
| natural-language domain | 23% |
| route/API | 12% |
| formatting | 8% |

The table above is the recorded baseline (`localize.json`). R1 reconnects the lexical ranker (`localize.r1.json`): the BM25F rank is now carried through to the candidate score (a stronger lexical match ranks higher, not just present), with one capped round of pseudo-relevance feedback that pulls in files sharing the top hits' vocabulary. Overall recall rises to 30.3 percent and the natural-language-domain bucket to 27 percent, at a precision cost (6.6 percent) and a higher median return (about 4,000 tokens) from the broader pool. Identifier-rich holds at 41 percent, short of the retired 49 percent lexical floor: recall here is averaged over multi-file PRs at a 20-candidate cap in a mostly-syntax corpus, so naming one file does not by itself lift the bucket past the floor. The dense reranker for prose titles (Suite C's largest bucket) is R2; low-signal abstention is R3.

Findings, including the known gap:

- Recall tracks the signal in the title. When the title names identifiers that appear in the code (identifier-rich, 41 percent after R1), localization does best; when the title is a domain phrase (27 percent), a route or API description, or a formatting note, it does worse. This is the expected shape of a title-only, no-base task.
- Low-signal detection is a known gap. The suite scores whether Fuse flags a title that carries no usable signal (so it can decline rather than guess). Low-signal detection F1 is 0.11 (1 true positive, 8 false positives, 8 false negatives). State this plainly: V3 does not yet reliably flag a no-signal title and instead returns full-text candidates. Closing this gap (declining or warning on a no-signal query rather than returning junk) is open work.
- This is the floor, not the routed path. With a Git base available, the routed mode is change impact (Suite B), which is where the recall and precision are. Suite C is what Fuse can do from a sentence with no base, and it is reported as the weakest mode on purpose.

## Suite D: Agent Context Sufficiency (model-dependent)

Suite D replaces a structural bound with a real agent. It runs one Claude Code CLI driver, model `claude-sonnet-4-6`, over 6 PRs in two arms (native filesystem tools, and the Fuse MCP tools), one rollout each, for 12 rollouts. It measures the file recall of the set the agent gathered and the cumulative tokens it spent.

This suite is model-dependent and not byte-reproducible. The numbers depend on the model, its sampling, and the day; read them as a small, wide-CI sample, not a fixed measurement, and always read recall together with tokens.

| Arm | Mean file recall | Median cumulative tokens |
|-----|-----------------:|-------------------------:|
| fuse | 21% | 135,012 |
| native | 27% | 211,850 |

The sample is small and the confidence interval is wide (0 to 54 percent), so neither arm's mean is a settled result.

Findings, including the shape:

- Read recall with tokens. The fuse arm reached 21 percent mean recall at a median 135,012 cumulative tokens; native reached 27 percent at a median 211,850. The fuse arm gathered a somewhat less complete set at a markedly lower token cost on this sample.
- The win is concentrated where the title names the wiring. On MediatR#1171, whose title names the licensing wiring, the fuse arm reached 100 percent recall in 4 tool calls at 121k tokens, while the native arm read 328k tokens to reach the same recall. When the semantic signal is present, the Fuse tools find the files fast and cheap.
- The sample is too small to generalize. Six PRs, one rollout per arm, one model, one day. This suite is a directional signal about whether the Fuse MCP tools help an agent, not a benchmark of agent task success.

## Suite E: Token Reduction and Fidelity (offline continuity)

Suite E is the offline continuity layer. For each repository it counts the raw o200k_base tokens of the C# files, reduces them at each level through the shipped reduction path, counts the reduced tokens, and at skeleton level checks how many public and protected types and methods survive. Fidelity is counted by parsing the raw source with Roslyn as an independent ground truth, so the measure is not circular. Run with `fuse eval reduce`.

| Repository | Raw tokens | Standard | Aggressive | Skeleton | Public API | Skeleton fidelity (types / methods) |
|------------|-----------:|---------:|-----------:|---------:|-----------:|-------------------------------------|
| AutoMapper | 458,486 | 13% | 20% | 44% | 51% | 2114/2114, 1051/1056 |
| FluentValidation | 257,354 | 30% | 35% | 55% | 60% | 249/249, 847/889 |
| MediatR | 80,635 | 20% | 24% | 51% | 57% | 256/256, 207/207 |
| Newtonsoft.Json | 1,452,192 | 34% | 40% | 52% | 57% | 1212/1212, 3050/3053 |
| Serilog | 199,026 | 39% | 45% | 37% | 47% | 140/140, 524/527 |
| eShopOnWeb | 65,569 | 11% | 16% | 42% | 47% | 226/226, 186/189 |

The Roslyn skeleton keeps every public and protected type and 99 to 100 percent of public methods while removing 37 to 55 percent of tokens; public-API level removes 47 to 60 percent. The percentages are reduction against the raw C# token count.

## What Is Not Measured on V3

These were measured on the retired V2 surface and have not been re-run on the V3 semantic engine. Do not quote the old percentages as V3 results:

- Warm-call latency. The warm persistent index model is described qualitatively in [Performance and Benchmarking](/docs/project/performance); no V3 latency number is published.
- The peer-scoper comparison against CodeGraph and coa-codesearch.
- Full task resolution: driving an agent across arms and scoring its patch against a test oracle.

Each of these is not-yet-measured on V3, not a claim. Treat any such figure as illustrative until a V3 suite produces it.

## How to Read These Results Together

Suite A is the clean, deterministic claim: with a semantic workspace, Fuse resolves .NET wiring exactly (10 of 10 edges). Suite B is the practical strength: review scopes a real PR at 89.6 percent precision and 100 percent changed-file recall in a median 874 tokens, with the semantic blast radius exercised on the application in partial mode. Suite C is the honest floor: from a title alone with no base, recall is 27.3 percent and the engine does not yet flag a no-signal title. Suite D is a small, model-dependent signal that the Fuse MCP tools can gather the right files cheaply when the task names the wiring. The recurring caveat across the corpus suites is index mode: most repositories load in syntax or partial mode in this environment, so the corpus numbers sit below the semantic ceiling that Suite A and the application demonstrate.

## Blocked and Out of Scope

- Full semantic load across the whole corpus. Most corpus repositories load in syntax or partial mode here because the offline harness does not provision a restored MSBuild workspace per repository. The full semantic graph is exercised on eShopOnWeb (partial) and the in-repo fixture (full). A restored-workspace corpus run is the way to lift the corpus suites toward the Suite A ceiling.
- Task resolution with a test oracle. Suite D measures context sufficiency (did a real agent gather the files the task needs, and at what cost). Full task resolution, a SWE-bench-style harness that scores an agent's patch against a test oracle, is not built here; it needs provisioned model credentials and compute and is scaffold-specific.
- Cross-machine timing. No V3 latency is published; see Performance for the qualitative warm-index model.
- Tokenizer scope. All counts use `o200k_base`. Absolute counts differ under other encodings.
