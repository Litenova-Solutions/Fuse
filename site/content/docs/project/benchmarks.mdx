title: Benchmarks
description: Reproducible measurements of Fuse V3 as a Roslyn-backed .NET semantic context engine, scored on four suites over a pinned corpus of real .NET repositories, with honest reporting of where Fuse loses.
---

Fuse V3 is a Roslyn-backed .NET semantic context engine with a warm persistent index. This page reports what it does on real .NET codebases, judged by one north star: can it resolve .NET wiring, scope a change with precision, help an agent gather the right files, and avoid returning junk on a low-signal query. The yardstick is the semantic engine, not tokens saved. Token efficiency is now a rendering and transport feature (a PR's context arrives in a median 874 tokens), not the headline claim.

Three readers are served below. A stakeholder can read the four suite headlines and the honest-reporting section. A new engineer can follow the reproduction steps and run a suite. A maintainer can read each suite's method, sample, index-mode distribution, and the cases where Fuse is weak.

The benchmark is four suites run by a C# eval driver in `tests/benchmarks/Fuse.Benchmarks`, each reproduced with `fuse eval <suite>` and each writing its scorecard to `tests/benchmarks/results/<suite>.json`:

- Suite A, semantic resolution: does the extracted graph match a hand-built edge ground truth. Deterministic.
- Suite B, change impact and review: does `fuse review` scope a real merged PR to the files it touches plus their semantic blast radius. Deterministic.
- Suite C, open-ended localization: given only a PR title and no Git base, does `fuse localize` find the changed files. Deterministic, and the weakest mode.
- Suite D, agent context sufficiency: under a real model, does an agent gather the task's files with the Fuse MCP tools. Model-dependent and not byte-reproducible.
- Suite E, token reduction and fidelity: how far each reduction level cuts tokens, and how much of the public API the skeleton keeps. Deterministic and offline.

## An Honest Caveat on Index Mode

Fuse's semantic graph is strongest when the corpus loads with a full MSBuild semantic workspace. A full load needs a restored workspace (a `project.assets.json` from `dotnet restore`); without one, indexing falls back to syntax mode and every semantic suite reads a thin graph. The harness can now restore each checkout before indexing with the `--restore` flag (see Reproduce), and records the achieved index mode per repository so the reader can see which mode produced which number.

V3.1 (item A3) rebuilt the corpus to repositories that restore and index in semantic or partial mode on the .NET 10 SDK, replacing the prior set whose pinned commits fell back to syntax. The result is that the semantic moat is now visible in our own numbers: review (Suite B) reaches partial or semantic mode on 28 of 53 PRs, where the prior corpus reached partial on only 12 of 48. Where a repository restores, it loads semantically; the deterministic moat (Suite A) remains the semantic ceiling.

**What A3 and A2 re-ran.** The corpus was rebuilt and Suites A, B, and C were re-run on it (A3, 2026-06-27), and the peer comparison was re-run on it with all four arms present (A2). The agent (Suite D), token-reduction (Suite E), and R4 adjudicated review tables further down this page were recorded on the prior corpus and are labelled as such; they are re-run on the rebuilt corpus in later V3.1 items (A1 dense, A4 cold-start). No prior-corpus number is presented as a rebuilt-corpus number.

## The Corpus

The corpus is three open-source .NET libraries and one open-source ASP.NET Core application, all pinned by commit and chosen to restore on the .NET 10 SDK, plus the in-repo SampleShop fixture, counted with the `o200k_base` encoding. The application (eShopOnWeb) is the only non-library: it carries DI wiring, MVC and Razor Pages, EF Core, and config sprawl that the clean library APIs do not exercise, so it tests external validity beyond tidy public surfaces. PR ground truth (`prs.json`, 53 PRs reconstructed from merge history) drops merged PRs whose titles are misleading maintenance (a CI tweak or a dependency or version bump over an unrelated C# diff), since a title-only scope cannot find their change sets.

| Repository | Size | C# files | PRs |
|------------|------|---------:|----:|
| Scrutor | micro | 45 | 3 |
| Ardalis.Specification | small to mid | 233 | 14 |
| NodaTime | mid | 488 | 18 |
| eShopOnWeb (application) | mid | 254 | 18 |

One corpus-pinning caveat is recorded honestly: Scrutor uses central package management with two NuGet sources, and nested under Fuse's repository NuGet config it triggers a deterministic NU1507 source-mapping conflict and restores to syntax. NodaTime and Ardalis.Specification restore clean; eShopOnWeb restores per-PR-worktree (Suite B) but its main checkout (Suite C) loads syntax, so the localize index-mode distribution is lower than review's.

## Reproduce

Prerequisites: the .NET SDK 10.0 or later and Git. The corpus-bound suites (review, localize, agent) skip gracefully when the pinned corpus is absent, so a bare invocation stays offline; the semantics suite runs on the in-repo fixture with no corpus.

Run a suite by name:

```bash
fuse eval semantics
fuse eval review
fuse eval localize
fuse eval agent
```

To index the corpus semantically, add `--restore`, which runs `dotnet restore` on each checkout (or, for review and agent, each PR worktree) before indexing so MSBuild and Roslyn can load it:

```bash
fuse eval review --restore
fuse eval localize --restore
```

A repository that cannot restore on the installed SDK stays in syntax mode and is recorded as such. Add `--require-semantic` to skip (rather than silently score at the syntax fallback) any checkout that does not reach semantic mode, so a restored-workspace run reports only semantic numbers.

Each suite writes its scorecard to `tests/benchmarks/results/<suite>.json` (one file per suite: `semantics.json`, `review.json`, `localize.json`, `agent.json`), with the per-task hits, misses, and the index-mode distribution that produced the numbers. A `--restore` run is recorded alongside as `<suite>.restore.json`. The agent suite additionally requires a model and the Claude Code CLI, and is not byte-reproducible.

## Suite A: Semantic Resolution (the deterministic moat)

Suite A checks the extracted semantic graph against a hand-built edge ground truth on the in-repo OrderingApp wiring fixture. The fixture covers the .NET wiring kinds that a regex or a tree-sitter scan cannot follow: DI registration and constructor injection, MediatR request-to-handler dispatch, ASP.NET route-to-action mapping, options binding and consumption, and (R5) hosted-service workers, MediatR pipeline behaviors, EF Core DbContext-to-entity and entity-to-configuration wiring, Scrutor decoration, factory-lambda registration, minimal-API endpoints, gRPC services, and SignalR hubs. Open and closed generic DI resolves through the existing typeof-pair registration path.

Result: 100 percent recall and 100 percent precision, 22 of 22 edges matched with zero false positives. This is the deterministic moat: when the workspace loads with semantics, Fuse resolves .NET wiring exactly, and nothing in the result depends on a model or a token budget. R5 widened it from 10 to 19 edges across nine new wiring kinds; R6 added the edge-case catalog (an explicit open generic, a TryAdd registration, and a multiple-implementation ambiguity where only the registered implementation resolves), taking it to 22 edges. The ambiguity case is the precision side of the moat: a second, unregistered implementation of the same interface is correctly not resolved.

R6 also adds a corpus sampled-adjudication mode (`fuse eval semantics --corpus-sample N`), which indexes the corpus repositories, extracts the predicted graph edges, samples a fixed number per edge type with a seeded shuffle, and writes them to `semantics-corpus-sample.json` for adjudication. Over the repositories that reached partial mode (FluentValidation and Serilog), it extracted 261 edges and sampled 24 (8 each of di_injects, implements, inherits). A single-adjudicator pass against the corpus source found all 24 structurally correct (Wilson 95 percent lower bound about 86 percent), with one honest nuance worth recording: the di_injects edge captures any constructor-parameter dependency, so it includes value and enum parameters (for example a LogEventLevel), not only DI-container services. Tighter, corpus-wide per-resolver precision is bounded here by the sparse semantic-mode corpus (most repositories load syntax, which emits no graph edges to sample); the mechanism produces a reproducible sample for a larger adjudicated run once more of the corpus restores.

This is the suite that runs on the full semantic graph (the fixture loads with a complete workspace). The corpus suites below run mostly in partial or syntax mode, so read them against this ceiling.

## Suite B: Change Impact and Review

Suite B measures whether `fuse review` scopes a real merged PR to the files it touches plus the semantic blast radius around them. On the V3.1 A3 rebuilt corpus the sample is 53 real merged pull requests across the four repositories, at a 25,000-token budget with `--restore` on (each PR's worktree is restored before indexing). Changed files are seeded as must-keep, so the question this suite answers is precision: beyond the files the PR changed, what else does review return, and is it relevant.

| Metric | Value |
|--------|------:|
| Changed-file recall | 100% |
| Precision | 79.8% |
| F1 | 0.89 |
| Median returned tokens | 958 |
| Mean returned tokens | 1,095 |

Index modes over the 53 PRs: partial 27, semantic 1, syntax 25. Because review restores each PR's worktree (outside the repository tree, so no nested NuGet-config conflict), a majority of PRs load in partial or semantic mode, and the moat is exercised rather than hidden in syntax. Versus a grep baseline (rank the repository's C# files by title-token matches, admit to the budget), review wins decisively: grep reaches 53 percent changed-file recall at 14 percent precision. The prior-corpus recording (48 PRs, precision 89.6 percent at partial 12 of 48) is superseded by this run; the lower aggregate precision here reflects more PRs loading semantically, where review adds the wiring and support files a reviewer wants and the changed-files-only truth counts them against it.

### R4: Adjudicated support set, grep baseline, and two budgets (prior corpus)

These R4 figures were recorded on the prior corpus and are retained for the adjudicated-set methodology; they are re-run on the rebuilt corpus in a later V3.1 item. Changed-file recall is 100 percent by construction, so it does not test blast radius. R4 adds three things (`review.r4.json`, all 108 PRs, budgets 25,000 and 50,000): a grep baseline to beat, two budgets to read the precision-recall curve, and an adjudicated reading set (the support files a reviewer must read beyond the changed files) for a curated pilot of PRs.

| Truth / arm (25,000 budget) | Recall | Precision | Median tokens |
|------|------:|------:|------:|
| fuse vs changed files | 100% | 78% | 1,108 |
| grep vs changed files | 51% | 9% | - |
| fuse vs adjudicated set (5-PR pilot) | 60% | 74% | - |

Review beats the grep baseline decisively on both axes (100 versus 51 percent recall, 78 versus 9 percent precision). The two budgets give the same numbers because review returns compact scoped context (a median 1,108 tokens) far under both ceilings, so the precision-recall frontier is not exercised here; the budget is not the binding constraint. Against the adjudicated set (a single-adjudicator pilot of 5 PRs, with the support files read from each diff and recorded as `reading_set` in `prs.json`), review reaches 60 percent of the changed-plus-reading files at 74 percent precision. That meets the crown target on precision and on beating grep, but the 60 percent adjudicated recall is short of the 0.85 target: the blast radius is bounded in this mostly-syntax corpus and the pilot is small. Scaling the adjudication and the semantic-mode coverage is the honest path to closing it.

Findings, including the boundary:

- Changed-file recall is 100 percent by construction, since the changed files are seeded as must-keep. The signal in this suite is precision (79.8 percent on the rebuilt corpus) and F1 (0.89): review rarely returns a file the change did not touch or pull in.
- Review delivers a PR's context compactly. The median PR is scoped in 958 returned tokens, the mean in 1,095. This is where token efficiency lives in V3: a rendering and transport property of the scoped review output, not a separate reduction claim.
- The semantic blast radius shows up on the application, not the libraries. On eShopOnWeb (partial mode) review adds files beyond the changed set through the semantic graph, which lowers precision where the extra files are the wiring a reviewer would want: eShopOnWeb#322 scopes at 21 percent precision, eShopOnWeb#300 at 35 percent. The library repos load in syntax mode and scope tightly to the changed files, which is why aggregate precision stays high. The honest reading: the partial-mode application is where the blast radius is exercised, and there the wider set is the point, not noise.

## Suite C: Open-Ended Localization (the weakest mode)

Suite C is the open-ended floor. It gives `fuse localize` only a PR title, with no Git base, and asks it to find the changed files from the title text alone. This is the hardest mode and the one where Fuse is weakest.

On the V3.1 A3 rebuilt corpus (53 PRs, title only, no model, restore on), the baseline is:

| Metric | Value |
|--------|------:|
| Changed-file recall | 13.1% (95% CI 7 to 19) |
| Precision | 8.8% |
| Median returned tokens | 835 |
| Low-signal detection F1 | 1.00 |
| False-rejection on answerable | 5.8% (3 of 52) |
| Precision when confident | 18.5% (9 tasks) |

By bucket: identifier-rich 20 percent (n 15), natural-language domain 14 percent (n 29), formatting and route/API and test-only 0 percent (small n). Localize main-checkout index modes are partial 2, syntax 2 (NodaTime and Specification partial; Scrutor's NU1507 nesting conflict and eShopOnWeb's main checkout fall to syntax). This is the honest open-ended baseline the dense channel (A1) and the graph-dependent priors are measured against next. The contract metrics (low-signal F1, false-rejection) hold on the rebuilt corpus.

The remainder of this section (the R1, R2, R3, and S-series tables below) was recorded on the prior 108-PR corpus and is retained for the methodology; the numbers are not directly comparable to the rebuilt-corpus baseline above.

By signal bucket on the prior corpus (how much the title tells the engine where to look):

| Bucket | Mean recall |
|--------|------------:|
| identifier-rich | 40% |
| test-only | 33% |
| no-signal | 33% |
| natural-language domain | 23% |
| route/API | 12% |
| formatting | 8% |

The table above is the recorded baseline (`localize.json`). R1 reconnects the lexical ranker (`localize.r1.json`): the BM25F rank is now carried through to the candidate score (a stronger lexical match ranks higher, not just present), with one capped round of pseudo-relevance feedback that pulls in files sharing the top hits' vocabulary. Overall recall rises to 30.3 percent and the natural-language-domain bucket to 27 percent, at a precision cost (6.6 percent) and a higher median return (about 4,000 tokens) from the broader pool. Identifier-rich holds at 41 percent, short of the retired 49 percent lexical floor: recall here is averaged over multi-file PRs at a 20-candidate cap in a mostly-syntax corpus, so naming one file does not by itself lift the bucket past the floor.

R2 adds a warm dense channel: a per-chunk MiniLM embedding is persisted at index time (opt-in with a cached model via `FUSE_DENSE`), and a prose query is embedded and ranked by cosine over that index, so a file is found by meaning rather than shared words. The ablation is lexical (FTS plus graph, `localize.r1.json`) versus lexical plus dense (`localize.r2.json`):

| Bucket | Lexical (R1) | + Dense (R2) |
|--------|-------------:|-------------:|
| natural-language domain | 27% | 34% |
| overall recall | 30.3% | 34.6% |
| route/API | 0% | 12% |
| identifier-rich | 41% | 41% |
| precision | 6.6% | 7.6% |

The dense channel lifts the largest, weakest bucket (natural-language domain, 27 to 34 percent) and recovers route/API, which is its design target: prose titles that share no tokens with the code. Identifier-rich is unchanged because the lexical channel already finds a title that names its symbol. With no model present, indexing writes no vectors and retrieval is byte-identical to lexical, so the no-model floor holds.

R3 closes the low-signal gap with abstention (`localize.r3.json`). A query classifier on the localize path detects a title that names no code (merge, dependency-bump, or CI noise, or an empty query with no route, symbol, service, request, config, or base) and returns a low-signal verdict with a suggested next input (a base, route, or symbol) instead of candidates. Measured over the 108 PRs, low-signal detection F1 rises from 0.11 to 1.0 (9 true positives, 0 false positives, 0 false negatives). The no-signal bucket recall is now 0 by design: the engine declines rather than returning the lucky junk a title cannot support, which is the honest ceiling on this bucket (recall there is bounded by the input). Overall recall is 31.0 percent with the solvable buckets unchanged. A request that carries any structured signal is never downgraded.

### The signal-sufficiency contract (S3)

Because Fuse runs as an MCP server in a loop with a model, the answer to a low-signal request is not a low-precision list, it is to refuse and route. S3 turns the localize path into a graded contract (`localize.s3.json`): every request is graded **confident** (a candidate stands clear; return the tight set), **partial** (some signal, no clear winner; return a small flagged set with a navigation map), or **insufficient** (no usable anchor; return only a navigation map and an explicit ask). The grade is computed from the candidate score distribution with no model. The navigation map carries the top candidate areas, entry-point files and routes, and the nearest indexed symbols, so a refusal is a navigation step an exploring agent acts on, not a dead end. Strictness is opt-in: the default still returns a best-effort set so a one-shot client is never stranded, and `strict` hard-requires an anchor.

This reframes the headline. Bare-title recall is no longer the measure for this path; the contract metrics are:

| Metric | Value | Target |
|--------|------:|-------:|
| Correct-refusal on no-signal | 100% (low-signal F1 1.0) | >= 90% |
| False-rejection on answerable | 8.1% (8 of 99) | <= 10% |
| Precision when confident | 21.3% (18 tasks) | >= 20% |

The cutoff trades recall for precision as designed: graded states over the 108 PRs are confident 18, partial 73, insufficient 17, and the returned-set recall falls to 16.4 percent at a median 956 returned tokens (from 31.0 percent at 4,642) because a confident result returns only its leading cluster and a partial result a capped best-effort set. Subword identifier indexing (next) widens recall back under this cutoff. Every number holds with no model present.

### Subword identifier indexing (S1)

The default full-text tokenizer treats `ApplyRoundingMode` as one opaque token, so the prose word `rounding` does not match it. S1 closes that gap deterministically and offline (`localize.s1.json`): at index time every identifier in a chunk's name, declared symbols, and signature is split on camelCase, snake_case, acronym, and letter/digit boundaries into a dedicated full-text subtokens column (computed in C#, no tokenizer plugin, weighted below the exact name and above the body), and the query is split the same way before it hits the index. So `rounding` now finds `ApplyRoundingMode`, while an exact name match still ranks first. The schema version bumps and the index rebuilds on the next run.

Measured over the same 108 PRs against the S3 baseline (the correct comparison, since S1 is S3 plus subwords), every recall bucket gains at improved precision:

| Metric | S3 | + Subwords (S1) |
|--------|---:|----------------:|
| overall returned-set recall | 16.4% | 18.3% |
| identifier-rich | 27% | 30% |
| natural-language domain | 15% | 17% |
| precision | 9.7% | 10.7% |
| false-rejection on answerable | 8.1% | 7.1% |

Subwords lift recall and precision together because they find the right file, not just more files, and the extra matches pull a few previously-insufficient queries into a usable answer (false-rejection falls). Precision when confident is 19.4 percent (from 21.3, a within-noise move on 18 tasks at the widened recall). Identifier-rich at 30 percent stays well short of the retired 49 percent lexical floor: that floor was measured at a 20-candidate cap without the S3 cutoff, over a corpus that here loads mostly syntax (2 partial, 4 syntax). The subword field is the lever; the graded cutoff and the syntax-mostly corpus are the ceiling. Every number holds with no model present.

### Stemming and a comment bridge (S2)

S2 adds two more deterministic, offline vocabulary bridges that compose with subwords (`localize.s2.json`). A Porter-stemmed `stems` full-text column (computed in C#, query terms stemmed the same way, weighted lowest as a fuzzy bridge) collapses inflected variants, so `calculate` finds `Calculation` and `rounds` finds `rounding`; only letters are stemmed, so code tokens and digits pass through unchanged. A weighted `comments` field, fed by a comment extractor that pulls the prose out of line, block, XML-doc, and hash comments, indexes the developer's own vocabulary above body weight, because that is the vocabulary a natural-language query is written in. The schema version bumps and the index rebuilds on the next run.

Measured over the same 108 PRs against the S1 baseline:

| Metric | S1 | + Stemming/comments (S2) |
|--------|---:|-------------------------:|
| overall returned-set recall | 18.3% | 18.8% |
| precision | 10.7% | 11.7% |
| identifier-rich | 30% | 32% |
| test-only | 31% | 36% |
| false-rejection on answerable | 7.1% | 5.1% |
| precision when confident | 19.4% | 23.3% |

The bridges lift recall and precision again, drop false-rejection to 5.1 percent, and bring precision when confident back over the 20 percent target. The natural-language-domain bucket, which comments were expected to help most, is flat at 17 percent here: in this corpus the PR titles share little stemmed vocabulary with the code's comments, so the gain lands on the identifier-rich and test buckets instead. An honest corpus limit, not an engine ceiling. Every number holds with no model present.

### A structural prior: dependency centrality (S5)

After scoring, a dependency-centrality prior multiplies each candidate's score by a small capped factor (at most plus ten percent) proportional to its file's normalized node degree, so a widely-depended-on file outranks a leaf for an otherwise-tied query, bounded so it cannot promote a near-zero-score file. It is computed over the semantic graph, so in syntax mode (no edges) it is a no-op. The prior is verified by unit tests on a crafted graph, but corpus-wide localize is unchanged from S2 (`localize.s5.json`): the four syntax repositories have no edges and the two partial ones barely move, so the prior's effect is bounded by the mostly-syntax corpus, not by the prior. The git co-change half of this item is deferred: the co-change table has no collector yet, so a git-history miner is a separate piece of work.

### Graph-aware discovery (S7, opt-in)

Localization can optionally expand the selected candidates one hop through the typed graph, so a query that lexically hits only an interface also returns its implementation and a caller even when those share no vocabulary with the query (verified on an interface-to-implementation fixture). The expansion is bounded (top two seeds, depth one, at most five new neighbors) and a no-op in syntax mode. It is off by default and enabled per request (the `expand` option), because the structural blast radius widens recall but pressures precision. Both operating points are recorded:

| Metric | Default (off, `localize.s7.json`) | Expansion on (`localize.s7-expand.json`) |
|--------|----------------------------------:|-----------------------------------------:|
| overall recall | 18.9% | 19.1% |
| precision | 11.9% | 10.8% |
| median returned tokens | 1,098 | 1,352 |

On this corpus expansion buys almost no recall (plus 0.2 points) for a real precision cost, because only two repositories carry a graph and the changed-files-only ground truth scores the blast radius as noise. So the default stays off (byte-identical to S5) and expansion is offered as an explicit discovery operating point; its value shows where a graph exists, as the fixture demonstrates, and is bounded here by the mostly-syntax corpus.

### Multi-language readiness (S10)

The syntax-tier indexer is now provider-driven: a language-provider seam claims file extensions and extracts symbols and chunks with no compiler, and the indexer selects a provider by extension. C# is the existing Roslyn-syntax extractor behind the seam (unchanged), and a minimal Python spike provider is registered alongside. The readiness baseline is a no-regression check plus a spike: Suite A is unchanged after the refactor (22 of 22 wiring edges, recall and precision 1.0) and corpus localize is unchanged (recall 18.9 percent, `localize.s10.json`), while a mixed-language workspace test confirms a Python file indexes to symbols and chunks and is full-text-searchable end to end, including by an identifier subword (so the S1 subword feature carries to the spike language for free). The retrieval features all operate over the language-agnostic symbol, chunk, and edge tables, so they carry to a new language without rework. A `language` tag through the index and a semantic-tier (typed-graph) provider seam are the remaining multi-language foundation work.

Findings, including the known gap:

- Recall tracks the signal in the title. When the title names identifiers that appear in the code (identifier-rich, 41 percent after R1), localization does best; when the title is a domain phrase (27 percent), a route or API description, or a formatting note, it does worse. This is the expected shape of a title-only, no-base task.
- Low-signal detection is now solved by abstention (R3). The suite scores whether Fuse flags a title that carries no usable signal so it can decline rather than guess. After R3 the engine classifies the query and abstains on merge, dependency-bump, and CI noise (and an empty query with no structured input), returning a suggested next input instead of candidates. Low-signal detection F1 is 1.0 (9 true positives, 0 false positives, 0 false negatives), up from the 0.11 baseline. The no-signal bucket recall is 0 by design: declining beats returning junk a title cannot support.
- This is the floor, not the routed path. With a Git base available, the routed mode is change impact (Suite B), which is where the recall and precision are. Suite C is what Fuse can do from a sentence with no base, and it is reported as the weakest mode on purpose.

## Suite D: Agent Context Sufficiency (model-dependent)

Suite D replaces a structural bound with a real agent. It runs one Claude Code CLI driver, model `claude-sonnet-4-6`, over 12 PRs (two per repository) in two arms (native filesystem Read/Grep/Glob, and the Fuse MCP tools), one rollout each, for 24 rollouts. The run goes through the real `fuse.exe` so the fuse arm's MCP server is correctly wired, with `--restore` so the fuse arm indexes semantically where a repository restores. It measures the file recall of the set the agent gathered and the cumulative tokens it spent.

This suite is model-dependent and not byte-reproducible. The numbers depend on the model, its sampling, and the day; read them as a small, wide-CI sample, not a fixed measurement, and always read recall together with tokens.

| Arm | Mean file recall | Mean precision | Median cumulative tokens |
|-----|-----------------:|---------------:|-------------------------:|
| fuse | 30% | 44% | 211,502 |
| native | 26% | 43% | 209,182 |

Findings, including the shape:

- Fuse edges out native at parity cost. On this 12-PR sample, the fuse arm reached 30 percent mean recall to native's 26 percent, at comparable precision (44 versus 43 percent) and comparable token cost (211,502 versus 209,182). With the MCP server correctly wired and semantic indexing where the repo restores, the Fuse tools gather a slightly more complete set without spending more.
- The sample is still small. Twelve PRs, one rollout per arm, one model, one day, so the confidence interval is wide and neither mean is settled. This suite is a directional signal about whether the Fuse MCP tools help an agent, not a fixed benchmark.
- The full scale remains. Running 50 to 100 tasks at 3 rollouts needs a larger agent-compute budget; this run is 12 PRs at one rollout.

### Peer comparison (Fuse versus CodeGraph, coa-codesearch, and Serena)

The peer layer (A2) gives each tool the PR title and scores the returned file set against the changed files. Fuse runs through its V3 index-then-localize path; CodeGraph (the offline tree-sitter graph peer) runs `codegraph init` then `codegraph explore`; coa-codesearch (anortham/coa-codesearch-mcp, a .NET Lucene MCP server, built from source) and Serena (oraios/serena, an LSP-backed symbol-search MCP server, via uvx) are each driven by one headless `claude` rollout restricted to that server's search tools. This is the run that makes a head-to-head claim falsifiable: all four arms are present, none stubbed. On the V3.1 A3 rebuilt corpus at a 50,000-token budget (`layer6-peers.json`):

| Arm | Sample (PRs) | Mean recall | Mean precision | Mean tokens |
|-----|------------:|------------:|---------------:|------------:|
| fuse | 12 | 19% | 19% | 10,717 |
| codegraph | 12 | 9% | 11% | 3,582 |
| coa | 4 | 9% | 1% | 3,382 |
| serena | 4 | 34% | 27% | 1,538 |

Read this carefully, because the aggregate hides a sampling difference. The deterministic arms (fuse, codegraph) ran all 12 sampled PRs; the model-driven arms (coa, serena) ran one claude rollout per PR and were bounded to 1 PR per repository (4 PRs total) to keep the rollout compute affordable, so their column is a smaller, different sample. On the deterministic 12-PR comparison, Fuse leads CodeGraph decisively on recall (19 versus 9 percent) at higher precision. On the four shared model-driven PRs, the honest reading is that Serena's higher aggregate is dominated by a single tiny-repo outlier: on Scrutor#6 (a 2-file change in a 45-file repo) Serena's symbol search returns the whole repo and scores 100 percent recall, while on the two substantive PRs (eShopOnWeb#949, NodaTime#621) Fuse leads or ties Serena (38 versus 12 percent, 33 versus 22 percent). coa returns a long ranked path list (102 files on NodaTime#621), so its recall is moderate but its precision is near zero. The token columns are not directly comparable: Fuse and CodeGraph return source the agent reads directly, while coa and Serena return path or snippet lists the agent would still open. The model-driven arms are not byte-reproducible and the per-rollout setup time is captured as `setup_ms`; a full 50-to-100-PR model-driven run remains a larger compute budget. This is a small sample with Fuse on its C# home field, reported straight, and no single arm-versus-arm point should be over-read.

## Suite E: Token Reduction and Fidelity (offline continuity)

Suite E is the offline continuity layer. For each repository it counts the raw o200k_base tokens of the C# files, reduces them at each level through the shipped reduction path, counts the reduced tokens, and at skeleton level checks how many public and protected types and methods survive. Fidelity is counted by parsing the raw source with Roslyn as an independent ground truth, so the measure is not circular. Run with `fuse eval reduce`.

The Roslyn skeleton keeps every public and protected type and 99 to 100 percent of public methods while removing 37 to 55 percent of tokens; public-API level removes 47 to 60 percent. The percentages are reduction against the raw C# token count.

## What Is Not Measured on V3

These were measured on the retired V2 surface and have not been re-run on the V3 semantic engine. Do not quote the old percentages as V3 results:

- The peer-scoper comparison at full scale. A2 ran all four arms (Fuse, CodeGraph, coa-codesearch, Serena) head-to-head on the rebuilt corpus (see Peer comparison above), but the model-driven arms (coa, Serena) ran a bounded 4-PR sample; a full 50-to-100-PR model-driven run remains a larger compute budget.
- Full task resolution at scale: driving an agent across arms and scoring its patch against a test oracle over many tasks. The deterministic core of that harness (apply a patch in an isolated worktree, run a test oracle, score pass@1) is built and tested (R9); the agent-driven, at-scale run is compute-bounded and not run here.

Warm-call latency is now measured (R7); see [Performance and Benchmarking](/docs/project/performance). Each remaining item above is not-yet-measured at scale on V3, not a claim. Treat any such figure as illustrative until a V3 suite produces it.

## How to Read These Results Together

Suite A is the clean, deterministic claim: with a semantic workspace, Fuse resolves .NET wiring exactly (22 of 22 edges across the wiring catalog). Suite B is the practical strength: review scopes a real PR at high precision and 100 percent changed-file recall in around a thousand tokens, beating a grep baseline, with the semantic blast radius exercised on the application in partial mode. Suite C is the honest floor lifted by hybrid retrieval: from a title alone with no base, the lexical-plus-dense stack reaches 34 percent on the natural-language bucket, and the engine now abstains on a no-signal title (low-signal detection F1 1.0) rather than returning junk. Suite D is a small, model-dependent signal that the Fuse MCP tools can gather the right files cheaply when the task names the wiring. The recurring caveat across the corpus suites is index mode: most repositories load in syntax or partial mode in this environment, so the corpus numbers sit below the semantic ceiling that Suite A and the application demonstrate.

## Blocked and Out of Scope

- Full semantic load across the whole corpus. Most corpus repositories load in syntax or partial mode here because the offline harness does not provision a restored MSBuild workspace per repository. The full semantic graph is exercised on eShopOnWeb (partial) and the in-repo fixture (full). A restored-workspace corpus run is the way to lift the corpus suites toward the Suite A ceiling.
- Task resolution at scale. Suite D measures context sufficiency (did a real agent gather the files the task needs, and at what cost). The deterministic core of a SWE-bench-style harness is now built and tested (R9): it materializes an isolated worktree at the base commit, applies a candidate patch, runs a test oracle, and scores pass@1, with a trivial fixture proving the oracle (a known-good patch passes, a known-bad patch fails, and a failing task does not corrupt the next). The agent-driven, at-scale run across arms (with and without Fuse) over many tasks needs provisioned model credentials and a large compute budget and is not run here.
- Cross-machine timing. No V3 latency is published; see Performance for the qualitative warm-index model.
- Tokenizer scope. All counts use `o200k_base`. Absolute counts differ under other encodings.
