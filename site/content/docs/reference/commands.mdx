Fuse exposes its work through ten commands. Three of them (`fuse`, `fuse dotnet`, and `fuse wiki`) produce a fusion from a source directory and differ only in which file types they collect and how they reduce them. A fourth, `fuse ask`, takes a task and routes it to a scoping strategy automatically. A fifth, `fuse reduce`, compacts a caller-supplied set of files or piped content instead of walking a directory. Two more (`fuse explain` and `fuse verify`) run a fusion in memory to report on it without writing output. The remaining three (`fuse init`, `fuse mcp install`, and `fuse mcp serve`) manage configuration and the agent server and run no fusion of their own. This page states what each command does, when to reach for it, and which option set applies.

This page is for engineers choosing a command and for maintainers who need the exact dispatch behavior. Stakeholders can read the summary table alone.

## Purpose and Scope

This page covers the command surface: names, descriptions, and one example invocation each. It does not document the flags themselves. Every fusion command (`fuse`, `fuse dotnet`, `fuse ask`, `fuse wiki`) shares the option set in the [Options reference](/docs/reference/options), and `fuse dotnet` adds .NET-specific flags described there; `fuse ask` adds `--task`. The `fuse explain` and `fuse verify` commands share the same options plus a subset of the .NET scoping and reduction flags. The commands `fuse init`, `fuse mcp install`, and `fuse mcp serve` take no fusion options.

## Summary

| Command | Fuses | Reduction | Scoping | Options |
|---------|-------|-----------|---------|---------|
| `fuse` | Any files, using template defaults and shared options | None beyond shared content trimming | None | Shared |
| `fuse dotnet` | A .NET project, including C#, F#, and web files | C# reduction | Focus, changes, query | Shared plus .NET |
| `fuse ask` | A .NET project, scoped by a task it routes for you | C# reduction (skeleton or aggressive) | Auto: skeleton, focus, or search | Shared plus `--task` |
| `fuse wiki` | An Azure DevOps wiki repository (Markdown only) | None beyond shared content trimming | Changes | Shared |
| `fuse explain` | Nothing; previews a .NET fusion in memory | Same as `fuse dotnet` | Focus, changes, query | Shared plus a scoping subset |
| `fuse verify` | Nothing; checks a .NET fusion in memory | Same as `fuse dotnet` | Focus, changes, query | Shared plus a scoping subset |
| `fuse init` | Nothing; writes a config file | Not applicable | Not applicable | None |
| `fuse mcp install` | Nothing; registers Fuse with MCP clients | Not applicable | Not applicable | `--client`, `--scope`, `--rules`, `--command` |
| `fuse mcp serve` | Nothing; starts the MCP server | Not applicable | Not applicable | None |
| `fuse reduce` | A named set of files, or stdin content (to stdout) | C# reduction by `--level` | None | `--files`, `--stdin`, `--ext`, `--level`, `--max-tokens` |
| `fuse models` | Nothing; manages optional models for opt-in features | Not applicable | Not applicable | `status`, `download`, `remove` |

## fuse

The root command is a flexible file combining tool for developers. It collects files using template defaults and the shared options, then emits one fusion. Use it when no language-specific reduction is needed and you want direct control over which extensions to include.

```bash
fuse --directory ./docs --include-extensions .txt,.md
```

The shared options for this command are documented in the [Options reference](/docs/reference/options).

## fuse dotnet

This command fuses a .NET project, including C#, F#, and web files. It applies C# reduction and supports scoping a fusion to the files relevant to a task through focus, change, and query modes. It carries .NET-specific options beyond the shared set, including the C# reduction flags, skeleton extraction, and the scoping flags.

```bash
fuse dotnet --directory ./src --level aggressive
```

The .NET-specific flags and the three mutually exclusive scoping modes are documented in the [Options reference](/docs/reference/options). C# structural reduction is set with a single `--level` flag (`none`, `standard`, `aggressive`, `skeleton`, or `publicApi`). The same command also exposes the table-of-contents survey (`--toc`), generated-code collapse (`--collapse-generated`), and the on-disk analysis store (`--index`). For task-oriented usage, see the [Scoping to What Matters](/docs/concepts/scoping) and [Reducing Tokens](/docs/concepts/reduction-levels) guides.

## fuse ask

This command takes a natural-language task and a token budget and picks the scoping strategy for you, the same deterministic routing the `fuse_ask` MCP tool uses, so the routed agent surface is available on the CLI (and to the benchmark harness and CI) without an MCP client. A task that asks about the codebase as a whole routes to a skeleton; a task naming exactly one type routes to focus on that type, falling back to search when the type does not resolve; everything else routes to relevance search. The choice is deterministic: the same task and budget always resolve to the same plan. The budget comes from `--max-tokens` (default 20,000).

```bash
fuse ask --task "where is rate limiting handled" --directory ./src --max-tokens 25000
```

It accepts the shared options plus `--task`. No model is called; the routing is heuristic and explainable. See [Ask one question](/docs/scenarios/ask-one-question) for the MCP equivalent.

## fuse wiki

This command fuses an Azure DevOps wiki repository and includes only `.md` files. Use it to combine a wiki tree into a single document for review or for an agent to read in one call.

```bash
fuse wiki --directory ./my-wiki
```

This command accepts the shared options in the [Options reference](/docs/reference/options). It applies no language reduction beyond the shared content handling.

## fuse explain

This command previews a .NET fusion without writing anything. It runs collection, scoping, and in-memory reduction exactly as `fuse dotnet` would, then prints which files would be included with a per-file token estimate, which collected files would be excluded, and the estimated token total. Use it to check the effect of a focus seed, a query, a change range, or a reduction mode before committing to a full run.

```bash
fuse explain --directory ./src --focus OrderService --depth 2
```

It accepts the shared options plus a subset of the `fuse dotnet` scoping and reduction flags (`--focus`, `--query`, `--query-top`, `--depth`, `--level`), documented in the [Options reference](/docs/reference/options).

## fuse verify

This command reports how much of a project's public API surface survives a .NET fusion. It runs a fusion in memory, then compares the public and protected types and methods, and ASP.NET route templates, declared in the included source against the fused output, printing the preserved percentage for each category. Pass `--json` for a machine-readable result. Use it to confirm that a reduction level keeps the API you expect: a drop under the `none`, `standard`, or `aggressive` level signals lost API, while `skeleton` and `publicApi` are signatures only by design.

```bash
fuse verify --directory ./src --level aggressive
fuse verify --directory ./src --level skeleton --json
```

The source side is parsed by Roslyn. It accepts the same scoping and reduction subset as `fuse explain`, plus `--json`, documented in the [Options reference](/docs/reference/options).

## fuse init

This command creates a `fuse.json` configuration file in the current directory. It writes the file only when one is absent; if a `fuse.json` already exists, the command writes nothing and reports an error. It takes no fusion options.

```bash
fuse init
```

The scaffold it writes and every supported key are documented in the [Configuration Keys reference](/docs/reference/configuration-keys).

## fuse mcp install

This command registers Fuse as an MCP server with Claude Code, Cursor, and GitHub Copilot.
It writes the client-specific JSON config (or invokes the Claude CLI for user-scope Claude
registration) so the client can launch `fuse mcp serve` automatically. You do not run
`fuse mcp serve` manually for normal agent use.

```bash
fuse mcp install --rules
fuse mcp install --client cursor --scope user
```

Use `--client` to pick `claude`, `cursor`, `copilot`, or `all` (default). Use `--scope
project` (default) to configure the current directory, or `--scope user` to register Fuse
for every project on this machine. Add `--rules` (recommended) to also write a short rule
into each client's instruction file (`CLAUDE.md`, `.cursor/rules/fuse.mdc`,
`.github/copilot-instructions.md`) that biases the agent toward the `fuse_*` tools; it is
project-scoped and idempotent. Use `--command` to override the executable the client
launches (default: the running `fuse` binary or `fuse` on PATH).

For what each client reads and the manual JSON shapes, see the [Agent Integration overview](/docs/start/connect-your-ai).

## fuse mcp serve

This command starts the Fuse MCP server for AI agent integration. In normal use your MCP
client launches it as a child process; you run this command directly only when debugging
or testing the server. The server communicates over stdio: stdout is reserved for the
Model Context Protocol byte stream, so all logging is routed to stderr to avoid corrupting
the protocol. It takes no fusion options; each fusion is requested by the connected client
through a tool call.

```bash
fuse mcp serve
```

The server keeps the on-disk store at `.fuse/fuse.db` warm across calls, so a multi-call task pays the analysis cost once.

For client setup and the exposed tools, see the [Agent Integration overview](/docs/start/connect-your-ai).

## fuse reduce

This command compacts a specific set of files, or content piped on stdin, and writes the
reduced result to stdout. Unlike the fusion commands it does not walk a directory: it
reduces exactly what you name, which is useful for shrinking a known file set or piping
content through Fuse as a filter. The agent-facing equivalent is the `fuse_reduce` MCP tool.

```bash
fuse reduce --files src/Order.cs src/OrderService.cs --level skeleton
cat generated.cs | fuse reduce --stdin --ext .cs --level skeleton
```

Use `--files` for a list of paths (absolute or relative to `--directory`), or `--stdin` to
read content from standard input with `--ext` selecting the reducer. The two are mutually
exclusive. `--level` sets the C# reduction level and `--max-tokens` caps the output. The
result goes to stdout so the command composes in a pipeline.

## fuse models

This command manages the optional on-disk models that back opt-in features. Today that is the dense rerank
embedder (all-MiniLM-L6-v2), used when `FUSE_RERANK=1` is set. Downloading is always explicit; scoping never
fetches a model mid-run, and absent a model the query path stays on the lexical BM25F path.

```bash
fuse models status     # whether the model is cached, and where
fuse models download   # fetch it (about 23 MB to ~/.fuse/models), verified by SHA-256
fuse models remove     # delete the cached model
```

Dense rerank is off by default and is not the MCP default, because it did not beat the lexical path on the
benchmark corpus (see [Benchmarks](/docs/project/benchmarks)). The cache location follows `FUSE_USER_DATA`
(default `~/.fuse`).
