The Fuse MCP server exposes thirteen tools an AI client can call to index a .NET workspace and serve precise, provenance-backed context from a Roslyn-backed semantic graph. MCP (Model Context Protocol) is the open protocol that lets an AI client call external tools. This page documents each tool and its parameters so an agent or its author can choose the right call and supply correct arguments.

This page is for engineers writing agent instructions and for experts who need the exact default of each parameter. For the recommended order in which to call these tools, see [Context for an agent](/docs/scenarios/context-for-an-agent). The equivalent CLI flags are in [Commands Reference](/docs/reference/commands).

## Purpose and scope

Read tools build the persistent index on first use, so an explicit `fuse_index` call is optional. The index lives in a single SQLite file at `.fuse/fuse.db` inside the workspace. Tools return their result in the tool response; only `fuse_index` writes (the index). Token budgets are honored by the plan, and secret redaction runs before any source reaches the payload.

Choosing a tool:

- Use `fuse_review` for PR or change work when a git base exists.
- Use `fuse_resolve` when a task names a route, interface, service, request, handler, or config section.
- Use `fuse_localize` for open-ended tasks.
- Use `fuse_context` after localize or resolve, unless you want one-shot context.
- Use `fuse_find` for exact text, path, or symbol lookup.
- Use `fuse_neighbors` to explore iteratively: the graph neighborhood of a file, the callers and implementers of a symbol, or the central files of an area.

## fuse_index

Build or refresh the persistent semantic index. Run once before the read tools, or to pick up changes. Returns a summary (mode, files, projects, symbols, routes).

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory to index |

The index mode is `semantic` when the workspace loads through MSBuild/Roslyn, `partial` when some projects fall back, and `syntax` when no project loads.

## fuse_map

Print a map of the workspace: indexed symbols, routes, and counts. The cheap first call to understand structure before fetching context.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory |
| `detail` | string | `all` | `symbols`, `routes`, or `all` |
| `maxRows` | int | 200 | Maximum rows per section |

## fuse_localize

Localize a task to ranked candidate files and symbols. Returns candidates with reasons and token costs, no source bodies. The cheap first step of an iterative workflow; follow with `fuse_context`.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory |
| `task` | string | null | Free-text task or query |
| `route` | string | null | A route to resolve, for example `POST /api/orders/{id}` |
| `symbol` | string | null | A symbol to focus on |
| `service` | string | null | A service to resolve to its implementation |
| `request` | string | null | A request or command to resolve to its handler |
| `config` | string | null | A config section to resolve to its options type |
| `changedSince` | string | null | A git base ref whose changed files seed the candidates |
| `maxCandidates` | int | 50 | Maximum candidates to return |
| `strict` | bool | false | Refuse an insufficient request and return only a navigation map, instead of a low-confidence best-effort guess |
| `expand` | bool | false | Enrich the selected candidates with their typed-graph neighbors (implementers, callers, config) for discovery; widens recall but pressures precision |

### The graded response

`fuse_localize` grades each request into one of three states, computed from the candidate
score distribution with no model, and leads its output with the state (`localize [confident]: ...`):

- **confident**: a candidate or a tight cluster stands clear; the tight set is returned and precision is high.
- **partial**: there is signal but no clear winner; a small best-effort set is returned, flagged low-confidence, with a navigation map of refinement options.
- **insufficient**: no usable anchor; the engine refuses and routes, returning a navigation map (top candidate areas, entry-point files and routes, nearest symbols) and an explicit ask for a sharper input, instead of a guess.

The navigation map turns a refusal into a navigation step: an exploring agent reads the
areas and nearest symbols to pick its next, better-anchored query. By default (best-effort)
a partial or insufficient request still returns a candidate set so a one-shot client is
never stranded; with `strict: true` an insufficient request returns only the map. A
no-signal title (a merge, dependency-bump, or CI title with no code reference) abstains in
both modes, since it can only return noise. Provide one of the structured parameters above,
or a name from the navigation map, to get a confident answer.

## fuse_resolve

Deterministically resolve .NET wiring: a service to its implementation, a request to its handler, a route to its action, a config section to its options, or a symbol to its declaration. No source bodies.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory |
| `service` | string | null | Service to resolve to its implementation |
| `request` | string | null | Request or command to resolve to its handler |
| `route` | string | null | Route to resolve to its action |
| `config` | string | null | Config section to resolve to its options type |
| `symbol` | string | null | Symbol name to resolve to its declaration |

Supply exactly one of the five.

## fuse_context

Plan and emit context (source bodies at mixed render tiers, a semantic manifest, and per-file provenance) for a set of seeds. Feed it the file paths from `fuse_localize` or the names from `fuse_resolve`.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory |
| `seeds` | string[] | null | Symbol seeds |
| `files` | string[] | null | File path seeds (for example paths returned by localize) |
| `services` | string[] | null | Service seeds to resolve and expand |
| `requests` | string[] | null | Request or command seeds to resolve and expand |
| `configs` | string[] | null | Config section seeds to resolve and expand |
| `routes` | string[] | null | Route seeds |
| `depth` | int | 2 | Graph expansion depth |
| `maxTokens` | int | 0 | Token budget; must-keep seeds are always included (0 means no limit) |
| `format` | string | `xml` | `xml`, `markdown`, or `json` |
| `sessionId` | string | null | Session id; files already sent unchanged in the session are elided |

## fuse_review

Review the semantic impact of a change since a git base ref: the changed files, the blast radius (callers, DI consumers, route and request handlers, options consumers, tests), and the packed context. The flagship tool for PR and change work.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory |
| `changedSince` | string | `HEAD` | Git base ref to diff against (branch, commit, or `HEAD~N`) |
| `maxTokens` | int | 0 | Token budget; changed files are always kept (0 means no limit) |
| `includeTests` | bool | true | Include related test files |
| `format` | string | `xml` | `xml`, `markdown`, or `json` |
| `sessionId` | string | null | Session id; files already sent unchanged in the session are elided |

## fuse_find

Exact lookup: a symbol by name, a file by path fragment, or text by full-text search. Use instead of broad grep when the name or path is known.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `query` | string | required | The name, path fragment, or text to find |
| `path` | string | `.` | Workspace directory |
| `kind` | string | `all` | Restrict to `symbol`, `path`, `text`, or `all` |

## fuse_neighbors

Iterative exploration primitives, ranked and bounded with provenance and no source bodies. Specify exactly one of `file`, `symbol`, or `centralIn`. Chain these calls to turn a weak first guess into a strong few-call funnel.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory |
| `file` | string | null | A file whose graph neighborhood to return (callers, implementers, consumers, config, plus same-folder cohesion) |
| `symbol` | string | null | A symbol whose callers and implementers to return (incoming typed edges) |
| `centralIn` | string | null | An area (folder prefix, or empty for the whole workspace) whose structurally central files to return |
| `limit` | int | 20 | Maximum results to return |

The neighborhood falls back to same-folder cohesion where the graph is sparse, so a result is never empty; central files are ranked by node degree and are empty in syntax mode (no edges). These primitives compose the structural priors and graph expansion into operations shaped for exploration.

## fuse_refactor

Compiler-executed solution-wide rename: renames a symbol and every reference through Roslyn's rename engine and returns the change as a staged diff, never touching the working tree. Because Roslyn drives it, a same-named but unrelated symbol (a local, a different type) is not renamed, which a textual find-and-replace cannot guarantee. It answers only when the whole solution loads cleanly and abstains otherwise, because a partial rename is worse than none. Review the diff and re-check with `fuse_check` before applying; a rename that crosses a boundary Roslyn does not see (an identifier in a string, a reflection call) surfaces there rather than as a silently committed bug. The first cut supports rename; change-signature is future work.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory |
| `symbol` | string | required | The simple name of the symbol to rename |
| `newName` | string | required | The new name |

## fuse_check

Typecheck a proposed single-file edit and get the compiler errors and warnings the change would produce, without writing the file. Verification never shrugs (the grade ladder, Decision D11): every answer is stamped with a `verification grade`.

- **oracle** grade: a speculative in-memory typecheck against the tier-1 build-captured compilation (via the out-of-process build-capture worker, configured with `FUSE_BUILD_CAPTURE_WORKER`). The patch is applied to the rehydrated compilation in memory; no disk write, no second build.
- **build** grade: when no oracle substrate is available, Fuse runs the real `dotnet build` toolchain scoped to the owning project and parses its diagnostics into the same shape. This is ground truth (the compiler itself answered) at the cost of build latency (tens of seconds), reported with the elapsed time. The working tree is never written: the owning project is mirrored to a temporary directory, the proposed content replaces the one file in the copy, its `<ProjectReference>` includes are rewritten to absolute paths pointing at the untouched originals, and the build runs there.
- **abstain**: returned only when even the toolchain cannot run (no buildable project, or a build that exceeds the timeout), always naming the missing prerequisite.

The response opens with the grade line so a client knows the class of truth and the latency to expect before reading the verdict. When a diagnostic is an API-shape error an agent commonly hits (CS1061 a member that does not exist, CS0246 a type that could not be found), the response attaches a repair packet (R6): the receiver type's real members with the nearest names first, or the nearest type names in the index, drawn from the persisted symbol table so the fix does not cost another round-trip. A packet is added only where a concrete suggestion is possible, never as empty padding.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Workspace directory |
| `file` | string | required | The repo-relative path of the file being changed |
| `content` | string | required | The proposed full new content of that file |

## fuse_impact

Blast radius for a symbol before you edit it: the callers, implementers, consumers, and referencing types a change would touch, enumerated from the persisted semantic graph (the R5 reference edges plus the wiring edges). No source bodies. The exact signature-change break set (which call sites would no longer bind) is an oracle-grade answer that needs a tier-1 load; until one is available the tool reports it as unavailable rather than guessing, and returns the graph-grade blast radius. Reverse traversal is the store's incoming-edge query, so the result is deduped by source node. The response also lists the covering tests for the symbol (the M1 selection primitive): the tests that reach it through R5's DI-resolved `tests` edges, called out distinctly from the blast radius so you can run just that subset with your own `dotnet test --filter`. The covering set is a lower bound bounded by R5 edge completeness (a test reached only by reflection has no edge and is not listed), never presented as "all the tests". The response opens with an availability header (index mode, whether tier-1 build capture is configured, and the freshness stamp) so a client can see the grade of the answer before reading it.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `symbol` | string | required | The symbol (simple or qualified name) whose blast radius to compute |
| `path` | string | `.` | Workspace directory |
| `limit` | int | 50 | Maximum impacted items to return |

## fuse_signatures

Batch exact-signature lookup: for a set of symbol names (simple or fully qualified), returns each declared signature, kind, accessibility, containing type, and location from the semantic index, in one call instead of many grep-and-read round-trips. This is the compiler-shaped answer to "what is the exact shape of these members". A signature is recorded at the semantic tier; in syntax mode a symbol may have no recorded signature, and the tool says so per symbol rather than inventing one. A requested name with no match is reported, not silently dropped. The response opens with an availability header (index mode, whether tier-1 build capture is configured, and the freshness stamp) so the grade of the answer is explicit.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `names` | string[] | required | Symbol names to look up (simple name or fully qualified) |
| `path` | string | `.` | Workspace directory |
| `limitPerName` | int | 5 | Maximum matches to return per requested name |

## fuse_changeset

The speculative staging area: propose a changeset, verify it, select its tests, then commit or abandon it, all without touching the working tree until you choose to. An agent creates a session, stages one or more single-file edits into it, diagnoses them with the speculative typecheck (`fuse_check`), selects the tests that cover the changed symbols (from the persisted `tests` edges), and then either promotes the changeset (which writes the staged edits to the working tree) or discards it (which leaves the tree untouched). Two changesets over the same base are isolated: staging into one never affects the other. The edit-verify loop becomes propose-oracle-commit. This first cut re-captures the build per diagnose (no resident compilation yet), so diagnose latency tracks a build; in-process test execution is out of scope by design (the select operation returns the covering set to run with your own `dotnet test --filter`).

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `op` | string | required | The lifecycle operation: `create`, `stage`, `list`, `diagnose`, `select`, `promote`, or `discard` |
| `path` | string | `.` | Workspace directory |
| `session` | string | null | The session id (required for every op except `create`) |
| `file` | string | null | The repo-relative file path (for `stage`) |
| `content` | string | null | The proposed full new content of the file (for `stage`) |
| `limit` | int | 50 | Maximum covering tests to return (for `select`) |

Only `promote` writes to disk, and only when called. A `discard` (or never promoting) leaves the working tree exactly as it was.

## fuse_reduce

Compact a specific set of files (or raw content) by running Fuse's reduction, without collecting a whole directory. Pass `files` or `content` with `extension`.

| Parameter | Type | Default | Meaning |
|-----------|------|---------|---------|
| `path` | string | `.` | Base directory for resolving relative file paths |
| `files` | string[] | null | File paths to reduce |
| `content` | string | null | Raw content to reduce instead of files |
| `extension` | string | `.cs` | Extension that selects the reducer for `content` |
| `level` | string | `standard` | `none`, `standard`, `aggressive`, `skeleton`, or `publicApi` |
| `maxTokens` | int | 0 | Maximum tokens the reduced output may use (0 means no limit) |
