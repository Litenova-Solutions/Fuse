---
title: Cut tokens for a .NET project
description: Render a .NET solution, or a known set of files, into a token-efficient payload while keeping the public API intact.
---

**Goal:** take a .NET solution, or a set of files you already have, and produce a payload
that costs far fewer tokens than the raw source, without losing the public API.

## Reduce a Known Set of Files

When you already know which files you want, `fuse reduce` compacts them without indexing a
directory:

```bash
fuse reduce --files src/OrderService.cs --files src/Order.cs --level aggressive
```

`--level aggressive` applies the full C# structural reduction: it removes comments, usings,
namespaces, and regions, then compresses aggressively, while keeping the public API. The
levels run from `none` (lossless cleanup) to `skeleton` and `publicApi` (signatures only).
See [Reduction levels](/docs/concepts/reduction-levels) for the full ladder.

## Reduce an Area to a Budget

When you want an area of the solution rather than named files, plan context to a token
budget and let Fuse choose the render tier per file:

```bash
fuse context ./src --seed OrderService --max-tokens 25000
```

`fuse context` reduces the seed neighborhood to fit the budget: it keeps the must-keep
seeds in full and renders peripheral files more aggressively (compressed, or skeleton) so
the whole set fits. The output opens with a manifest listing each file and its token cost,
then the bodies, with per-file provenance.

## The Measured Cut

On the benchmark corpus, the default and `aggressive` reductions remove 6 to 46 percent of
tokens while keeping 99 to 100 percent of public types and methods, verified by an
independent Roslyn oracle. The skeleton tier removes 39 to 56 percent for an architecture
view. The cut is not deletion: the public surface a model reads the code for is still
there.

## When to Use It

Reach for `fuse reduce --level aggressive` when you have a known set of files and want the
most reliable cut with the API intact. Reach for `fuse context --max-tokens` when you want
an area scoped and reduced to fit a window. If you only need the shape of the codebase, a
[skeleton survey](/docs/scenarios/survey-cheaply) cuts far more. If the result is still too
large, [scope it tighter](/docs/concepts/scoping) to the code the task touches.
