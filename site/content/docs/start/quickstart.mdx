This takes an installed Fuse 4.0.0 tool to one graded result from a real
repository. If you have not installed Fuse yet, see [Install](/docs/start/install).

Fuse runs two ways: handed to an AI agent through the MCP server, or as a CLI in your
terminal. The agent path is primary because `fuse_check` can typecheck a proposed
single-file edit before it is written. The CLI `fuse check` command is different: it
reports the ambient delta from on-disk edits through a running host and does not accept
proposed content.

## Hand It to an Agent

From your project root, run one setup command:

```bash
fuse mcp install --rules
```

Restart or reload your AI client, then send one activation command:

```text
Call fuse_workspace with action=status for this repository. Return its index mode,
freshness, and verification grade exactly as Fuse reports them.
```

These two commands end in a real Fuse result with a grade. If the agent only lists tools
or says the server is connected, ask it to make the `fuse_workspace` call.

Your client launches `fuse mcp serve` as a local child process when MCP is enabled. Source
and the derived `.fuse/fuse.db` index stay on the machine. The first call builds the index,
so a large repository can take from a few seconds to tens of seconds before the first
answer. Later calls reuse the warm index.

Fuse serves a syntax index first, then attempts an MSBuild and Roslyn upgrade. Repositories
with missing SDKs, failed restores, or project build errors can remain in `partial` or
`syntax` mode. Fuse reports that downgrade. Compiler verification falls back to a scoped
real `dotnet build` and labels the answer build grade when no resident or build-captured
compilation is available.

The `--rules` flag writes MCP configuration and client instruction files. Use
`--scope user` to register Fuse for every project on this machine. See
[Connect to your AI](/docs/start/connect-your-ai) for exact paths, network posture, and
rollback steps.

For change work, one `fuse_review` call returns the changed files and their blast radius:

```text
fuse_review(
  path="C:/Projects/MyApp/src",
  changedSince="origin/main",
  maxTokens=25000
)
```

For an open-ended task, localize first, then plan context for the candidate files:

```text
fuse_find(path="C:/Projects/MyApp/src", kind="task", query="where is the discount applied at checkout?")
fuse_context(path="C:/Projects/MyApp/src", files=["src/Checkout/DiscountPolicy.cs"], maxTokens=25000)
```

## Build the Index

Point Fuse at a .NET source directory once to build the persistent index:

```bash
fuse index ./src
```

Fuse loads the workspace through MSBuild and Roslyn when it can, and falls back to
syntax-level indexing otherwise. The index lives in a single SQLite file at
`.fuse/fuse.db`. The read commands build it on first use, so this explicit step is
optional, but running it once up front makes the first read call fast.

## Read What It Reports

When indexing finishes, Fuse reports the shape of what it built:

```text
Indexed workspace ./src
Mode: semantic
Files: 218  Projects: 4  Symbols: 3,140  Routes: 12
Store: .fuse/fuse.db
```

The mode is `semantic` when the workspace loaded through MSBuild and Roslyn, `partial`
when some projects fell back, and `syntax` when no project loaded. The counts tell you
what the graph holds. `fuse diagnostics ./src` prints this state at any time.

## Survey the Workspace

Print a cheap map of what was indexed before fetching any source:

```bash
fuse map ./src --detail all
```

`fuse map` lists indexed symbols, routes, and counts, no bodies, so you can judge the
shape of the workspace at low token cost.

## Resolve What Runs

Ask the graph what a name actually maps to:

```bash
fuse resolve ./src --service IOrderService
fuse resolve ./src --request CreateOrderCommand
fuse resolve ./src --route "POST /api/orders/{id}"
```

Each prints the resolved target (the concrete implementation, the handler, the action)
with provenance, and no bodies.

## Plan and Emit Context

Turn a set of seeds into packed source bodies at a budget:

```bash
fuse context ./src --seed IOrderService --max-tokens 25000 --format xml
```

The output opens with a **manifest** that lists each included file with its token cost,
then the source bodies at mixed render tiers, with per-file **provenance** that records
why each file is present. Set `--format markdown` or `--format json` for other shapes.

## Review a Change

When a git base exists, review the semantic impact of a branch in one call:

```bash
fuse review ./src --changed-since origin/main --max-tokens 25000
```

This returns the changed files, the blast radius (callers, DI consumers, route and
request handlers, options consumers, tests), and the packed context.

## Set Project Defaults

To avoid repeating flags, write a configuration file once:

```bash
fuse init
```

This scaffolds a `fuse.json` you can edit. See
[Configuration keys](/docs/reference/configuration-keys) for every supported key.

## Next

Run a branch-scoped review and inspect the provenance attached to each included file:

```bash
fuse review ./src --changed-since origin/main --max-tokens 25000 --format markdown
```

To reset the walkthrough, stop the client and delete `.fuse/`. It contains derived data
only. Remove MCP configuration and rule entries with the
[rollback steps](/docs/start/connect-your-ai#rollback).
