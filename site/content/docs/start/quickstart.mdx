This walks you from an installed tool to a fused payload you can read. If you have
not installed Fuse yet, see [Install](/docs/start/install).

Fuse runs two ways: handed to an AI agent through the MCP server, or as a CLI in your
terminal. The agent path is the primary use; the CLI walkthrough below is the local one.

## Hand It to an Agent

Register Fuse with your client once. From your project root:

```bash
fuse mcp install
```

Your AI client (Claude Code, Cursor, or GitHub Copilot) then launches `fuse mcp serve`
automatically when MCP is enabled. You do not run the server in a terminal yourself.
Use `fuse mcp install --scope user` to register Fuse for every project on this machine.
See [Connect to your AI](/docs/start/connect-your-ai) for per-client details.

That collapses the explore phase into one scoped call instead of the agent opening file
after file. Give `fuse_ask` a task and a token budget and it picks the scoping strategy
and packs to fit:

```text
fuse_ask(
  path="C:/Projects/MyApp/src",
  task="where is the discount applied at checkout?",
  tokenBudget=20000
)
```

See [Connect to your AI](/docs/start/connect-your-ai) for client setup and
[Context for an agent](/docs/scenarios/context-for-an-agent) for the full workflow.

## Run One Command

Point Fuse at a .NET source directory:

```bash
fuse dotnet --directory ./src
```

The `dotnet` command selects the file extensions and exclusions appropriate for a
.NET solution (C#, project files, Razor, configuration) and applies C# reduction.

## Read What It Prints

When the run finishes, Fuse reports the shape of the result:

```text
Fused 511 files
Estimated tokens: 420,320 (-9.8%)
cache: 0 hit / 511 miss
Output: AutoMapper_2026-06-20_0130_420k.txt
```

It tells you the number of files included, the estimated token count and how much
that cut from raw concatenation, the reduction cache split as `cache: N hit / M miss`,
and where the file was written. By default the output goes to a `Fuse` folder inside
your Documents directory, and the file name ends with a token estimate such as `420k`.

## Look Inside the Output

The file opens with a **manifest**: a header that lists every included file with its
token cost, so a reader or an agent can judge the shape and cost of the fusion before
reading any file body. After the manifest, each file appears as an entry. In the
default XML format an entry looks like this:

```xml
<file path="src/Services/OrderService.cs">
public class OrderService { }
</file>
```

Files are ordered largest first by token count, so the most expensive content is at
the top.

## Choose How Much to Reduce

The default run does light, lossless cleanup. To cut more while keeping the public
API, raise the reduction level with `--level aggressive`:

```bash
fuse dotnet --directory ./src --level aggressive
```

For an architecture-level survey, drop method bodies to signatures with a
**skeleton**, which typically runs 66 to 93 percent smaller than a full fusion:

```bash
fuse dotnet --directory ./src --level skeleton
```

The [Reduction levels](/docs/concepts/reduction-levels) concept page explains each
step, and [Cut tokens for .NET](/docs/scenarios/cut-tokens-dotnet) is the task-shaped
version of this.

## Control the Output

Name the file and choose where it lands:

```bash
fuse dotnet --directory ./src --output ./context --name myproject
```

## Set Project Defaults

To avoid repeating flags, write a configuration file once:

```bash
fuse init
```

This scaffolds a `fuse.json` you can edit. See
[Configuration keys](/docs/reference/configuration-keys) for every supported key.
