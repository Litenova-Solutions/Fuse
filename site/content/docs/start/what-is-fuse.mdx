Fuse is the compiler oracle for .NET coding agents. Its thesis is that the compilation is
the source of truth and everything else is a projection of it, so it answers an agent's
edit-time questions from the compiler rather than from a guess. It typechecks a proposed
edit before the agent writes it, computes the blast radius of a change, stages
compiler-executed refactors as diffs that compile or are not handed over, and resolves
what the code actually runs: which implementation is injected for an interface, which
handler processes a request, which endpoint serves a route, which options type binds a
config section, and what a git diff actually impacts. Every answer carries a grade so the
agent knows what it is trusting, and Fuse abstains rather than guess when it cannot answer
at compiler grade.

It works from a warm, persistent index built with Roslyn: an index that stays built
between calls, in a single SQLite file at `.fuse/fuse.db`, so the first call pays the
analysis cost and later calls are fast.

Hand it to an agent through the MCP server. Before an edit, check it against the compiler
and see what a change would break:

```text
fuse_check(path="C:/Projects/MyApp/src", file="OrderService.cs", content="<proposed edit>")
  -> CS1061 at OrderService.cs:41: 'Order' has no member 'TotalAmount'
     repair packet: 'Total' exists on Order; grade: oracle

fuse_impact(path="C:/Projects/MyApp/src", symbol="IBasketService.Checkout")
  -> callers, implementers, and referencing types from the typed graph
```

For change and pull-request work, one call returns the changed files plus their blast
radius (callers, DI consumers, route and request handlers, options consumers, tests) as
packed context:

```text
fuse_review(
  path="C:/Projects/MyApp/src",
  changedSince="origin/main",
  maxTokens=25000
)
```

For an open-ended task with no anchor to check or resolve, localization is the fallback:
rank candidate files first, then plan context for them:

```text
fuse_localize(path="C:/Projects/MyApp/src", task="where is the discount applied at checkout?")
fuse_context(path="C:/Projects/MyApp/src", files=["src/Checkout/DiscountPolicy.cs"], maxTokens=25000)
```

It also runs as a CLI for humans and CI:

```bash
# install once
dotnet tool install -g Fuse

# build the index, then review a branch
fuse index ./src
fuse review ./src --changed-since origin/main --max-tokens 25000
```

## What It Does, in One Line

Fuse gives a .NET coding agent a compiler's answer instead of a guess: it verifies an edit,
scopes a change, and resolves what the code actually runs, from a warm Roslyn-backed graph,
so the agent stops guessing from file names and grep and stops waiting on a full build to
learn what it broke.

A verify call returns a graded verdict: oracle-grade when the build-captured compilation
answered speculatively, and an honest abstention (with the missing prerequisite named) when
it cannot. A read call returns a **context plan**: a scoped set of source bodies at mixed
render tiers, a semantic manifest, and per-file provenance, returned as an MCP tool response
to an agent or written to a file from the CLI.

## Why That Matters

On a solution with hundreds or thousands of C# files, a coding agent spends the start of
every task hunting: every file it opens to orient itself burns context window, each open
is a separate round-trip, and reading files in isolation loses the wiring (which concrete
type runs, who calls it, what a change breaks). Fuse front-loads that work into a warm
index and resolves the wiring deterministically, then hands the agent scoped context that
fits the window. For the mechanism, see [How Fuse works](/docs/concepts/how-fuse-works).

## Resolves What Actually Runs

The core is semantic resolution over a typed graph, not text search. Fuse resolves:

- a service interface to the concrete type registered for it in DI,
- a request or command to the handler that processes it,
- a route to the action or endpoint that serves it,
- a config section to the options type that binds it,
- a symbol to its declaration.

From any of those it expands through typed edges (calls, implements, injects, handles,
binds) to the neighborhood a task needs, and prunes the rest.

## Reduction without Losing the Surface

Reduction is supporting machinery: the rendering step that fits context to a budget, not the
product. It feeds the scoped context that verification and resolution hand back. Across
a commit-pinned corpus of real .NET libraries, the default and `aggressive` reductions cut
6 to 46 percent of tokens while keeping 99 to 100 percent of public types and methods. The
**skeleton** render tier, which keeps signatures and drops bodies, removes 39 to 56 percent
for an architecture-level view.

## Two Ways to Run It

Fuse ships from one package in two forms:

- The **`fuse` global tool** indexes a workspace and writes context to a file from your
  terminal. Start at the [Quickstart](/docs/start/quickstart).
- The **`fuse mcp serve` MCP server** exposes a set of verbs to an AI client such as Claude
  Code, Cursor, or GitHub Copilot, so the agent verifies edits and requests resolved context
  directly. See [Connect to your AI](/docs/start/connect-your-ai).

## What It Is Not

Fuse is not a generic file packer that concatenates text, and it is not an embedding
index that returns fuzzy chunks. It does not guess from vocabulary similarity: it resolves
.NET wiring from a parsed graph and is deterministic for the same index. It understands
C# structure, preserves the public API, and refines across calls in a session. The
[Why Fuse](/docs/start/why-fuse) page covers the landscape and the trade-offs in full.
