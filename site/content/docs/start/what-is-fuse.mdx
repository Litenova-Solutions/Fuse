---
title: What is Fuse
description: Fuse is a local compiler and typed .NET wiring service for existing coding agents, with graded answers, abstention, and compact context as supporting machinery.
---

Fuse is a local compiler and typed .NET wiring service that plugs into an existing coding
agent through Model Context Protocol (MCP). It does not replace Claude Code, Cursor,
GitHub Copilot, or another MCP client. It gives that agent operations grounded in Roslyn
and the .NET toolchain:

- typecheck a proposed single-file edit without writing it,
- inspect callers, implementers, and referencing types before a signature change,
- resolve a service, request, route, or config section through typed wiring,
- stage compiler-executed refactors as verified diffs,
- assemble git-seeded branch context with provenance.

Fuse runs on the developer's machine. It requires no hosted model, sends no source to a
Fuse service, and downloads no embedding model. Its persistent index lives in
`.fuse/fuse.db`.

## Ask the Compiler Before Writing

An agent can check an in-memory proposal and inspect typed impact before changing the
working tree:

```text
fuse_check(path="C:/Projects/MyApp/src", file="OrderService.cs", content="<proposed edit>")
  -> CS1061 at OrderService.cs:41: 'Order' has no member 'TotalAmount'
     repair packet: 'Total' exists on Order; grade: oracle

fuse_impact(path="C:/Projects/MyApp/src", symbol="IBasketService.Checkout")
  -> callers, implementers, and referencing types from the typed graph
```

Each check reports its evidence grade. Oracle-grade means the build-captured compilation
answered. When that compilation is unavailable, Fuse runs a scoped build and reports
build grade. If neither compiler path can answer, Fuse abstains and names the missing
prerequisite. Resident millisecond checks require the opt-in `FUSE_RESIDENT=1` mode; they
are not the default.

`fuse_check` does not write the source file. The server's explicit write operation is
`fuse_workspace` with `action=apply`; read, impact, and check operations leave the working
tree unchanged.

## Resolve Typed .NET Wiring

The graph connects code using compiler and framework evidence instead of vocabulary
similarity. Fuse can resolve:

- a service interface to a registered implementation,
- a request or command to its handler,
- a route to its action or endpoint,
- a config section to its options type,
- a symbol to its declaration and typed neighborhood.

The recorded semantic benchmark matched all 24 hand-built edges in the curated
OrderingApp wiring fixture, with no false positives. That result validates the fixture's
covered patterns; it is not a claim that static analysis resolves every use of reflection
or runtime dispatch.

## Review a Branch from Its Git Diff

For change and pull-request work, one call starts with the git-changed files and adds
graph-selected support files under a token budget:

```text
fuse_review(
  path="C:/Projects/MyApp/src",
  changedSince="origin/main",
  maxTokens=25000
)
```

Across the recorded 69-PR corpus, this git-seeded context retained every changed file by
construction, reached 93.4% precision, and used a median 1,026 tokens. It is compact branch
context. It does not prove that the result contains every file a reviewer may need.

For an open-ended task with no anchor to check or resolve, localization is the fallback:
rank candidate files first, then plan context for them:

```text
fuse_find(path="C:/Projects/MyApp/src", kind="task", query="where is the discount applied at checkout?")
fuse_context(path="C:/Projects/MyApp/src", files=["src/Checkout/DiscountPolicy.cs"], maxTokens=25000)
```

It also runs as a CLI for humans and CI:

```bash
# install once
dotnet tool install -g Fuse

# build the index, then review a branch
fuse index ./src
fuse review ./src --changed-since origin/main --max-tokens 25000
```

## Reduction Supports the Compiler Service

Reduction is supporting machinery: the rendering step that fits context to a budget, not the
product. Across the four repositories in `reduce.json`, the **skeleton** tier removed 38%
to 44% of tokens while retaining every public type and 97% to 100% of public methods. The
public-API tier removed 47% to 60%. These ranges describe the recorded corpus.

![Fuse benchmark results covering compiler-labeled OrderingApp mutations, a reduced-scope agent loop with confidence intervals, the curated 24-edge OrderingApp wiring fixture, and git-seeded branch context over 69 pull requests.](/fuse-benchmarks.svg)

Every number comes from a harness anyone can rerun against the same pinned commits.
See [Benchmarks](/docs/project/benchmarks) for the method and the full results,
including where Fuse loses.

## Two Ways to Run It

Fuse ships from one package in two forms:

- The **`fuse` global tool** indexes a workspace and writes context to a file from your
  terminal. Start at the [Quickstart](/docs/start/quickstart).
- The **`fuse mcp serve` MCP server** exposes a set of verbs to an AI client such as Claude
  Code, Cursor, or GitHub Copilot, so the agent verifies edits and requests resolved context
  directly. See [Connect to your AI](/docs/start/connect-your-ai).

## Evidence Boundaries

The current loop benchmark was reduced-scope. It observed 89% true pass@1 for Fuse (95%
CI 82%-95%) and 82% for native tools (95% CI 74%-90%). The intervals overlap. Agent-visible
build and test calls were 3.1 for Fuse and 3.2 for native tools, so the run does not show
a build-round-trip collapse.

The check gate returned no wrong verdicts on 1,000 compiler-labeled OrderingApp mutations,
split evenly between breaking and neutral edits. That finding applies to the tested fixture,
not universally. Fuse communicates these boundaries through evidence grades and abstention
rather than treating every index mode as compiler truth.
