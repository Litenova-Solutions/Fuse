title: Why Fuse, and how it compares
description: The problem agentic coding tools hit on large .NET codebases, what generic packers and RAG indexers do and where they fall short, and what Fuse's semantic resolution does differently.
---

Most tools that feed code to a language model fall into two camps: packers that dump
the repository as text, and retrieval indexes that return fuzzy chunks. Fuse is
neither. It is the compiler oracle for .NET coding agents: it verifies an edit against the
compiler before it lands, computes what a change breaks, and resolves what code actually
runs, from a Roslyn graph. Finding and serving context (the packer-and-RAG problem this
page compares against) is the machinery that feeds those answers, and Fuse does it with the
public API intact and scoped to the task. This page explains the problem that machinery
solves, the landscape, and the trade-offs.

The deeper difference is not how much context Fuse returns but what it can promise about it.
A packer and a RAG index both answer "here is some text that might be relevant". Fuse also
answers "this edit does not compile, here is the error and the fix" and "changing this
signature breaks these nine call sites", which is a different kind of answer than any
retrieval tool gives, and the one the rest of the program is built around.

## The Problem: Finding the Code That Actually Runs

An agentic coding tool such as Claude Code, Cursor, or GitHub Copilot does not start
by editing. It starts by finding the right code. To make a safe change it has to learn
the wiring: which concrete type is injected for an interface, which handler processes a
request, which endpoint serves a route, what a change will break. So it lists
directories, runs searches, and opens files one at a time, reading each result into its
context window before deciding what to open next.

On a small project that is cheap. On a solution with hundreds or thousands of C#
files it is the expensive part of the task:

- **Token cost.** Every file the agent opens to orient itself consumes context window
  it could have spent on the actual change. Much of that content is comments, using
  directives, and boilerplate the agent does not need to reason about structure.
- **Round-trips.** Discovery is sequential. The agent reads, decides, reads again.
  Each hop is a model call with latency, and the agent often re-reads files it
  already saw because nothing kept a compact map of the codebase.
- **Lost wiring.** Reading files in isolation, the agent reconstructs the dependency
  relationships, the DI registrations, the request and route handlers, and the public
  surface by hand. Grep finds the interface but not the type that runs.

The measured shape of this is on the [Benchmarks](/docs/project/benchmarks) page
(Layer 4). Over the real merged pull requests tested, a blind agent must read each file a
change needs at least once, a lower bound of about seven round-trips, while one scoped
Fuse call acquires that context in a single response at about 37,000 tokens and 49
percent recall, against roughly 425,000 tokens for a generic-packer dump at the same
one call. The round-trip count is a bound from ground truth, not a measured agent;
the wall-clock benefit (an explore loop of minutes collapsing to one call) is a
theory-grounded illustration of the mechanism, labeled as such, not a benchmark.

## The Landscape

### Generic Repository Packers

Tools such as Repomix, Code2Prompt, and Gitingest concatenate a repository into one
text blob with a header and per-file separators. They are language-agnostic and easy
to run, which is their strength.

Their limit for code structure is that they treat source as plain text. They do not
know what a C# type or a method signature is, so they cannot reduce by structure,
cannot preserve the public API while dropping noise, cannot resolve which type a DI
registration injects, and cannot scope to the files a task touches. The
envelope they add often makes the output larger than raw concatenation: on the benchmark
corpus, Repomix output measured 1.3 to 3.9 percent larger than the raw files, while Fuse
default was smaller than both. Because a packer dumps the whole repository, both arms
acquire a task's context in one call, but the packer pays for everything: in the Layer 4
scenario, a Repomix dump averaged about 425,000 tokens against about 37,000 for one
scoped Fuse call (at 49 percent recall of the changed files). The packer's recall is 1.00
only because it includes the entire repository.

### RAG and Embedding Indexers

Retrieval-augmented generation indexes a codebase into vector embeddings and returns
the chunks most similar to a query. This is useful for open-ended search across a
very large corpus and for natural-language questions.

Its limits for code are structural. Embeddings return ranked chunks, not whole types
or a faithful public surface, so recall is partial and the result is non-deterministic:
the same query can return different chunks as the index or model changes. A chunk
boundary can split a method from its signature. And it answers "what looks similar to
this text", not "which concrete type is injected here" or "give me this type and the
handler that processes its request, intact". Fuse resolves wiring from a parsed graph,
not similarity, so the answer is deterministic and structural.

## What Fuse Does Differently

State the value against the three things you would otherwise reach for, strongest first.

1. **Versus no tool, the baseline an agent lives in.** This is the headline. A blind
   agent must read each file a task touches at least once before it has the context: a
   measured structural lower bound of 6.9 files on average over the real merged pull
   requests tested, and up to 11.9 on a large library. Fuse acquires the same context in
   one scoped call. That is the round-trip collapse, and it is the value you feel.

2. **Versus a generic packer (Repomix).** Both acquire context in one call, so Fuse ties
   on round-trips and wins on tokens: about 37,000 tokens for the scoped set against about
   425,000 for the whole-repo dump, roughly 11 times fewer, because Fuse scopes and the
   packer dumps. To be plain: Fuse does not beat Repomix on round-trips; it ties at one
   call and wins on tokens.

3. **Versus RAG and embedding indexes.** Fuse resolves .NET wiring and returns whole types
   and their dependency neighborhood, deterministically, where a vector index returns
   ranked fuzzy chunks and can split a method from its signature.

4. **And the code it returns keeps the surface intact.** Default and `aggressive`
   reduction remove 6 to 46 percent of tokens while keeping 99 to 100 percent of public
   types and methods; the skeleton render tier removes 39 to 56 percent for an
   architecture view.

Fuse is not only one call. Across a task it is a few cheap calls (survey with `fuse_map`,
localize with `fuse_localize`, resolve wiring with `fuse_resolve`, review with
`fuse_review`) instead of dozens of blind reads, and a `sessionId` means later calls do
not resend files already returned. A packer dumps once; it cannot refine. See
[How Fuse works](/docs/concepts/how-fuse-works) and
[Sessions and deltas](/docs/concepts/sessions-and-deltas).

### The Choices Behind It

Fuse is built specifically for finding and serving context on a .NET codebase, and it
makes a different set of choices:

- **Semantic, not lexical.** It loads the workspace through MSBuild and Roslyn and builds
  a typed graph of nodes (types, members, routes, registrations) and edges (calls,
  implements, injects, handles, binds), with a syntax-only fallback when a project does
  not load. It resolves the type that actually runs, not the one a name matches.
- **Warm and persistent.** The index lives in a single SQLite file at `.fuse/fuse.db` and
  stays built between calls, so a multi-call task pays the analysis cost once.
- **Structural reduction.** It parses C# enough to render at mixed tiers: full bodies,
  signature-only **skeletons**, or compressed source, and removes comments, usings, and
  whitespace. The output represents the same source in fewer tokens.
- **Fidelity-preserving.** Reduction is not deletion. An independent Roslyn oracle
  confirms the default and `aggressive` tiers keep 99 to 100 percent of public types and
  methods.
- **Deterministic.** Same index, same answer. Resolution comes from the parsed graph, not
  similarity ranking, so there is no index drift in the answer.
- **MCP-native.** `fuse mcp serve` is a Model Context Protocol server, so the agent verifies
  edits and fetches resolved context in one call instead of reading files one by one.

## How It Compares

| Capability | Generic packers | RAG / embeddings | Fuse |
|------------|-----------------|------------------|------|
| Round-trips to acquire context | One dump | One query, partial | One scoped call, or a few cheap refining calls |
| Context per call | Whole repo | Ranked chunks | Scoped set, about 11x fewer tokens than a packer dump |
| Understands C# structure | No, plain text | No, opaque chunks | Yes, types and signatures |
| Resolves DI / request / route wiring | No | No | Yes, from the typed graph |
| Cuts tokens vs raw | Often larger than raw | Per chunk only | 6 to 46 percent at full API fidelity |
| Keeps the whole public API | Only by including everything | No, partial recall | 99 to 100 percent, verified |
| Deterministic output | Yes | No, similarity-ranked | Yes |
| Scope to a task | No | By query only | Localize, resolve, or git change, with graph expansion |
| Built for agents (MCP) | Rarely | Varies | Verify and context verbs, built in |
| Setup and freshness | None | Build and refresh an index | A persistent index, refreshed on demand |

Fuse is complementary to the other two where their strengths lie. For a polyglot
repository a generic packer is simpler, and for open-ended semantic search across a
huge corpus an embedding index has its place. For giving a .NET agent the code that
actually runs with the API intact and the tokens cut, Fuse is purpose-built.

## Honest Boundaries

When a project does not load through MSBuild, Fuse falls back to syntax-only indexing,
which resolves fewer edges than the full semantic graph. Resolution can still miss edges
that come from dynamic dispatch or reflection because those targets are not visible to
static analysis. The [Benchmarks](/docs/project/benchmarks) page reports exactly where
Fuse ties or loses, not only where it wins.
